The Policy Framework

One of the strengths of Okta is its ability to respond to dynamic environments and provide responsive authentication flows. Okta uses a Policy Framework to accomplish this. Editorial note: The opening sentence is vague, because it's unclear what a dynamic environment is, or what the other strengths of Okta are. Be careful about redundancies like using `respond` and `responsive` together. Try to go with something that summarizes the whole topic in a single line, so the reader can decide whether they need to read it: Okta policies establish the requirements for user authentication flows.

The Policy Framework consists of several types of policies. Each policy one manageshandles a different part of the authentication flow, such as user sign-ins to Okta or to apps, or requiring users to enroll in multifactor authentication. You use policies to create the rules for accessing your Okta org or apps, and enrolling in MFA. Okta enforces these rules when a user attempts to sign in to Okta or access an app.

When combined with groups and nNetwork zZones, policies work together to respond to virtually any type of authentication scenario and safeguard access to Okta and your apps. For example, you can configure policies that cause Okta to respond differently depending on whether users sign in from inside or outside your in-house network.

This topic describes each type of policy and how you can use them together to build a Zero Trust architecture.

EvaluationExecution

When you create a policy, it appears in a list. Similarly, each rule in a policy appears in a list inside their policy. The order in which the policies and rules appear is important. Okta evaluates executes each policy and rule in the order in which they appear in the Admin Console in the list, starting at the top of the list. Similarly, Okta executes each rule inside each policy in the order in which they appear in the list, also starting at the top.

When you create policies or rules, you can drag policies and rules them into the order in which you want Okta to process them. Okta recommends that you place your more restrictive policies or rules at the top of the list and , and your less restrictive ones at the bottom. Okta also recommends that you leave the default policy or rule as the last (lowest) item in the list.

Okta checks each policy in the list until it finds one that satisfies the requirements of the rules inside it. Okta only executes the default policy or rule when none of the custom policies or rules above it in the list can be fulfilled. When it finds a policy that can be fulfilled, it stops processing further policies and carries out the instructions in the policy. If Okta doesn't find a policy that can be fulfilled, it blocks the user from signing in to Okta, accessing the app, or enrolling in MFA, depending on the policy type.

Set up a Zero Trust architecture

A Zero Trust architecture is one in which all users must be authenticated and authorized before they can receive access to Okta or an app.

First, you create groups of users according to the type of access they need to do their jobs. Next, you create nNetwork zZones so you can establish authentication requirements according to where users sign in from. For example, you might want to have fewer restrictions when users sign in from your in-house network, and more restrictions when they sign in from outside that network. You can also create Network Zones that let you block sign-ins from specific countries or regions.

Then, After you've created groups and Network Zones, you create and configure policies that set the requirements for signing in to Okta, determine when users must enroll in MFA, and determine which users can access which apps.
