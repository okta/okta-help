The () factor lets you use a biometric method, such as fingerprint reading, to authenticate. This factor supports three authentication methods:

  • Security keys, such as or Google Titan.
  • Platform authentication that's integrated into a device and uses biometric data, such as Windows Hello or Apple .
  • Sign-ins to URLs that are different from the org's URL, custom domain URL, trusted cross-origin, or cross-Relying Party Identifier, require validation when using the Trusted Origins API. See Configure Trusted Origins.

() follows the FIDO2 Web Authentication (WebAuthn) standard. After this factor is enabled, end users can select it when signing in and use it for additional authentication.

Current limitations

  • When you configure your org with User Verification set to Discouraged, users who enroll a () factor don't see the () enrollment names of the factors they enroll. Instead, they're listed generically as "Authenticator," and no other details about the factor are provided.
  • On browsers, the () factor isn't usable if the browser requires an update. functionality is restored when you restart the browser after applying the update.
  • Each user can configure a maximum of 10 enrollments.

User experience

If this factor is enabled, users can select it when signing in and set it up so it can be used for additional authentication. Depending on your configuration, users may also be required to provide User Verification. This verification can include a biometric challenge, PIN, or password in addition to tapping the device.

When enrolling a Security Key or Biometric factor, users are prompted to allow to have information about that particular enrolled factor. This allows each () factor to appear by name in the Extra Verification section of the user's Settings page.

If a user is only enrolled in the () factor, they risk being unable to authenticate into their account if something goes wrong with their () factor or device. To ensure that users can always access their account if one of their devices malfunctions, is lost, or stolen, encourage users to do the following:

  • Set up other MFA factors, in addition to (), that aren't bound to a particular device.
  • Create multiple enrollments in multiple browsers and on multiple devices.

() factor enrollments, such as , are attached to a single browser profile on a single device.

If users want to use a () factor on multiple browsers or devices, advise them that they must create a () enrollment in each browser, and on each device, in which they want to use the factor.

For example, if a user has and browsers on a Microsoft Windows computer, and and browsers on an Apple Macintosh computer, they must create a enrollment in each of those four browsers.

If they have multiple Google account profiles in the browser, they must also create a enrollment for each of those Google account profiles.

In addition, if you enable the () factor on your *.okta.com URL, the () factor only allows access to your org using your *.okta.com URL. If you enable the () factor using the custom URL for your org, the () factor only allows access to your org through that custom URL. To allow your users to access your org through both URLs, you must enable the () factor in both URLs.

Passkey Management

Passkeys are an implementation of the standard in which the () credential may exist on multiple devices, such as on phones, tablets, or laptops, and across multiple operating system platforms. Passkeys enable () credentials to be backed up and synchronized across devices. This preserves the strong key-based/non-phishable authentication model of () while trading off some enterprise security features, such as device-bound keys and attestations, that are available today with some () authenticators. Users no longer need to carry their security key or phone to pass multifactor authentication challenges. Instead, they can use any device they have already enrolled to authenticate themselves because their credential isn't confined to a single device.

In managed-device environments, users may be able to enroll unmanaged devices to a passkey credential and use such devices to gain access to corporate systems. allows admins to block the use of passkeys for new () enrollments for their entire org. When this feature is turned on, users can't enroll new, unmanaged devices using pre-registered passkeys. Admins can ensure that security policies are enforced on managed devices and address the risk of unmanaged and potentially compromised devices accessing corporate systems.

When you turn this feature on, that is, block the use of passkeys in your org, users running macOS Monterrey can't enroll in using the Safari browser.

For more information on passkeys, see <link to blog article by Mukul Hinge>.

This is the default setting. Turn this feature off and allow the use of passkeys in your org:

  2. Click the toggle switch for the Block Passkeys for FIDO2 (WebAuthn) Authenticators option. The toggle switch turns gray.
