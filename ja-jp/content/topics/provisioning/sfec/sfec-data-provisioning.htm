SAP SuccessFactors Employee Centralデータプロビジョニングについて

下の表はSAP SuccessFactors Employee CentralがOktaに統合された場合に特定のデータ型がどのようにプロセスされるかを示しています。

データ型 説明
複数ジョブの割り当て

複数ジョブが割り当てられている従業員がいる場合（グローバル割り当てまたは同時割り当て）、以下のようにOktaでプロセスされます。

グローバル割り当てには"person___employment_information_GA1___"というプレフィックスが付きます。例："person___employment_information_GA1___job_information___job_title"

標準割り当てには"person___employment_information_ST1___job_information"というプレフィックスが付きます。例："person___employment_information_GA1___job_information___job_title"。ST1/ST2でemployment_information.employment_idの値を定義します。例："employment_information.employment_id=123"が"ST1"としてプロセスされ、"employment_information.employment_id=456"はST2としてプロセスされます。
ユーザーステータス

ユーザーのステータスはEmployee Centralのジョブ情報エンティティから取得されます。

ユーザーのjob_information.emplStatus == "A"の場合、ユーザーはOktaでアクティブとして扱われます。

ユーザーのステータスがジョブ情報エンティティでアクティブでない場合、Oktaはユーザーのjob_information.start_date)を再度確認します。

ユーザーの開始日がpreHire Interval（雇用前インターバル）期間内の場合、ユーザーはOktaでアクティブな雇用前ユーザーとして識別されます。Oktaは各ユーザーに対して最も早い値のjob_informationレコードを選択し、次のようなチェックを実行します： "now > startDate > now + preHireInterval" — この条件に当てはまる場合、ユーザーはアクティブとして扱われます。

ユーザーが非アクティブであり、雇用前検証を通過しない場合、退職後検証が実施されます。Oktaは直近の*___job_information___start_date（退職開始日）の値を選択し、"inactiveStartDate < (now - postTerminationInterval)"と照合します。

ユーザーがすべての検証に通らなかった場合、非アクティブと識別され、Oktaにはインポートされません。

下の図はOktaユーザーステータス検証のワークフローを示しています。

SuccessFactors統合はMultiple Active User Statuses（複数のアクティブユーザーステータス）をサポートし、そこで管理者は様々なユーザーステータスをOktaでアクティブとして扱うよう選択できます。たとえ管理者が何も選択しなくても、デフォルトのステータスは常に［Active（アクティブ）］です。たとえば、キャシーは現在有給休暇中ですが、それでもOktaでアプリへのアクセスを希望しているとしています。この機能を使用すれば、管理者は有給休暇中をアクティブユーザーステータスとして選択し、キャシーはアプリにアクセスすることができます。

SucessFactorsのプロビジョニングを構成する際に（［Provisioning（プロビジョニング）］タブ >［To Okta（Oktaへ）］>［General（一般）］セクションの［Active User Statuses（アクティブユーザーステータス）］フィールド）、［Active（アクティブ）］とみなされるユーザーステータスを選択します。

注：

  • 2020年10月より前に作成されたアプリでは、OktaがSuccessFactorsから利用可能なユーザーステータスの一覧を取得できるように、プロビジョニング設定を再保存したりProfile Editorで属性リストを更新する必要があります。
  • 統合ユーザーがジョブステータス情報を取得するにはpicklists_sys_admin権限が必要です。［Provisioning（プロビジョニング）］タブにステータスリストが表示されない場合、APIユーザーがこの権限を持っているか確認してください。
電話およびメール

電話およびメールのどちらも、"isPrimary = true"とマークされたエンティティがOktaユーザープロファイルにマップされます。ライトバック機能が有効化されている場合、Oktaは電話およびメールフィールドにライトバックし、そのタイプをPrimary（プライマリ）として設定します。

API制限により、Oktaのメールアドレスは、プライマリにマークされた設定済みのメールを持ち、アクティブなメールタイプ（通常は「Business」タイプ）を持つユーザーに対してのみアップデート可能です。
請負業者から正社員への変換

SAP SuccessFactors Employee Centralでは、請負業者から正社員への変換は派遣契約を終了した後、正社員の新規採用として扱われます。

派遣要員がSAP SuccessFactors Employee Centralで退職すると、次回ユーザーデータが同期化されたときに、退職後プロセスが有効化されるまでユーザーはOktaで非アクティブ化されます。ユーザーがSAP SuccessFactors Employee Centralで正社員として追加されると、そのデータがOktaにインポートされ、新規作成されたSAP SuccessFactors Employee Centralの正社員と、非アクティブ化された既存のOktaユーザーを一致させるためにインポートマッチングルールが適用されます。ただし、ユーザーの再アクティブ化は手動で行う必要があります。インポートマッチングルールを使用した自動再アクティブ化はサポートされていません。
