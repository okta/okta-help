Oktaサインオン ポリシーの設定

Oktaサインオン ポリシーを作成するには、 ［Add New Okta Sign-on Policy（新しいOktaサインオンポリシーを追加）］をクリックします。開いた画面で、使用するポリシーの名前と説明を入力します。

ポリシーを特定のグループにのみ適用できるように指定できます。ポリシーをグループに割り当てるには、使用するグループ名を［Assign to Groups（グループに割り当てる）］フィールドに入力します。グループ名をポリシーに割り当てるには、その前にそのグループ名が存在していることが必要です。終了したら、 ［Create Policy and Add Rule（ポリシーを作成してルールを追加する）］をクリックします。

ポリシーを評価するとき、ポリシー内の条件がルール内の条件と統合されます。これらの条件がすべて満たされたら、ルールが適用されます。ルールのないポリシーは適用できません。

また、ポリシーを追加する順序も重要です。ルールに最初に一致したポリシーのみが、条件が満たされたときに適用されます。他のポリシーは適用されません。

  • 新しいポリシーを作成したら、それが有効になるためにはすべてのアクティブなセッションを閉じる必要があります。
  • サインオン ポリシーはAPIトークンの有効性とライフタイムに影響しません。APIトークン管理

Oktaサインオン ポリシーの作成

新しいポリシーを作成した後、すべてのアクティブセッションを閉じて、新しいポリシーをアクティベートします。

  1. 管理コンソールで、[セキュリティー] > に移動します ［Authentication（認証）］。

  2. ［Sign On（サインオン）］ タブをクリックします。

  3. ［Add New Okta Sign-On Policy（Oktaサインオン ポリシーを追加）］をクリックします。

  4. 以下のフィールドに入力します。

    • Policy Name（ポリシー名）：サインオン ポリシーの名前を入力します。

    • Policy Description（ポリシーの説明）：任意。サインオン ポリシーの説明を入力します。

    • Assign to Groups（グループに割り当てる）： ポリシーを適用するグループの名前を入力します。ポリシーは複数グループに適用できます。

  5. ［Create Policy and Add Rule（ポリシーを作成してルールを追加する）］をクリックします。

Oktaサインオン ポリシー ルールの追加

  1. ［Add Rule（ルールを追加）］ をクリックしてポリシーにルールを追加します。必要に応じて以下のフィールドに入力します。

Rule name（ルール名）：作成するルールの説明的な名前を追加します。
Exclude users（ユーザーを除外）：必要なら、グループ内の個別のユーザーをルールから除外できます。
If a user is located...（ユーザーの場所）： ドロップダウン メニューを使用して場所パラメータを割り当てます。認証を求める場所のタイプを指定できます。
Manage configuration for Network（ネットワークの設定を管理）：［Manage configuration for Network（ネットワークの設定を管理）］リンクをクリックして、選択したアクセスを可能にするゲートウェイ設定にアクセスします。このオプションの使い方の詳細は、「 Oktaセキュリティの使用」ページの「パブリックゲートウェイIP」セクションを参照してください。
Authenticates via...（認証手段）： このドロップダウン メニューを使用して必要な認証手段を指定します。
And Behavior is（動作）： 作成されている既存の動作の名前を入力します。動作を追加するには、動作の名前を入力し始めます。すべての一致する定義済み動作のドロップダウン リストが表示され、それから動作を選択できます。追加したい各動作に対してこれを繰り返します。複数の動作を追加するときは、OR条件として取り扱います。「サインオン ポリシー ルールに動作を追加する」を参照してください。
Then Access is...（アクセスを許可/拒否）：前のドロップダウン メニューの認証フォームに基づき、これを用いて条件によりアクセスを許可するか拒否するかを確定します。
Prompt for Factor（要素の入力を求める）：少なくとも1つの要素タイプがあるときにのみ使用可能と表示されます。このボックスを選択すると、プロンプトを表示するのはデバイスごとか、サインオンごとか、または指定するセッション時間ごとかを選択するラジオボタンが表示されます。［Every Time（毎回）］ を選択してもエンドユーザーはMFAプロンプトを制御することはできません。これらのオプションに関するユーザー エクスペリエンスの詳細は、「MFAプロンプトのエンドユーザー制御」を参照してください。
Remember Device By Default（デフォルトでデバイスを記憶する）： この機能を有効にすると、サインイン時にエンドユーザーオプションを自動選択します： Do not challenge me on this device again（このデバイスについてチャレンジしない）。
Manage configuration for Multifactor Authentication（多要素認証の選択を管理）： Manage Configurations for Multifactor Authentication（多要素認証の選択を管理）リンクをクリックすると、［Authentication（認証）］ ページと ［Multifactor（多要素）］ タブにクイックアクセスできるようになります。各認証オプションの詳細は、「多要素認証の設定」を参照してください。プロンプトを表示するのはデバイスごとか、サインオンごとか、または指定するセッション時間ごとかを選択するラジオボタンを表示するボックスを選択します。セッションごとを指定する場合、セッションにデフォルトライフタイムが設定されているものの、ユーザーがOktaセッションからサインアウトするとセッションは終了することに注意してください。
Factor Lifetime（要素のライフタイム）：このドロップダウン メニューを使用して、ユーザーがMFAにチャレンジするまでの待ち時間を指定します。デフォルトのライフタイムは15分で、最大値は6か月です。要素のライフタイムを設定すると、エンドユーザーが［Factor Lifetime（要素のライフタイム）］で指定されている期間ほどサインアウトした後、次回のサインインでMFAに対して再びサインインする必要がありません。この設定を適用するには、エンドユーザーはそれを確定するボックスを選択する必要があります。その例の一つは、「Do not challenge me on this device for the next 15 minutes（今後15分間チャレンジしない）」です。この場合、サインアウトした後、新しいサインインが前回のMFAサインイン後15分以内であれば、MFAプロンプトは表示されません。ユーザーがボックスを選択しないと、常にMFAプロンプトが表示されます。前回のサインインからの経過時間がダッシュボードの下部に表示されますが、最新値を見るためには画面を更新する必要があります。
And Session Lifetime is…（セッションライフタイム）： このドロップダウン メニューを使用して、認証プロンプトをトリガーするまでの最大待ち時間を指定します。このオプションに指定できる最大値は90日です。これは総接続時間ではありません。デフォルトのセッションライフタイムは2時間です。これは、セッションの残り時間が5分になった時点でユーザーに対してカウントダウンタイマーを表示するまでの待ち時間です。

最大セッションライフタイム値は、Okta APIを介して設定できます。以前にこの値をOkta APIで設定していれば、ここで指定する最大値はこの値を超えることはできません。

複数のサインオン ポリシーを追加した場合、選択した基準に一致する最初のもののみが適用されます。

グローバルサインオン ポリシーアクション

  • ポリシー1の左側に示すように、ポリシー名の横にある点線表示の縦棒をグラブして、リスト内の目的の位置に移動することで、デフォルトポリシー以外のポリシーの順序を変更できます。

  • ポリシー内のルールの順序は、ルール名の左にある縦棒をグラブすることで変更できます。
  • ［Add New Okta Sign-on Policy（新しいOktaサインオン ポリシーを追加）］を選択して新しいポリシーを追加します。

個別のサインオン ポリシーアクション

選択したポリシーのみに影響する下記のアクションのみを実行できます。リストでポリシー名をクリックして選択します。選択したポリシーが青色で表示されます。

  • 選択したポリシーをアクティベートまたはディアクティベートします。ポリシーをディアクティベートすると、どのユーザーにも適用されませんが、あとでリアクティベートできます。
  • ポリシーを編集するには、［Edit（編集）］をクリックします。
  • ポリシーを削除するには、［Delete（削除）］をクリックします。デフォルトポリシーは削除できません。削除したポリシーは復元できません。
  • 選択したポリシーにルールを追加するには、［Add Rule（ルールを追加）］をクリックします。ポリシー内でルールをアクティベート、ディアクティベート、編集、または削除できます。
  • ルールの詳細を見るには、［Add Rule（ルールを追加）］の下のルール名をクリックします。

MFAプロンプトのエンドユーザー制御

下に示すように［Prompt for Factor（要素をプロンプト）］チェックボックスを選択すると、そのセッションでエンドユーザーにプロンプトする方法に影響する3つのオプションが表示されます。

そのうち2つのオプションはエンドユーザーがこれらのプロンプトを制御することを許可しますが、もう1つのオプションは許可しません。

  • Per Device（デバイスごと）：［End user MFA challenge(エンドユーザーMFAチャレンジ)］ダイアログボックスに［Do not challenge me on this device again（このデバイスに再度チャレンジしない）］オプションを表示します。このオプションは新しいデバイスのみをプロンプトできるようにします。
  • Every Time（毎回）：エンドユーザーはOktaにサインインするとき毎回プロンプトされ、いつプロンプトされるかを変更できません。
  • Per Session（セッションごと）：［End user MFA challenge(エンドユーザーMFAチャレンジ)］ダイアログボックスに［Do not challenge me on this device for the next(minutes/hours/days) （このデバイスにチャレンジしない（分/時間/日））］オプションを表示します。表示される［Factor Lifetime］フィールドで［Factor Lifetime（要素ライフタイム）］を指定します。セッションごとを指定する場合、セッションにデフォルトライフタイムが設定されているものの、ユーザーがOktaセッションからサインアウトするとセッションは終了することに注意してください。

事前認証サインオン評価 ポリシー

認証 APIを使用してサインインするエンドユーザーは、パスワードや他の要素の確認の前にサインオン ポリシーの評価を受けます。この評価により、組織内のアカウントロックアウト数を削減できます。

サインオン ポリシーを［deny（拒否）］に設定すると、ユーザーのサインオン試行が拒否され、「Authentication failed（認証に失敗しました）」というエラーが生成されます。このシナリオでは、失敗したログイン数のカウンターは増分されず、事前認証サインオン ポリシー評価を示すイベントがトリガーされます。

  • このバックエンド機能を有効にする場合、管理コンソールでのUI の変化や必要なセットアップはありません。
  • このポリシーは、JITプロビジョニングで作成された新規アカウントの初期認証では機能しません。このポリシーの影響を受けるためには、エンドユーザーアカウントがOktaに存在する必要があります。
  • このポリシーは、ユーザーが拒否された場所から資格情報をリセットすることを防止しません。
