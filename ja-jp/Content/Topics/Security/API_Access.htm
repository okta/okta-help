API Access Management

API Access Managementを使用すると、Oktaでカスタム認証サーバーを構築できます。これを使用して独自のAPIエンドポイントを保護したり、Oktaの組織認証サーバーを使用したりできるようになります。利用できる認証サーバーのタイプをご覧ください。

認証サーバーとは何ですか？

認証サーバーは、「ステージング」や「本番」などのセキュリティ境界を定義します。各認証サーバー内で、独自のOAuthスコープ、クレーム、アクセス・ポリシーを定義できます。これにより、アプリとAPIを中央の認証ポイントに固定できるようになります。また、属性を変換するためのユニバーサルディレクトリ、エンドユーザー用のアダプティブMFA、アナリティクス、システム・ログなど、Oktaの豊富なID機能を活用してAPIコミュニティに拡張できます。

基本的に、認証サーバーをはOAuth 2.0トークンを作り出すだけのエンジンです。各認証サーバーには、セキュリティドメイン間の境界を適切に管理するための、一意の発行者URIとトークン用の独自の署名キーが含まれます。認証サーバーはOpenID Connectプロバイダーとしても機能します。つまり、認証サーバーのエンドポイントからアクセストークンの他にIDトークンも要求できます。

認証サーバーはどのように選べばよいですか？

Oktaの組織認証サーバーを使用するか、カスタムの認証サーバーを使用するかをどのように判断すればよいですか？

  • 非Oktaリソースは保護する必要があります。
  • どの認証ポリシーが必要かは、その人が誰か（従業員、パートナー、エンドユーザー）や、その他の専門的項目によって異なります。

注

従業員、パートナー、エンドユーザーの全員がシングルサインオンで同じ認証ポリシーを使用できる場合は、組み込みの組織認証サーバーを試してください。

認証サーバーはどのように設定できますか？

クライアントとOktaとの間の認証を管理するには：

  • クライアントアプリでスコープとクレームを特定し、Oktaに登録します。
  • 1つ以上の認証サーバーを作成し、アプリで想定されるものと一致するようにスコープとクレームを定義します。

最初にどのタスクを実行するかは重要ではありませんが、クライアントアプリがスコープ名を認識し、認証サーバーで定義されたクレームを想定しているようにする必要があります。

認証サーバーを作成して構成する手順は次の通りです：

認証サーバーの作成
  1. ダッシュボードから［Security（セキュリティ）］>［API］の順に選択し、［Authorization Servers（認証サーバー）］タブを選択します。その後、［Add Authorization Server（認証サーバーの追加）］をクリックして、以下の編集可能な情報を入力します。［Issuer（発行者）］、［Metadata URI（メタデータURI）］、［Last Rotation（最終ローテーション）］は編集できません。
    • Name（名前）
    • Audience（オーディエンス） – アクセストークンを使用するOAuthリソースのURI。この値は、アクセストークンのデフォルトオーディエンスとして使用されます。
    • Description（説明）
    • Signing Key Rotation（署名キーのローテーション） – 自動または手動（手動によるローテーションは、認証サーバーをポーリングして署名キーのリストを自動的に更新できないクライアントに対してのみ使用する必要があります。手動を使用する場合、3か月ごとにキーをローテーションすることを推奨します。）

完了すると、認証サーバーの［Settings（設定）］タブに入力した情報が表示されるので、ここで編集できます。

スコープの作成

スコープは、APIエンドポイントで実行できる高レベルの操作を表します。スコープはアプリケーションにコードとして組み込まれます。その後、アプリケーションが認証サーバーからスコープを要求し、アクセス・ポリシーが許可するスコープと拒否するスコープを決定します。

提供される予約済みのスコープの他にスコープが必要な場合は、これを作成します。

  1. 表示する認証サーバーの名前を選択してから、［Scopes（スコープ）］を選択します。

  2. ［Scopes（スコープ）］>［Add Scope（スコープを追加）］の順に選択します。
  3. 名前と説明を入力します。
  4. ［User Consent（ユーザーの同意）］をチェックして、このスコープに対するユーザーの同意を必須にします。
  5. オプションです。［User Consent（ユーザーの同意）］を選択した場合に、［Block services from requesting this scope（サービスがこのスコープを要求するのをブロック）］のチェックボックスのチェックを解除します。

    チェックを外すと、ユーザーがアプリケーションを操作するときに同意が必要になりますが、スコープがサービスアプリケーションから認証サーバーに直接要求される場合は除きます。詳細については、Flexible consent（柔軟な同意）をご覧ください。

  1. Oktaに、権限付与リクエストでスコープを指定しないアプリに権限付与リクエストを許可することを許可する場合は、デフォルトのスコープを選択します。 クライアントが権限付与リクエストでスコープパラメーターを省略した場合、Oktaはアクセス・ポリシー・ルールで許可されているアクセストークンのすべてのデフォルトスコープを返します。

  2. [保存]をクリックします。

これらのスコープは、クレームダイアログにより参照されます。

注

User Consent for OAuth 2.0 and OpenID Connect Flows機能を使用するアプリを作成する場合は、以下に示すようにそのスコープで［User Consent（ユーザーの同意）］を［Yes（はい）］に設定します。

クレームの作成

トークンにはクレームが含まれます。クレームは、（名前、ロール、メールアドレスなど）そのサブジェクトや別のサブジェクトに関するステートメントです。

IDトークンのクレームは動的です。アクセストークンや認証コードで要求された場合、認証サーバーはデフォルトでクレームをIDトークンに含めません。このデフォルトを使用する場合は、［Include in token type（トークンタイプに含める）］で［When Requested（リクエストされた時）］を選択します。

IDトークンのクレームをIDトークンに常に含めるように認証サーバーに強制するには、［Include in token type（トークンタイプに含める）］で［Always（常に）］を選択します。

注

クレームが含まれていない場合、クライアントはアクセストークンを使用してクレームをUserInfoエンドポイントから取得する必要があります。

OpenID Connect用のIDトークンクレームか、OAuth 2.0用のアクセストークンを作成します：

  1. 表示する認証サーバーの名前を選択してから、［Claims（クレーム）］を選択します。Oktaが、デフォルトのサブジェクトクレームを提供します。マッピングの編集や、独自のクレームの作成ができます。

  2. ［Add Claim（クレームの追加）］を選択して、必須の情報を入力します。

    • Name（名前）
    • Include in token type（トークンタイプに含める）：1番目のドロップダウンボックスでトークンタイプを選択します。アクセストークン（Oauth 2.0）またはIDトークン（OpenID Connect）を選択します。2番目のドロップダウンボックスで、［Always（常に）］または［Userinfo/id_token request（ユーザー情報/IDトークンの要求）］を選択します。
    • Value type（値タイプ）：クレームをグループフィルターで定義するか、Okta式言語で記述された式で定義するかを選択します。
      • Mapping（マッピング）：このオプションは、前のフィールドで［Expression（式）］を選択した場合に表示されます。appuser.usernameなどのOktaの式言語を使用してここにマッピングを追加します。式が想定通りの結果を返すことを確認してください。この式はここで検証されません。
      • Filter（フィルター）：このオプションは、前のフィールドで［Groups（グループ）］を選択した場合に表示されます。これを使用して、グループフィルターを追加します。空欄のままにすると、このクレームですべてのユーザーが指定されます。
    • Disable claim（クレームを無効化）：テストやデバッグのためにクレームを一時的に無効化する場合は、このオプションをオンにします。
    • Include in（含める）：クレームを任意のスコープに対して有効にするように指定するか、有効にするスコープを選択します。

クレームリストでは以下を行えます：

  • タイプごとにクレームを並べ替える。

  • 自身で作成したクレームを削除するか、テストやデバッグ用にクレームを無効化する。

アクセス・ポリシーの作成
  1. 認証サーバーの名前を選択します。
  2. ［Access Policies（アクセス。ポリシー）］>［Add New Access Policy（新しいアクセス・ポリシーを追加）］の順に選択します。
  3. 必須の情報を入力します：
    • Name（名前）
    • Description（説明）
    • ［All clients（すべてのクライアント）］に割り当てるか、［The following clients:（次のクライアント：）］を選択して、このアクセス・ポリシーでカバーされるクライアントの名前を入力します。

アクセス・ポリシー・リストでは、次のことができます：

  • アクセスポリシーをアクティブに設定する、テストやデバッグ用にアクセスポリシーをディアクティベートする。
  • ポリシーを並べ替えて、ポリシーが評価される順番を変更する。

ポリシーのルールと同様に、ポリシーは優先順位に従い評価されます。クライアントリクエストに最初に一致したポリシーとルールが適用され、それ以上のルールやポリシーは処理されません。

各アクセス・ポリシーのルールの作成

ルールは、クライアント、ユーザー、カスタムスコープのマッピングを管理します。たとえば、アクセス・ポリシーのルールを指定して、ユーザーがクライアントに割り当てられた場合にカスタムスコープScope1が有効になるようにできます。

  1. 認証サーバーの名前を選択して、［Access Policies（アクセスポリシー）］を選択します。
  2. アクセスポリシーの名前を選択して、［Add Rule（ルールを追加）］を選択します。

    SAML 2.0アサーション付与タイプは早期アクセス機能です。これを有効にするには、早期アクセス機能とベータ機能の管理 の説明に従い早期アクセス機能マネージャーを使用します。詳細については、SAML 2.0アサーションフローの実装をご覧ください。

  3. 必須の情報を入力します：
    • Rule Name（ルール名）
    • IF Grant type is – クライアントがそれ自体に代わり機能するか、ユーザーに代わり機能するかを選択します。両方選択することもできます。クライアントがユーザーに代わり機能している場合は、いずれかまたはすべてを選択します。
    • AND User is – このオプションは、上で［Client acting on behalf of a user（ユーザーの代わりに機能するクライアント）］を選択した場合にのみ表示されます。［Any user assigned to the app（このアプリに割り当てられた任意のユーザー）］を選択することも、さらにユーザーを定義することもできます。
    • AND Scopes requested – ユーザーが条件のいずれかを満たした場合に付与されるスコープ（任意のスコープか、指定したリスト）を選択します。
    • THEN Use this inline hook – 該当する場合は、インライン・フックを選択します。インラインフックをご覧ください。
    • AND Access token lifetime is – アクセストークンが期限切れになるまでの期間を選択します。
    • AND Refresh token lifetime is – 更新トークンが期限切れになるまでの期間を選択します。

      ［Refresh token lifetime（更新トークンのライフタイム）］で、指定されたライフタイムを検証して継続するためにトークンを使用する必要がある期間を入力します。

      注

      期限切れになるまでの期間は、アクセストークンと更新トークンのライフタイムよりも短くする必要があります。期限切れになるまでの期間は最大で5年です。有効期限内に使用されなかったトークンは、ライフタイムがUnlimited（無期限）に設定されていたとしても期限切れになります。

  4. ［Create Rule（ルールを作成）］を選択してルールを保存します。

アクセス・ポリシーのルールリストでは、次のことができます：

  • テストやデバッグ用にルールを非アクティブに設定する。

  • デフォルトポリシーのデフォルトルール以外のルールを並べ替える。ルールは優先順位に従い評価されます。クライアントリクエストに最初に一致したポリシーの最初のルールが適用され、それ以上は処理されません。

    注

    サービスアプリケーション（クライアント資格情報フロー）にはユーザーが含まれません。このフローを使用する場合は、No user（ユーザーなし）の条件を指定するルールが少なくとも1つあることを確認します。

  • 情報アイコンを選択するかルール名をクリックすると、以下に示すように、ルールが適用されるユーザーとグループと、そのユーザーとグループに付与されるスコープが表示されます。

認証サーバーの構成のテスト

認証サーバー、スコープとクレーム、クライアント、およびユーザーの各組み合わせごとに、API呼び出しを発行し、IDトークンやアクセストークンの内容が想定どおりであることを確認します。

OpenID Connect IDトークンまたはOAuth 2.0トークンのトークンプレビュー

OpenID Connect IDトークンやOauth 2.0アクセストークンを使用するようにアプリケーションや統合を構成すると、多くの手間やエラーが発生する可能性があります。Oktaでは、以下に示すように構成設定を選択して、認証サーバーのページの［Token Preview（トークンプレビュー）］タブで結果のトークンを簡単に確認できるようになりました。

左側に値を追加して、右側のトークンがどうなるかを確認します。User以外のフィールドはすべて選択ボックスです。Userで、最初の数文字を入力して選択するユーザー名を表示します。

さまざまな値の組み合わせを試して、結果のトークン（およびエラーメッセージ）を確認できます。適切な組み合わせを見つければ、認証サーバーやその他のコンポーネントの構成が簡単になります。

認証サーバーの削除

認証サーバーを削除するには、認証サーバーの名前の横にある［Actions（アクション）］ボタンをクリックしてから、［Delete（削除）］をクリックします。

認証サーバーの検索


検索フィールドを使用して、認証サーバーやリソースURIの正確な名前で検索します。

手動によるキーのローテーション

キーを手動でローテーションするように認証サーバーを設定できます。キーはデフォルトで自動的にローテーションされます。

重要：自動キーローテーションは、手動キーローテーションより安全です。自動キーローテーションを使用できない場合のみ、手動キーローテーションを使用します。

認証サーバーの構成を変更するには：

  1. Okta組織にログインします。
  2. 管理者ダッシュボードから、［Security（セキュリティ）］>［API］に進みます。
  3. 編集のために認証サーバーを開きます。
  4. ［Signing Key Rotation（署名キーのローテーション）］の値を［Manual（手動）］に変更して保存します。
  5. 認証サーバーの［Settings（設定）］タブで、［Rotate Signing Keys（署名キーのローテーション）］ボタンをクリックしてキーを手動でローテーションします。［Signing Key Rotation（署名キーのローテーション）］が［Automatic（自動）］に設定されている場合、このボタンは表示されません。
