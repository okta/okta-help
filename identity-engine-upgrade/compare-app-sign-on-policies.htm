The most significant addition to policies is the assurance requirement: users who meet all conditions of an authenticate into the app only with the proper assurance requirements. Other enhancements include risk scoring and custom expressions, which ensure that users meet customized security levels before they are allowed access.

to migration scenarios

When you set up an app sign-on policy in , all Client options are selected by default. You add rules based on who the users are, which groups they belong to, their device's client and platform, and whether they are on or off network.

When you upgrade to , your app sign-on policies migrate with a few conditions.

Policy names

In , migrated policies are named for their applied app (<appname> policy). If multiple apps had identical policies in , those policies are consolidated into one shared policy. The consolidated policies use the same name structure, with the prefix "Merged" ([Merged] <appname1>, <appname2>, <appname3> policy).

Default policies

Apps in that used the default app sign-on policy migrate to an Okta with the default catch-all rule enabled. Apps that use only the catch-all rule allow access to any user who is assigned to the app, without additional authentication (beyond what is required in the ).

  • IF User's type is: Any
  • AND User's group membership is: Any
  • AND Device identity is: Any
  • AND User's IP is: Any zone
  • AND Risk: Any
  • AND The following custom expression is true: N/A
  • THEN Access is: Allowed
  • AND User must authenticate with: Any 1 factor type 
  • AND Possession factor constraints are: N/A
  • user interaction: Always required
  • Password re-authentication frequency is: Once per device

Every app in Okta has the catch-all rule in its policy. The catch-all rule can be edited, but for more customized access you can add more rules and prioritize them.

Re-authentication + factor verification

app sign-on policies with re-authentication and factor verification enabled migrate to an Okta with password and any additional factor enabled. Re-authentication frequency is determined by the duration in the policy. For example, a  policy that requires password re-authentication every two hours and a prompt for factor every time migrates to an policy rule of:

  • IF User's type is: Any
  • AND User's group membership is: Any
  • AND Device identity is: Any
  • AND User's IP is: Any zone
  • AND Risk: N/A
  • AND The following custom expression is true: N/A
  • THEN Access is: Allowed
  • AND User must authenticate with: Password + Another factor
  • AND Possession factor constraints are: N/A
  • user interaction: Always required
  • Password re-authentication frequency is: 2 hours
  • Re-authentication frequency for all other factors is: Every sign-in attempt

Multifactor with no re-authentication

app sign-on policies that require multifactor every time with no re-authentication migrate to an Okta that requires a possession factor for authentication with a re-authentication interval equivalent to the settings.

  • IF User's type is: Any
  • AND User's group membership is: Any
  • AND Device identity is: Any
  • AND User's IP is: Any zone
  • AND Risk: N/A
  • AND The following custom expression is true: N/A
  • THEN Access is: Allowed
  • AND User must authenticate with: Possession factor
  • AND Possession factor constraints are: N/A
  • user interaction: Always required
  • Re-authentication frequency for all other factors is: Every sign-in attempt
