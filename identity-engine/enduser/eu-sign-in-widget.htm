New user registration, activation, and sign-in experience

New users register and activate their profiles in the Sign-In Widget, allowing you to fully customize and brand the onboarding experience.

The Sign-In Widget varies by use case, according to the policies you've set for your org.

Self-registration

If your org has a profile enrollment policy for self-registration and the end user enters a username that doesn't exist in the org, they are prompted to sign up and register for an account.

  1. End users enter their full app Username, including the domain, and then click Next.
  2. The Sign-In Widget displays this message: "There is no account with the Username. Sign up for an account."
  3. End users click Sign Up to launch the Create Account registration form.

If self-service registration is disabled and the end user enters a username that doesn't exist in the org, they are prompted to enter a password but aren't permitted to sign in. In this scenario, the option to sign up for an account is not displayed.

See Configure profile enrollment policies.

Sign-in flows

The sequence of a sign-in flow depends on the authentication assurance requirements that you set in your policies. Remember that requires that the authentication assurance specified in both the Okta and are satisfied before it allows the end user to access an app.

Sign-on policies supply the context necessary for the user to advance to the next step and specifies actions to take, such as allowing access, prompting for a challenge, and setting the time before prompting for another challenge. The defines access globally, across all apps. The enforce end user authentication in the context of the requested application. The user's location and profile are verified against both policies' group membership and authentication criteria.

Note the following about the end-user sign-in experience:

  • When a user is allowed to sign in without a password, optimizes the sign-in experience. If the device is enrolled with Okta Verify and a biometric authenticator is enabled, the biometric authenticator is always the first factor used for user authentication.
  • If assurance requires end users to sign in with a password, the password prompt will always be displayed first. This corresponds to any configuration where the and password authenticator plus other factor types are defined in the sign-on policy.

Password / IDP / any factor allowed by app sign on rules

If any of your rules has the primary factor set to Password / IDP / any factor allowed by app sign on rules, end users see the identifier-first Sign-In Widget as the first screen during their access flow.

  1. End users enter their full app Username, including the domain, and then click Next. Keep me signed in retains their identifier as well as authenticator verification information on their device for the amount of time designated by the sign-on policy. This replaces the Remember me option in orgs.
  2. On the security method page, end users select one of the primary authenticators allowed by the combined Okta sign-on and .
  3. After they've selected an authenticator, end users move to the verification step. End users supply the required authenticator and then click Verify.

Password/Okta Verify with user verification (FastPass)

If any of your Okta sign-on policy rules require Okta Verify with user verification, end users see the Identifier-first sign-in widget as the first screen during their access flow.

  1. End users see both an Okta Verify prompt and Username and Password fields. They must select one method on the sign-in widget; either they click Sign in using Okta Verify on this device, or they enter their full app username, including the domain, in the Username field, provide their password in the Password field, and then click Sign in. Selecting Keep me signed in retains their identifier as well as authenticator verification information on their device for the amount of time designated by the sign-on policy. This replaces the Remember me option in orgs.

  2. End users who click Okta Verify see the Okta Verify prompt and do not see a list of other authenticators. End users who enter their username and password see a list of authenticators.

  3. End users supply the required authenticator and click Verify.

Password / IDP

If any of your rules has the primary factor set to Password / IDP, end users see the password-first Sign-In Widget.

  1. End users enter their full app Username, including the domain, and then click Next. Keep me signed in retains their identifier as well as authenticator verification information on their device for the amount of time designated by the sign-on policy. This replaces the Remember me option in orgs.
  2. End users enter their password in the Password field, and then click Next.
  3. If either of the sign-on policies required a secondary factor, end users make a selection on the security method page. End users then complete the verification step.

Remember user on sign-in

Admins can configure the Remember user on sign-in feature in Security General Organization Settings. Remember user on sign in pre-populates the Username field on the Sign-In Widget each time the widget is shown for the lifetime of the browser window or until the user clears their cookies. Remember user on sign in doesn't keep the user signed in or store MFA preferences.

    First sign-in attempt on a new device

    All sign-in attempts from new devices follow the same process:

    • If your org uses email for SSO, the email and password sign-in options are always available to users, even when the user profile does not exist or the user can't sign in.
    • The user can't sign in if authentication errors occur.
    • If your org doesn't use email for SSO, the user is always prompted for a password.
    • When the same user attempts to sign in again from the browser, a full list of available authenticators is displayed.

    Email used for SSO

    If your org has email enabled for SSO, end users who try to access an app on a new device see the identifier-first Sign-In Widget.

    1. The end user enters their full app Username, including the domain, and then clicks Next.
    2. If the end user's profile exists, the Security Method page displays Email and Password options.
      1. If the end user selects Password, they enter their password and then click Next.
      2. The Sign-In Widget prompts the end user for Email verification.
      3. The end user selects Email, and then authenticates by clicking a link provided in the email.
    3. If the end user's profile doesn't exist, the Security Method page displays Password as the only option.
      1. The end user enters their password and then clicks Next.
      2. The Sign-In Widget displays the message "Unable to sign in."

    Email not used for SSO

    If your org does not use email for SSO, existing end users who try to access an app on a new device see the identifier-first Sign-In Widget.

    1. End users enter their full app Username, including the domain, and then click Next. Keep me signed in retains their identifier as well as authenticator verification information on their device for the amount of time designated by the sign-on policy. This replaces the Remember me option in orgs.
    2. On the security method page, end users select one of the primary authenticators allowed by the combined Okta sign-on and .
    3. The Sign-In Widget displays an authenticator error.

    Expected behavior

    If an org upgrades from to while any of the following flows are ongoing, end users will complete the flow in the Classic Sign-In Widget:

    • Standard sign-in flow
    • Self-service password recovery
    • Factor enrollment during sign-in flow

    Subsequent instances of these flows will occur entirely in the Sign-In Widget.

    When an org upgrades to , some end users may experience a delay in Sign-In Widget refresh. End users will continue to see the Sign-In Widget:

    • If their org is upgraded while they are signing in.
    • If their org is upgraded while they are recovering their password.
    • If their org is upgraded while they are enrolling in a factor during sign-in.

    Subsequent instances of these flows will occur entirely in the Sign-In Widget.


    When email is configured as a required policy factor, the user can either authenticate using a code provided in the email or click a link.

    • If no subsequent factors are required, the link signs the user in automatically, as long as the user clicks it in the same browser session.
    • If the user opens a link in a separate browser session (or on a different device entirely), the user is asked to return to the original browser session to be signed in.
