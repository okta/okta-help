Configure the Password authenticator

The Password authenticator enables you to require users to provide a password when signing in to or one of your apps. You can also allow users to reset forgotten passwords.

Passwords are strings of characters that the user types into a password field on the . Admins can customize the required complexity of passwords, configure their age, expiry and history, and enable lockout conditions.

The Password authenticator is active by default. To use the Password authenticator, you configure a password policy and rules for it.

Add a password policy rule

Password policy rules allow you to configure:

  • Which self-service actions are available to users.
  • How users can initiate recovery.
  • Whether additional verification is required during recovery.
  • Which individual users are excluded from the password policy.
  • Restrictions on users attempting to access from network zones that you specify.
  2. On the Setup tab, find the Password item in the list of authenticators and click Actions Edit .
  3. Select the password policy to which you want to add a rule and click Add Rule.
  4. Complete the following fields and options:
    • Rule Name: Type a name for the rule.
    • Exclude Users: Start typing the name of a user that you want to exclude from the rule. As you type, suggests usernames that match your text. Select the desired user from the list. Repeat for each additional user you want to exclude.
    • IF User's IP is:
      • Anywhere: Apply the rule to all users regardless of whether their IP address is listed in the Public Gateway IP list.
      • In zone: Apply the rule to users in all or specific network zones.
        • All Zones: Select this checkbox to apply this rule to users in all zones.
        • Zones: Apply this rule to users in network zones that you specify. Start typing the name of a zone. As you type, suggests zone names that match your text. Select the desired zone from the list. Repeat for each additional zone you want to add.
      • Not in zone: Apply the rule to exclude users in all zones or in specific zones.
        • All Zones: Select this checkbox to exclude users in all zones.
        • Zones: Apply this rule to exclude users in network zones that you specify. Start typing the name of a zone. As you type, suggests zone names that match your text. Select the desired zone from the list. Repeat for each additional zone you want to add.

        • See Network zones for information on the Public Gateway IP list and other IP Zones features.

    • THEN User can perform self-service:
      • Password change (from account settings): Allow users to change their password and make the perform self-service password reset option available to them.
      • Password reset: Allow users who are unable to sign in or who forgot their password to perform self-service password resets and show the Forgot password? link on the .
      • Unlock account: Allow users to unlock their account by clicking the Unlock account? link on the .

        • When you select this option, LDAP-sourced user accounts are unlocked in but remain locked in the on-premises LDAP instance. If you don't allow self-service unlock, see Reset a user password for other options.

  5. Configure ways for users to initiate recovery and provide additional verification:
    • AND Users can initiate recovery with:

      • Phone (SMS / Voice call): Allow users to initiate recovery with either text messages or voice phone calls. See Configure the Phone authenticator.
      • Email: Allow users to initiate recovery with an email message that contains a one-time password or a magic link. See Configure the Email authenticator.

About lockouts

AD / LDAP Softlock

can help prevent AD and LDAP-sourced users from getting locked out of their Windows account and hardware device due to too many failed sign-in attempts. This feature also helps prevent a malicious third party from using to lock out users.

To prevent Active Directory and LDAP lockouts, make sure that the number entered for Lock out user after # unsuccessful attempts is lower than the failed sign-in attempt limit configured in AD and LDAP. For example, if the maximum number of failed Windows sign-in attempts is set to 10 in AD and LDAP, recommends setting your maximum failed sign-in limit to 9. If a user exceeds the sign-in limit set in , additional failed attempts aren't sent to AD or LDAP. This prevents users from locking themselves out of their Windows account. In AD, locked out users can use self-service account unlock or seek help from an admin. Only an admin can unlock LDAP-sourced accounts.

Detect lockouts caused by unknown devices

See Detect lockouts caused by unknown devices for instructions.

You can also allow individual users to sign in using an unknown device. This is helpful to users who may have lost their previous device and need to sign in with a new, unknown device for the first time. See Allow unknown devices to sign in for instructions.

End-user experience

End users create an password according to the syntax, minimum length, age, and history requirements in your password policy. Unless an rule for passwordless authentication is enabled, end users are always prompted for a password.

Current limitations

  • AD users aren't supported.
  • You can't delegate authentication to AD.
  • You can't send a warning message to users before their password expires.
  • You can't configure the expiration time for reset and unlock account recovery emails.
