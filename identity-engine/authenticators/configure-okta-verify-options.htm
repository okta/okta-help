Configure options

After you add as an authenticator, you configure how users interact with during enrollment, or when they authenticate. You can also enable passwordless authentication with .

Before you begin

  • If you activate push notification with number challenge, use the 3.3.0 or a later version. If your org calls the Authentication API directly, update your code to handle the number challenge API response. See Response example (waiting for 3-number verification challenge response).
  • If your users are behind a firewall that restricts traffic to or from the internet, they might not receive push notifications. Open ports 5228, 5229 and 5230 on the firewalls to allow connectivity with Google Firebase Cloud Messaging. Configure the firewall to accept outgoing connections to all IP addresses in the IP blocks listed by Google in ASN of 15169.
  • For security reasons, doesn't allow inspection or modification of traffic between and its endpoints. If you use an SSL proxy, exclude your organization's default domains from inspection. Typically domains are *.okta.com or *.oktapreview.com. For a complete list of domains, see Allow access to [%=okta-variables.ProductName%] IP addresses.
  • If you restrict access to Federal Information Processing Standard (FIPS)-compliant devices, Android users must enable a secure PIN on their devices to make them Federal Identity, Credential, and Access Management (FICAM)-compliant. Otherwise, they can't access your org. Some Android hardware isn't certified as FIPS-compliant. Consider the security implications of using hardware keystores against the need for FIPS compliance.
  • Configuring enrollment options is an Early Access feature. To enable it go to Settings and click the toggle for Block non-phishing-resistant enrollment methods for Okta Verify.

User verification Select if users must provide biometrics during authentication with . After setting up biometrics, users receive a biometric prompt each time they sign in with an push notification.

Push notification (number challenge) Choose whether to include a number challenge with an push notification. The number challenge verifies that a sign-in attempt to an app protected by came from the intended user and not from an unauthorized person. It presents a number in the Sign-In Widget and pushes a notification to on the user's mobile device. The user selects the number that matches what they see in the Sign-In Widget. If the selection is correct, the user can access the protected app. The number challenge helps prevent phishing by ensuring that the user possesses both and the device initiating the sign-in attempt. See the end-user documentation: Sign in with an Okta Verify push notification (iOS) or Sign in with an Okta Verify push notification (Android).

  • Never: Users never receive a number challenge regardless of the risk level of the authentication attempt.

  • Only for high risk sign-in attempts: Users receive a number challenge only if the sign-in attempt is assessed to be a risk. Admins must configure sign-on policy rules. See About Risk Scoring.

  • All push challenges: Users receive a number challenge with all push notifications regardless of risk level.

FIPS Compliance Restrict enrollment to FIPS-compliant Android or iOS devices. When this option is enabled, uses FIPS 140-2 validation for all security operations. also meets FedRAMP FICAM requirements by relying on FIPS-validated vendors.

  • FIPS compliant devices only: Users may only enroll a FIPS-compliant device in .

  • Any device: Users may enroll any device in Okta Verify.

About risk scoring

You can combine number challenges with Risk Scoring to increase the security of sign-ins to your org. assesses risk based on multiple criteria, including details about the device and its location. When enabled, Risk Scoring assigns a risk level to each sign-in, and admins can configure a sign-on policy rule to take different actions based on the risk level of the sign-in, such as prompting for multifactor authentication if the sign-in is considered high-risk. See Risk Scoring for instructions.

Known limitations

  • and push notifications aren't available for iPod Touch devices.
  • Authentication with biometrics isn't supported on Apple Watch.
  • For Android devices, only biometric methods classified by Google as Class 3-Strong (facial and fingerprint recognition) are supported.
  • Biometrics aren't supported on Android 12 if is installed on the work profile. End users receive a Keystore not initialized error and they can't enable biometrics. To unblock affected users, set User verification to Preferred, and then advise end users to skip the biometrics enablement step.
  • Push notification: Number challenge isn't supported in LDAPi and RADIUS environments. The three-number challenge appears in the app but the matching number doesn't appear in the end user's desktop browser. In this case, configure an MFA authenticator other than .
  • Multiple user profiles aren't supported on a single macOS or Windows device if you select as a verification option.

  • authentication doesn't function properly if HTTP Strict Transport Security (HSTS) is enabled for loopback. Users that develop, host, or debug websites locally often enable this option. If your organization doesn’t require HSTS for security reasons, advise your users to remove the URL from the list of domains that require HSTS. Consult your browsers' documentation for instructions and share them with your users.
