コネクターのドキュメントの準備

これは既存のWorkflowsユーザー向けの早期アクセス機能です。この機能を有効にするには、「早期アクセスおよびBeta機能の管理」で説明したように、早期アクセス機能マネージャーを使用します。

すべてのWorkflowsコネクターにはユーザードキュメントが必要であり、コネクター提出の一部として添付する必要があります。Okta統合ネットワークは、Okta検証プロセスの一環として、提出されたコネクターへの一般的な準拠をドキュメントで確認します。コネクターが公開され、Okta Workflowsで利用できるようになると、ドキュメントリンクを設定してお客様や一般が利用できるようにしたり、利用をコネクターユーザーに制限したりできます。

このようなコンテンツは、お客様がコネクターの構成と使用の方法を理解し、コネクターのアクションカードの入出力フィールドについて学び、構築した1つ以上のアクションカードが含まれるフローを実行する際に生じ得るパフォーマンス上の問題を予測する上で役立ちます。

コネクター用に提供されるドキュメントには、その内容が適用されるコネクターのバージョン番号が明示的に記載されている必要があります。コネクターの新バージョンが本番向けにリリースされた場合、新バージョンの機能更新に加えて、新しいバージョン番号をドキュメントセットに明確に記載する必要があります。

Workflows製品内のヘルプペインからユーザー提供のドキュメントサイトにアクセスできるようになります。ユーザーがカードやその他のダイアログでヘルプアイコンをクリックすると、ヘルプペインが表示され、外部でホスティングされているコネクターのヘルプドキュメントへのリンクが表示されます。

コネクターのユーザードキュメントは、HTML、Google Doc、Microsoft Word文書、PDFなど、Workflowsユーザーがホスティングしてアクセスできる形式であれば何でもかまいません。

コネクターのドキュメントセットには、次のアセットが含まれている必要があります。

  • 承認

  • カスタムAPIアクションカードを含め、各アクションカードのヘルプトピック

  • オプションとして、トラブルシューティング、FAQ、コネクターとそのカードのユーザー向けのその他のガイダンスのヘルプトピック

承認

Authorization（承認）トピックには、ユーザーが接続を作成および再認証する上で役立つ、あらゆる関連情報を含める必要があります。

ユーザーが作成者のコネクターを初めてフローに追加すると、接続の構成が求められます。この接続により、ユーザーは自分のアカウントまたは環境に接続し、資格情報を保存し、作成者のコネクターカードが含まれる新しいフローを構築する際に接続を再利用できます。

前提条件

このセクションには、ユーザーが作成者のコネクターへの接続を構成する前に完了しなければならない前提条件を指定します。たとえば、ユーザーがアカウントに接続する前にAPIキーを生成したり、OAuthアプリケーションを構成したりする必要がある場合、その情報を前提条件としてコネクターのAuthorization（承認）トピックに指定する必要があります。次に、Prerequisites（前提条件）セクションの例を示します。

APIトークンを生成する

エンタープライズAPIへの認証は、インスタンスレベルのベアラートークンを介して制御されます。生成方法：

  1. ログインして［Settings（設定）］をクリックします。

  2. ［Access（アクセス）］タブをクリックします。

  3. インスタンスに固有のAPI URLを特定します。

  4. ［Add（追加）］をクリックし、新しいAPIキーを作成します。

  5. APIトークンの名前を［Name（名前）］に指定します。

  6. ［Save（保存）］をクリックします。APIトークンを提供する新しいダイアログが表示されます。

  7. APIトークンが表示されるように可視性記号をクリックします。

  8. APIトークンをクリップボードにコピーするには、［Copy（コピー）］をクリックします。

  9. ［Close（閉じる）］をクリックします。

Set up a connection（接続のセットアップ）

Authorization（承認）トピックには、コネクターのユーザーがサービスへの接続を作成する方法について説明した一連の手順を含める必要があります。次に、［Set up（セットアップ）］セクションの例を示します。

  1. ［Connections（接続）］ページまたは任意のカードの［New Connection（新規接続）］をクリックします。［New Connection（新規接続）］ダイアログが表示されます。

  2. ［Connection Nickname（接続ニックネーム）］ダイアログに文字列を追加します。これは、チームで共有するために複数の接続の作成を計画している場合に便利です。

  3. ［Set up（セットアップ）］セクションで作成したAPIトークンを、ダイアログの［API Token（APIトークン）］フィールドに貼り付けます。

  4. ［Create（作成）］をクリックします。

サポートされているスコープ

OAuthまたはトークンベースのその他の認証では、コネクターの使用に必要なスコープのリストをドキュメントに記載する必要があります。セットアップの詳細は、［Prerequisites（前提条件）］および［Set up（セットアップ）］セクションに記載します。次に、［Supported scopes（サポートされるスコープ）］セクションの例を示します。

  • https://www.googleapis.com/auth/userinfo.profile

  • https://www.googleapis.com/auth/userinfo.email

  • https://www.googleapis.com/auth/gmail.settings.basic

すべてのコネクターカードの参照ヘルプ

コネクターのすべてのカードは、カードの目的を説明したり、カードのフィールドを定義したりするための参照ヘルプトピックでサポートされる必要があります。

Name and description（名前と説明）

カードのヘルプトピックには、カード名が明記され、カードによって実行できるアクションを説明する簡潔な命令文などの簡単な説明が記載されている必要があります。この情報は、カードに関連付けられているアクションフローの［Name and description（名前と説明）］セクションに追加した情報と一致させることができます。ここには、カード固有のその他の情報も含めることができます。

Definitions for fields on a card（カードのフィールドの定義）

同じヘルプトピックには、カードのすべてのオプション、入力、出力フィールドの情報が含まれている必要があります。フィールドごとに、以下を提供します：

  • Label（ラベル）：フィールドの名前。

  • Definition（定義）：フィールドの目的を説明します（このフィールドに入力または出力として渡されると考えられるデータなど）。

  • Type（型）：このフィールドで許可されるデータ型を指定します。たとえば、Number（数値）、List of Text（テキストのリスト）などです。

  • Required（必須）：フローの正しい実行のために、このフィールドに値を追加する必要があるかどうかを示します。

オプションフィールドについては、オプションフィールドのすべての選択肢をヘルプトピックに記載する必要があります。

Groups of fields（フィールドのグループ）

コネクターカードでは、すべてのオプション、入力、出力はグループ化されます。

オプションについては、グループ名にOptionsというラベルが付けられ、変更できません。入力と出力については、すべてのフィールドは1つ以上のグループに集約されている必要があり、グループ名のラベルは変更できます。ヘルプトピックには、すべてのグループラベルが含まれている必要があります。

Limitations and known issues（制限事項と既知の問題）

同じヘルプトピックには、そのカードの制限事項や既知の問題についての情報も含まれている必要があります。たとえば、カードが大量のレコードを処理する場合、最適なパフォーマンスが得られない可能性がある場合、またはオプションフィールドのドロップダウンがすべての選択肢のサブセットしか表示できない場合、ヘルプトピックにそのような情報を明確に記載する必要があります。

コネクターカードの参照ヘルプトピックの例をご覧ください。

  • Jiraコネクターの［Search Issues（課題の検索）］

  • Slackコネクターの［Send File to User（ユーザーへのファイルの送信）］

カスタムAPIアクションカードの参照ヘルプ

参照ヘルプトピックは、コネクターのカスタムAPIアクションカードもサポートする必要があります。

次の文章でトピックを紹介します。

カスタムAPIアクションを使用して、<サービス名> REST APIに対する認証済みのカスタムAPIリクエストを発行する。

オプション

［Request Type（リクエストタイプ）］フィールドには、必要に応じてこれらの定義を使用して、カスタムAPIアクションカードでサポートされるメソッドを指定します。

GET パラメーターに応じてWebサーバーからデータを取得します。このメソッドは指定されたリソースの表現を要求します。リクエストが成功した場合、要求したコンテンツとともに200 (OK)レスポンスメッセージが返されます。
POST パラメーターに応じてWebサーバーにデータを送信します (たとえば、ファイルのアップロード)。複数のPOSTリクエストは、単一のPOSTとは異なる結果になる場合があります。意図せず複数のPOSTリクエストを送信することを避けるために注意する必要があります。リクエストが成功した場合、200 (OK)レスポンスメッセージが返されます。
PUT パラメーターに応じてWebサーバーの特定の場所に格納されるデータを送信します (たとえば、ファイルのアップロードなど)。POSTリクエストとは異なり、PUTリクエストは冪等（べきとう）です。リクエストが成功する場合、単一のPUTリクエストの結果は同一のPUTリクエストを何回行っても同じです。リクエストが成功した場合、200 (OK)、201 (Created)、または204 (No Content)レスポンスメッセージを受信します。
PATCH パラメーターに応じてWebサーバー上のリソースに部分的な変更を適用します。PATCHは冪等（べきとう）ではありません。複数のPATCHリクエストによって意図しない結果が生じる可能性があります。PATCHが成功した場合、200 (OK)または204 (No Content)レスポンスメッセージを受信します。
DELETE パラメーターに応じてWebサーバーから特定のリソース（存在する場合）を削除します。DELETEが成功した場合、200 (OK)レスポンスメッセージが返されます。

入力

入力フィールドセクションには、次のフィールドの定義を指定します。

  • Relative URL（相対URL）

  • Headers（ヘッダー）

  • Query（クエリ）

  • Body（本文）

次に、Google CalendarコネクターのカスタムAPIアクションカード向け相対URLの定義の例を示します。

対話を試みるWebサーバーのURLアドレス。/{version}/{insert_remaining_URL}のように相対URLを指定します。「?」を使用して相対URL内にクエリパラメーターを指定したり、クエリ入力内のオブジェクトキーペアとしてクエリパラメーターを指定したりすることができます。

たとえば、Google Calendar APIエンドポイントの場合、
https://www.googleapis.com/calendar/v3/users/{userId}/calendarListであれば、相対URLは/v3/users/{userId}/calendarListとなります。

出力

出力フィールドセクションには、次のフィールドの定義を指定します。

  • Status Code（ステータスコード）

  • Headers（ヘッダー）

  • Body（本文）

次に、ステータスコードの定義の例を示します。

操作の結果。HTTPステータスコードはコネクタによって返され、カードによって実行されたアクションが成功したか失敗したかを示します。例：

  • 201 Createdステータスコードは、新しいリソースが作成された場合の成功を示します。

  • 403 Forbiddenエラーは、必要な権限が不足していたためにHTTPリクエストが処理されなかったことを示します。

発生する可能性があるステータスコードの完全なリストについては、「HTTPステータスコード」を参照してください。
