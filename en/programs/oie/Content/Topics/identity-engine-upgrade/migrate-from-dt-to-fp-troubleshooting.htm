Troubleshoot the migration of Device Trust to Okta FastPass

The following troubleshooting and verification procedures can be helpful when migrating from Device Trust (Classic Engine) to Okta FastPass.

"Deactivate this Device Trust (Classic Engine) configuration" error

If you try to deactivate Device Trust (Classic Engine), you will get one of these errors if an Okta Identity Engine app sign-on policy has not been configured correctly for devices that are not trusted. The error message varies, depending on the number of app sign-on policies that are affected.

or

For example, if your Device Trust (Classic Engine) app sign-on policy was configured as:

  • CLIENT: Windows and/or macOS
  • DEVICE TRUST: Not trusted
  • ACCESS: Denied

This policy migrated to Identity Engine as:

  • AND Device state is: Registered
  • AND Device management is: Not managed
  • THEN Access is: Denied

The following images illustrate these two configurations:

Classic Engine app sign-on policy

Okta Identity Engine app sign-on policy

Before you delete legacy Device Trust configuration, it is recommended that you revise the Okta Identity Engine app sign-on policy to deny access to devices that are not enrolled in Okta FastPass.

Take an allow-list approach to your app sign-on policy rules:

  1. Create one or more Allow rules to support the scenarios that will allow access to the app, and assign the rules the highest priority.
  2. Create a Denied catch-all rule that will apply to users who don't match the permissive scenarios you created in Step 1. Assign the Denied catch-all rule the lowest priority, just above Okta's Default Rule. In the allow-list approach described here, the Default Rule is never reached because it is effectively negated by the Denied catch-all rule.

Your first rule should allow registered, managed devices:

  • AND Device state is: Registered
  • AND Device management is: Managed
  • THEN Access is: Allow

A later rule should deny all other devices.

  • AND Device state is: Any
  • THEN Access is: Deny

Example 1:

Rule Rule name User group IF THEN
1 Group A managed A Require registered, managed devices Allow with 1FA
2 Group A other A Any device state Allow with 2FA
3 Group B managed B Require registered, managed devices Allow with 1FA
4 Catch-all Rule Any Any Denied

Troubleshoot Windows devices

Verify that Integrated Windows Authentication (IWA) is working as expected

On the client device, go to the following URLs (replace [IWA_DOMAIN_NAME] with your IWA domain name), and verify that they are working as expected:

  • http://[IWA_DOMAIN_NAME]/IWA/iwa_test.aspx

    • Verify that it displays correctly in the browser.

  • http://[IWA_DOMAIN_NAME]/IWA/authenticatd.aspx

    • Verify that the Userid, UserPrincipalname, UserprincipalName(Transformed), and Security Identifier display.

  • http://[IWA_DOMAIN_NAME]/IWA/auth.aspx

    • Verify that you are redirected to the Okta Sign-in page.


Verify that a Device Trust client is deployed to the client device
  1. In Windows, click Start > Add or remove programs.
  2. Search for Okta Device Registration Task.

Verify that a Device Trust mutual TLS client certificate is installed
  1. In the Microsoft Management Console (MMC), open the Certificate Manager (click Start > certmgr.msc).
  2. Verify that a certificate with subject Okta MTLS - [userName] is present.
  3. If the certificate was not present in step 2:
    1. Open Task Scheduler (click Start > Task Scheduler).
    2. Run okta-devicetrust-usertask.

Verify that Device Trust Enrollment is working as expected
  1. In the Microsoft Management Console (MMC), open the Certificate Manager (click Start > certmgr.msc).
  2. Delete the Okta MTLS - [username] certificate.
  3. Open a Command Prompt.
  4. Change directories to “Program Files\Okta\DeviceTrust”.
  5. Run OktaDeviceReg.exe --user --verbose --force.

    6. A new command window appears.

  6. Verify that errors do not exist, and that a new certificate is enrolled on the client device.

Verify that an app sign on policy rule requires devices to be registered and managed
  1. In the Admin Console, go to Security > Authentication Policy.
  2. Select the policy you want to update.
  3. Click the Rules tab.
  4. View the Sign On Policy information, and then verify that one of the rules in the policy specifies:
    • Device: Registered, Managed
    • User must authenticate with: Any 1 factor type

Review the Admin System logs

See the Troubleshooting > Basic Troubleshooting > SystemLog section of Enforce Okta Device Trust for managed Windows computers

If you are using a Trusted Platform Module (TPM), see Enhance Windows Device Trust security with Trusted Platform Module (TPM).

Troubleshoot macOS devices

Verify that a Device Trust client is deployed to the client device.
