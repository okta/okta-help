Application types

When creating an application configuration in Okta Access Gateway, there are various application templates available to speed the configuration process. The different application templates are described here for reference.

Application Types

Application type Description Configuration details

Sample Cookie

Out of the Box Access Gateway application based on cookies.

This sample is used to demonstrate passing iDPAn acronym for Identity Provider. It is a service that manages end user accounts analogous to user directories such as LDAP and Active Directory, and can send SAML responses to SPs to authenticate end users. Within this scenario, the IdP is Okta. data as cookie content. The sample application prints out all cookie data passed to it.

Sample Header

Out of the Box Access Gateway header web application type with IDCS attributes.

This sample is best used to demonstrate passing iDP data as headers to an application. The sample application will prints all the headers passed to it. For more information see Add a Sample Header Application.

Sample Policy

Out of the box Access Gateway Policy application type

This sample is best used to demonstrate how to create a set of sample policies and runs a series of tests against the example policies when tested. See Application Policy Overview for an overview of Policy and Administration User Policy Guide for complete details.

Sample Proxy

Out of the box Access Gateway Policy application type

This sample is based used to demonstrate a basic proxy application and when run displays a simulated access to th back end proxy host.

Header Based

A header web application type

This configuration is best used for applications which do not required specific settings. Access Gateway will provide SSOAn acronym for single sign-on. In a SSO system, a user logs in once to the system and can access multiple systems without being prompted to sign in for each one. Okta is a cloud-based SSO platform that allows users to enter one name and password to access multiple applications. Users can access all of their web applications, both behind the firewall and in the cloud, with a single sign in. Okta provides a seamless experience across PCs, laptops, tablets, and smartphones. via HTTP headers to this application. For more information see Add a Generic Header Application.

No-auth

No AuthenticationAuthentication is distinct from authorization, which is the process of giving individuals access to system objects based on their identity. Authentication merely ensures that the individual is who he or she claims to be, but says nothing about the access rights of the individual. Authentication methods and protocols include direct auth, delegated auth, SAML, SWA, WS-Fed, and OpenID Connect. Application

The no-auth configuration enables an application to be reverse-proxied by Access Gateway without any authentication. This is configuration is best used when exposing internal applications to the Internet. For more information see Add a no-auth app to Access Gateway.

Oracle Access Gateway

Oracle Access Gateway application type

The Oracle Access Gate app enables Access Gateway to SSO with Oracle E-Business applications. By default this application will pass the USER_NAME and USER_ORCLGUID headers to the Oracle Access Gateway. It is required that Access Gateway has been configured to enable SSO. Please contact Oracle for assistance with Access Gateway SSO configuration.

Oracle Agile PLM

Out of the Box Access Gateway Agile PLM application type

The Oracle Agile PLM app enables Access Gateway to SSO with Agile PLM applications. By default this application will pass the REMOTE-USER header to the Agile PLM. Please contact Oracle for assistance with Agile PLM SSO configuration.
