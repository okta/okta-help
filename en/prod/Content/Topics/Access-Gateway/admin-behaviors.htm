Administer Behaviors

Overview

The purpose of guide is to describe Access Gateway Behaviors and how they can be used and managed.

What’s covered in this guide

  1. Concepts
  2. Adding Behaviors

Concepts

Behaviors:

  • Provide a specialized mechanism for handling a variety of unusual or unexpected conditions.
  • Allow for the definition of specialized actions such as:
    • Logout - What to do when users log out.
    • Error - Define actions based on errors.
    • Policy - Define actions based on denying policy.
    • Inactive or offline applications - Define actions based on unavailable applications.
Adding Behaviors

To add a behavior:

  1. Navigate to your Access Gateway InstanceAn instance, or computer instance, is a virtual machine (VM) or individual physical computer, used to host a software appliance..

  2. From the Topology tab or the Applications tab open the application.
  3. In the Settings pane expand Behaviors.

Choose one of the following custom behaviors:

  • Login
  • Logout
  • Post logout URL
  • Error
  • No Session/Session Expired
  • Policy Denied
  • Session Integrity Failed
  • Application Maintenance
  • Application Inactive
  • Application Offline

Login

Allows you to define login endpoints that will aid in creating the user session.

Login supports the following:

Selection Description
Don't define login behavior Default. No specialized login behavior
Use Okta Access Gateway login page Okta Access Gateway will show the auth module login page when the Login path is called.
Use Application login page Unprotect path to application hosted login page. This must be a valid URL path in your application. For security reasons define this as specific as possible.
Define a custom login URL Okta Access Gateway will forward user to this location when the Login path is called.

Logout

Allows you to define logout endpoints that will aid in destroying the user session.

Fields:

Field Description
Logout Dropdown, behavior on logout
Logout path Path to logout endpoint
Single logout Enable to destroy both the Access Gateway and Okta session. Disable only the Access Gatewat session.

Logout dropdown values
Show Logout page Called after your application logout to destroy the Okta Access Gateway session cookie. After logout is complete Okta Access Gateway will render a logout page.

Show Login page

Call this location after your application logout to destroy the Okta Access Gateway session cookie. After logout is complete Okta Access Gateway will render the Identity Provider login page.
Use Application logout page Destroy Okta Access Gateway session cookie while rendering application logout page. This must be a valid URL path in your application. For security reasons be as specific as possible.


Post logout URL

Default URL end-users are directed to after logout. By default this will be the Post Login value.
Enable the field and enter an appropriate URL.

Error

Allows you to define error endpoints that will aid in handing error scenarios.

Error dropdown values
Use Okta Access Gateway error page Define an error path for this application. Will show a generic Okta Access Gateway error page. Default.
Use Application error page Show application hosted error page. This must be a valid URL path in your application. For security reasons define this as specifically as possible.
Define a custom error URL Okta Access Gateway will forward user to this location when the Error path is called.
In addition to Error path also enter a path defined as a error handler.
This path is called to invoke a error flow.

Dont define any Error Behavior

Access Gateway will not provide any error behavior.

Policy Denied

Allows you to define the enduser experience when the Access Gateway denies access to a resource in the case of a policy failure.

Dropdown values
Show default policy failure page Show enduser default Okta Access Gateway policy failure page. Default
Return 403 status code Return blank page with HTTP 403 status code

Session Integrity Failed

Allows you to define the enduser experience when the Access Gateway detects a sesion integrity failure. Common ifendusers are changing networks while maintaining an active application session.
Access Gateway will finger print the remote-IP and deny access.

Dropdown values
Show default Security warning page Show enduser default Okta Access Gateway Security warning page. Default.
Return 405 status code Return blank screen with 405 status code.
Redirect to IDP If IDP session is still active, enduser will SSO back into app.

Force reauthentication at IDP,

Force user to re-authenticate. Once re-authenticated will SSO back into app.

Do not enforce

Do not enforce session integrity.
