Configure a password policy

Password policies enforce password settings at the group and authentication-provide level. Okta provides a default policy to enforce the use of strong passwords. You can also create additional policies that are less or more restrictive.

The custom source integration password policy doesn't support policy rules with self-service actions. For example, password change, self-service password reset, and self-service unlock are disabled by default and can't be enabled in the Admin Console or by using an API.

Existing password policies for Okta, AD, and LDAP authentication providers can't be changed to a custom source authentication provider when the existing password policy enables self-service actions. All self-service actions must be disabled in every rule in the existing policy before the authentication provider can be changed to custom source.

The custom source integration password policy doesn't support delegated unlock functionality. Unlock requests unlock only the Okta user and not the user in the custom source.

Temporary passwords created by admins are not supported.

Okta creates a default password policy when you first create a custom source integration. If you would like to create an additional policy, follow these steps:

  1. In the Admin Console, go to Security > Authentication.
  2. Click the Password tab and then click Add New Password Policy.
  3. Complete these fields:
    • Policy name: Enter a unique name for the policy.
    • Policy description: Enter a description for the policy.
    • Add group: Enter the name of the groups to which the policy will apply.
  4. In the Authentication Providers section, select Custom Source.
  1. Complete these fields in the Password Settings section:
    • Minimum length: Enter a minimum password length of four to 30 characters (the default minimum is eight characters). For custom-sourced users, these requirements are set and enforced by the custom source. Ensure that the settings here duplicate the minimum settings of your custom source.
    • Complexity requirements: Select any or all of the options to define the level of complexity that is required for user passwords.
    • Common password check: Select Restrict use of common passwords to prevent the use of common, weak passwords.
    • Password age: Select these options:
      • Enforce password history for last: Enter the number of distinct passwords that a user must create before they can reuse a previous password. This prevents users from reusing a previous password for a specified period of time. You can configure this setting from one to 30 passwords.
      • Minimum password age is: Enter the minimum time interval required between password changes. This setting prevents users from bypassing the enforce password history requirement. You can set this value to a maximum of 9,999 minutes.
      • Prompt user: Enter the number of days prior to password expiration that users are prompted to change their password. Users can change their password when prompted, or wait until the expiration date. You can set this value to a maximum of 999 days.
    • Lock out: Select these options:
      • Lock out user after: Enter the number of times users can attempt to sign in to their accounts with an invalid password before their accounts are locked. The maximum is 100 invalid login attempts. See About lockouts .
      • Account is automatically unlocked after: Enter the number of minutes a locked account remains locked before it is unlocked automatically. Auto unlock requires no additional user action, and the minimum setting is one minute. You can set this value to a maximum of 9,999 minutes. It's not enabled by default.
      • Show lock out failures: Select this option to show a message on the sign-in page to alert users when they have been locked out of their account due to too many failed sign-in attempts.
      • Send a lockout email to user: Select this option to send an email to users when their account is locked due to too many failed sign-in attempts. You can customize the Account Locked email in Settings > Email & SMS.
  1. Click Create Policy.
  2. Optional. Create a password policy rule:
    1. On the Authentication page, click Add Rule.
    2. Complete these fields:
      • Rule Name: Enter a name for the rule.
      • Exclude Users: Start entering the name of a user you want to exclude from the rule. As you enter the name, Okta suggests usernames that match your text. Select the desired user from the list. Repeat for each additional user you want to exclude.
      • IF User is located: Select Anywhere to apply the rule to all users regardless of whether or not their IP address is listed in the Public Gateway IPs list, select On Network to apply the rule to users in all zones or in specific zones, or select Off Network to apply the rule to users outside all zones or specific zones. For information on the Public Gateway IPs list and the IP Zones features, see Manage Network Zones.
      • THEN Users can perform self service: Self-service actions aren't supported and no options can be selected.
    3. Click Create Rule.

About lockouts

Okta provides a feature to help prevent custom sourced users from getting locked out of their on-premises account and hardware device due to too many failed Okta sign-in attempts. This feature also helps prevent a malicious third party from using Okta to lock out users.

To prevent custom source lockouts, make sure that the number entered for Lock out user after <#> unsuccessful attempts is lower than the failed sign-in attempt limit configured in the custom source. For example, if the maximum number of failed sign-in attempts is set to 10 in the custom source, Okta recommends setting your maximum Okta failed sign-in limit to 9 in the password policy. If a user exceeds the sign-in limit set in Okta, additional failed attempts aren't sent to the custom source, and this prevents users from locking themselves out of their on-premises account. Delegated unlock isn't supported.
