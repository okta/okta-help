1要素認証でシンプルなセキュリティを構成する

このユースケースでは、認証に1要素タイプのみが必要です。主な目的は、Okta FastPassでパスワードなしのエクスペリエンスをどのように提供するかを示すことです。ポリシーで生体認証が必要ない場合、この例は中保証レベルに分類されます。

ブックマークアプリを作成する

  1. Okta Admin Consoleで、［Applications（アプリケーション）］>［Applications（アプリケーション）］に移動します。
  2. ［Browse App Catalog（アプリカタログを参照）］を選択します。
  3. Bookmark Appを検索し、検索結果に表示されたら選択します。
  4. ［Add Integration（統合を追加）］をクリックします。
  5. アプリケーションのラベルをBookmark App 1に変更します。
  6. ［URL］を入力し、［Done（完了）］をクリックします。このアプリはデモ用なので、任意のURLを選択できます。実際の環境では、SSOをセットアップするアプリのURLを使用します。
  7. ［Assignments（割り当て）］タブを選択します。
  8. ［Assign（割り当て）］>［Assign to Groups（グループに割り当て）］を選択してから、Okta FastPass Group 1の［Assign（割り当て）］を選択します。グループのプロパティーを確認する場合以外は、グループ名をクリックしないでください。
  9. ［Done（完了）］をクリックします。

認証ポリシールールを作成する

ユーザーがパスワードなしでサインインできるように、1要素認証を許可する認証ポリシールールを作成します。ルールを保存したら、デフォルトのキャッチオールルールよりも優先度を高くします。

  1. Okta Admin Consoleで、［Security（セキュリティ）］>［Authentication Policies（認証ポリシー）］に移動します。
  2. 認証ポリシーを作成するか（手順については「認証ポリシーを作成する」を参照）、リストからポリシーを選択します。
  3. 新しい認証ポリシーを作成する場合は、［Applications（アプリケーション）］タブを選択し、Bookmark App 1アプリケーションを追加します。手順については、「認証ポリシーにアプリを追加する」を参照してください。それ以外の場合は、使用する認証ポリシーを選択します。
  4. ［Rules（ルール）］タブを選択します。デフォルトのキャッチオールルールでは、1つの要素（パスワード）によるアクセスが許可されます。
  5. ［Add Rule（ルールを追加）］をクリックします。
  6. ［Add Rule（ルールを追加）］ダイアログで、ルール名を追加します（Bookmark App 1 ruleなど）。
  7. IF［User's user type is（ユーザーのユーザータイプ：）］フィールドで、［Any user type（任意のユーザータイプ）］をクリックします。
  8. AND［User's group membership includes（ユーザーのグループメンバーシップ：）］フィールドで、［At least one of the following groups（次のグループのうち少なくとも1つ：）］を選択します。
  9. 前提条件で作成したグループの名前（Okta FastPass Group 1）の入力を開始し、そのグループを選択します。
  10. AND［User is（ユーザー：）］フィールドで、［Any user（任意のユーザー）］を選択します。
  11. AND ［Device State is（デバイスの状態：）］フィールドで、［Registered（登録済み）］を選択します。
    • デバイスが登録されている場合、Okta Sign-in Widgetはユーザーのデバイスのポーリングを実行して、Okta Verifyがインストールされているかどうかを確認します。インストールされていて、その特定のorgのアカウントがOkta Verifyに登録されている場合、Oktaはユーザーを識別できます。
    • また、このオプションを選択すると、最初の使用時にOkta Verifyをインストールしてデバイスを登録するように求めるプロンプトがユーザーに表示されます。

      ［Registered（登録済み）］の代わりに［Any（すべて）］を選択すると、ユーザーに常にサインインページが表示されるようになり、ユーザーを認証するためにOkta Sign-in WidgetによりOkta Verifyのポーリングがサイレントに実行されなくなります。また、場合によってはOkta FastPassが確実には機能しなくなります。

      これを回避するには、［Registered（登録済み）］を選択します。また、［Registered（登録済み）］では、Okta Verifyがインストールされている場合、他のオーセンティケーターがインストールされていても、ユーザーには他のオーセンティケーターが表示されなくなります。

      Okta FastPassを含む寛容な認証ポリシーが必要で、AND ［Device State is（デバイスの状態：）］フィールドに［Any（すべて）］オプションを選択した場合は、2つのポリシーを構成して、最初のポリシーのランクを2番目のポリシーより高くします。

      1. AND［Device State is（デバイスの状態：）］フィールドが［Registered（登録済み）］に設定された、Okta FastPassが有効な認証ポリシー

      2. 有効にする追加のオーセンティケーターの使用を許可するために、AND［Device State is（デバイスの状態：）］フィールドが［Any（すべて）］に設定された認証ポリシー

  12. AND［Device Platform is（デバイスプラットフォーム：）］とAND［User's IP is（ユーザーのIP：）］にはデフォルト値を使用します（それぞれ、［Any platform（任意のプラットフォーム）］と［Any IP（任意のIP）］）。
  13. AND［The following custom expression is true（次のカスタム式をtrueとする）］フィールドは空白のままにします。
  14. AND［User must authenticate with（ユーザーが認証に使用する要素：）］フィールドで、［Possession factor（所有要素）］を選択します。

    ［Any 1 factor type（任意の1要素タイプ）］を選択した場合、認証ポリシーでは、（パスワードを含む）登録済みオーセンティケーターのリストがユーザーに提示され、使用するオーセンティケーターを選択することができます。

  15. AND［Access with Okta FastPass is granted（Okta FastPassを使用したアクセスを許可）］フィールドで、［Without the user approving a prompt in Okta Verify or providing biometrics（ユーザーがOkta Verifyでプロンプトを承認したり、生体認証を提供したりしない場合）］を選択します。もう片方のオプションである［If the user approves a prompt in Okta Verify or provides biometrics（ユーザーがOkta Verifyでプロンプトを承認するか、生体認証を提供する場合）］を選択した場合は、ユーザーに常にサインイン画面が表示され、［Sign in with Okta FastPass（Okta FastPassでサインインする）］を選択する必要があります。
  16. AND［Password re-authentication frequency（パスワードの再認証の頻度）］フィールドで、［Every sign-in attempt（すべてのサインイン試行）］を選択します。この場合、ユーザーがアプリにアクセスするたびにオーセンティケーターを検証する必要があります。［Re-authenticate after（次の期間ごとに再認証：）］を選択した場合は、オーセンティケーターが指定された時間枠内に使用されていない場合に再認証されます。再認証頻度のタイムスタンプは認証期間の開始時点を示し、別のアプリで認証が再利用されても変更されません。認証期間が期限切れになり、再認証が必要になると変更されます。
  17. ［Save（保存）］をクリックします。

ユーザーサインインエクスペリエンスをテストする

設定を変更したときは、テストする前にブラウザーをクリアしてください。

  1. 別のブラウザーインスタンスまたはシークレットウィンドウで、このorgのOkta End-User Dashboardに移動します。
  2. Bookmark App 1を選択します。
  3. ユーザーインタラクションなしでアプリにサインインすることを検証します。

別のサインインフロー

［Device State（デバイスの状態）］フィールドで［Registered（登録済み）］の代わりに［Any（すべて）］を選択した場合、

ユーザーは常にOkta Sign-In Widgetにリダイレクトされます（サイレント認証は行われません）。
