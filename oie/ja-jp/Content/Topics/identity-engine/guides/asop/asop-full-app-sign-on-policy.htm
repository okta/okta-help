完全な認証ポリシーのデプロイメント

これは、より高度な認証ポリシーです。このシナリオでは、組織は、アプリに対して3つの異なるレベルのセキュリティ要件（低、中、高）を使用する、より高度なアプローチを採用しています。

この戦略は、認証方法の要件に加えて、デバイスコンテキストにも依存しています（「デバイスコンテキストのデプロイメントガイド」を参照）。次の表は、各セキュリティレベルの要件を満たす認証とデバイスコンテキストの設定をまとめたものです。

注：このユースケースでは、Okta FastPassが有効になっています。

Okta FastPassの詳細については、「パスワードなしの認証のデプロイメントガイド」を参照してください。

前提条件

Okta FastPassをアクティブ化する

  1. Okta Admin Consoleで、［Security（セキュリティ）］>［Authenticators（オーセンティケーター）］に移動します。

  2. Okta Verifyの行で、［Actions（アクション）］>［Edit（編集）］をクリックします。
  3. ［Okta FastPass (All platforms)（Okta FastPass（すべてのプラットフォーム））］が選択されていることを確認します。

使用中のグローバルセッションポリシーがMFAを必要としないことを確認する

  1. Okta Admin Consoleで、［Security（セキュリティ）］> ［Global Session Policy（グローバルセッションポリシー）］に移動します。

  2. 使用中のポリシーを選択します。
  3. 該当するルールの右側にあるアイコンをクリックします。
  4. ［Multifactor authentication（多要素認証）］が選択されていないことを確認します。
  5. 必要に応じて、追加のポリシーとルールに対して上記の手順を繰り返します。

セットアップと構成

  1. Okta Admin Consoleで、［Security（セキュリティ）］>［Authentication Policies（認証ポリシー）］に移動します。

  2. ［Add a policy（ポリシーを追加）］をクリックします。
  3. ポリシーの［Name（名前）］と［Description（説明）］を入力します。
  4. ［Save（保存）］をクリックします。
  5. ［Rules（ルール）］タブで、［Add rule（ルールを追加）］をクリックします。
  6. ［Rule name（ルール名）］フィールドに、ルールの名前を入力します。わかりやすい名前にする必要があります（Salesforce Low Securityなど）。これは、システムログを確認してアクセスに関する問題をトラブルシューティングする場合に役立ちます。
  7. 目的の保証レベルの設定を適用します（前の表に詳述されています）。以降のセクションで構成された条件を参照します。

低保証アプリ

  1. IFセクションで、AND ［Device state is（デバイスの状態：）］を［Any（すべて）］に設定します。
  2. THENセクションで、次のように設定します。
    1. THEN ［Access is（アクセス：）］を［Allowed after successful authentication（認証の成功後に許可）］に設定します。
    2. AND ［User must authenticate with（ユーザーが認証に使用する要素：）］を［Any 1 factor type（任意の1要素タイプ）］に設定します。
    3. AND ［Access with Okta FastPass is granted（Okta FastPassを使用したアクセスを許可）］を［Without the user approving a prompt in Okta Verify or providing biometrics（ユーザーがOkta Verifyでプロンプトを承認したり、生体認証を提供したりしない場合）］に設定します。
    4. AND ［Re-authentication frequency is（再認証の頻度：）］を［Never re-authenticate if the session is active（セッションがアクティブの場合は再認証しない）］に設定します。
  3. ［Save（保存）］をクリックします。
  4. ［Applications（アプリケーション）］タブで、［Add app（アプリを追加）］をクリックします。
  5. このポリシーに追加するアプリを検索し、［Add（追加）］をクリックします。

中保証アプリ

  1. IFセクションで、AND ［Device state（デバイスの状態）］を［Registered（登録済み）］に設定します。
  2. THENセクションで、次のように設定します。
    1. AND ［Device management is（デバイス管理：）］を［Not managed（非管理対象）］に設定します。
    2. THEN ［Access is（アクセス：）］を［Allowed after successful authentication（認証の成功後に許可）］に設定します。
    3. AND ［User must authenticate with（ユーザーが認証に使用する要素：）］を［Any 2 factor types（任意の2要素タイプ）］に設定します。
    4. AND ［Access with Okta FastPass is granted（Okta FastPassを使用したアクセスを許可）］を［Without the user approving a prompt in Okta Verify or providing biometrics（ユーザーがOkta Verifyでプロンプトを承認したり、生体認証を提供したりしない場合）］に設定します。
    5. AND［Re-authentication frequency is（再認証の頻度：）］を［Re-authenticate after（以下の後で再認証：）］に設定し、期間を［2 hours（2時間）］に設定します。
  3. ［Save（保存）］をクリックします。
  4. ［Applications（アプリケーション）］タブで、［Add app（アプリを追加）］をクリックします。
  5. このポリシーに追加するアプリを検索し、［Add（追加）］をクリックします。

高保証アプリ

  1. IFセクションで、AND ［Device state（デバイスの状態）］を［Registered（登録済み）］に設定します。
  2. THENセクションで、次のように設定します。
    1. AND ［Device management is（デバイス管理：）］を［Managed（管理対象）］に設定します。
    2. THEN ［Access is（アクセス：）］を［Allowed after successful authentication（認証の成功後に許可）］に設定します。
    3. AND ［User must authenticate with（ユーザーが認証に使用する要素：）］を［Any 2 factor types（任意の2要素タイプ）］に設定します。
    4. AND ［Possession factor constraints are（所有要素の制約：）］を［Hardware protected（ハードウェアで保護）］に設定します。
    5. AND ［Access with Okta FastPass is granted（Okta FastPassを使用したアクセスを許可）］を［If the user approves a prompt in Okta Verify or provides biometrics (meets NIST AAL2 requirements)（ユーザーがOkta Verifyでプロンプトを承認するか、生体認証を提供する（NIST AAL2要件を満たす）場合）］に設定します。
    6. AND ［Re-authentication frequency is（再認証の頻度：）］を［Every sign-in attempt（すべてのサインイン試行）］に設定します。
  3. ［Save（保存）］をクリックします。
  4. ［Applications（アプリケーション）］タブで、［Add app（アプリを追加）］をクリックします。
  5. このポリシーに追加するアプリを検索し、［Add（追加）］をクリックします。

このシナリオでは、中および高の保証レベルでは、それぞれ登録または管理されていないデバイスからのアクセスを拒否することが期待されます。これを実現するには、アプリポリシーのキャッチオールルールを、アクセスを拒否するデフォルトの状態から変更してはなりません。このルールは、上記のルールで設定されたIF条件を満たさないデバイスに適用されます。

ユーザーサインインエクスペリエンス

ユーザーが認証ポリシーによって管理されるアプリを起動しようとしたときのユーザーのエクスペリエンスは、ポリシーに適用されるセキュリティレベルによって決定されます。同じユーザーが異なるアプリを起動すると、ポリシーに基づいて、異なるエクスペリエンスが得られます。

上記の設定を使用してアプリを構成したら、次のいずれかのアプローチを試します。

低保証アプリ

ユーザーが任意のデバイスからアプリを起動すると、アクセス権が付与される前に、認証方法が1つだけ満たされる必要があります。パスワードを入力すると、この要件が満たされます。

中保証アプリ

  • ユーザーが未登録デバイスからアプリを起動した場合、次の図に示されているように、デバイスにOkta Verifyがインストールされて起動されるまで先に進むことはできません。
  • ユーザーが登録済みデバイスからアプリを起動した場合、アプリに進む前に2つの要素タイプを満たす必要があります。たとえば、パスワードを入力し（知識要素）、Okta Verifyプッシュに応答する（所有要素）場合があります。
  • ユーザーがアプリケーションを最初に起動した後に2回アプリを閉じて再起動すると、認証を求めるプロンプトが表示されます。

高保証アプリ

中保証アプリのエクスペリエンスと似ていますが、次の点が異なります。

  • ユーザーが未登録、登録済み - 非管理対象のデバイス、またはハードウェアで保護されていないデバイスからアプリを起動すると、両方の要件が満たされるまで先に進むことはできません。
  • ユーザーは、保護されているアプリを起動するたびに、両方の要素プロンプトに応答する必要があります。
