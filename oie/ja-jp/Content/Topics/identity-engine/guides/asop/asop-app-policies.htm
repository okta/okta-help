認証ポリシーの使用を開始する

このユースケースでは、使用される要素のタイプによる高いレベルの保証のために、認証ポリシーをセットアップします。たとえば、パスワードのような知識要素は、指紋のような固有要素ほど安全ではありません。orgレベルでの二次認証方法が必要になっています。

セットアップと構成

グローバルセッションポリシーを作成する

  1. Okta Admin Consoleで、［Security（セキュリティ）］> ［Global Session Policy（グローバルセッションポリシー）］に移動します。

  2. ［Add a New Global Session Policy（新しいグローバルセッションポリシーを追加）］をクリックします。
  3. ［Add Policy（ポリシーを追加）］ダイアログで、［Policy Name（ポリシー名）］とオプションの［Policy Description（ポリシーの説明）］を追加します。
  4. ［Assign to Groups（グループに割り当て）］フィールドで、［Everyone（全員）］を選択してから、［Create Policy and Add Rule（ポリシーを作成してルールを追加）］をクリックします。
  5. ［Add Rule（ルールを追加）］ダイアログのAND ［Establish the user session with（次を使用してユーザーセッションを確立：）］フィールドで、［Any factor used to meet the Authentication Policy rules（認証ポリシーのルールを満たすために使用される任意の要素）］を選択します。
  6. ［Multifactor authentication is（多要素認証：）］フィールドで、［required（必須）］を選択してから、［At every sign in（サインイン毎）］を選択します。このオプションは、ユーザーが独自のMFAプロンプトを制御できなくなるため、リスクの高いサインインイベントを検出する行動条件での使用に適しています。
  7. ［Create Rule（ルールを作成）］をクリックします。

［When signing in with a new device cookie（新しいデバイスCookieでサインインする場合）］オプションと［After MFA lifetime expires（MFAのライフタイムが期限切れになった後）］オプションは、ユーザーのグループのMFAを減らします。ユーザーが（ブラウザーのCookieに基づいて）同じデバイスからサインインしているように見える場合にMFAをバイパスできるため、低リスク、低保証のユースケースにのみ適しています。行動条件では使用しないでください。

グループの作成

サインイン用のグループを作成し、グループにユーザーを追加します。この手順例ではブックマークアプリを使用しますが、各アプリに必要なアプリの統合を使用してください。

  1. Okta Admin Consoleで、［Directory（ディレクトリ）］>［Groups（グループ）］に移動します。
  2. ［Add Group（グループを追加）］をクリックします。
  3. ［Add Group（グループを追加）］ダイアログで、［Name（名前）］を入力します（Group for low assurance appなど）。
  4. ［Group Description（グループの説明）］を入力し、［Save（保存）］をクリックします。
  5. ［Directory（ディレクトリ）］>［People（ユーザー）］に移動し、［Add person（ユーザーを追加）］をクリックします。
  6. ［Add Person（ユーザーを追加）］ダイアログで、［First name（名）］、［Last name（姓）］、［Username（ユーザー名）］、および［Primary email（プライマリメール）］を追加します。［Secondary email（セカンダリメール）］フィールドに自分のメールを追加します。
  7. ［Groups（グループ）］フィールドで、ステップ2で作成したグループを選択します。
  8. ［Password（パスワード）］フィールドで、［Set by admin（管理者が設定）］を選択します。
  9. パスワードを入力してから、［User must change password on first login（ユーザーは初回のログイン時にパスワードを変更する必要があります）］をクリアします。
  10. ［Save（保存）］をクリックします。
  11. （高保証アプリの）別のグループに対してこのプロセスを繰り返します。

ブックマークアプリを作成する

  1. Okta Admin Consoleで、［Applications（アプリケーション）］>［Applications（アプリケーション）］に移動します。
  2. ［Browse App Catalog（アプリカタログを参照）］を選択します。
  3. ［Search（検索）］ボックスにbookmark appと入力し、［Bookmark App（ブックマークアプリ）］を選択して［Add（追加）］をクリックします。
  4. ［Add Bookmark App（ブックマークアプリを追加）］画面で、アプリケーションラベルをBookmark for low assurance appに変更します。
  5. ［URL］を入力し、［Done（完了）］をクリックします。このアプリはデモ用なので、任意のURLを選択できます。実際の環境では、SSOをセットアップするアプリのURLを使用します。
  6. ［Assignments（割り当て）］タブを選択します。
  7. ［Assign（割り当て）］>［Assign to Groups（グループに割り当て）］を選択し、Group for low assurance appに対して［Assign（割り当て）］を選択します。グループのプロパティーを確認する場合以外は、グループ名をクリックしないでください。
  8. ［Done（完了）］をクリックします。
  9. 2番目の認証ポリシー（Bookmark for high assurance app）を作成するために使用する別のブックマークアプリに対して、このプロセスを繰り返します。

認証ポリシーを作成する

1つの要素タイプを必要とする認証ポリシーと、2つの要素タイプを必要とする別の認証ポリシーを作成します。認証ポリシーを作成するときは、ポリシーが誰に適用されるかを考慮する必要があります。このポリシーを適用する特定のユーザー、グループ、ユーザータイプを決定してください。

低保証アプリの認証ポリシー

  1. Okta Admin Consoleで、［Security（セキュリティ）］>［Authentication Policies（認証ポリシー）］に移動します。

  2. ［Add a policy（ポリシーを追加）］をクリックします。

  3. ポリシーの［Name（名前）］と［Description（説明）］を入力します。

  4. ［Save（保存）］をクリックします。

  5. ［Rules（ルール）］タブで、［Add rule（ルールを追加）］をクリックします。
  6. ［Rule Name（ルール名）］を入力します。
  1. IF ［User's user type is（ユーザーのユーザータイプ：）］フィールドで、［Any user type（任意のユーザータイプ）］をクリックします。
  2. AND ［User's group membership includes（ユーザーのグループメンバーシップ：）］フィールドで、［At least one of the following groups（次のグループのうち少なくとも1つ：）］を選択します。
  3. 前提条件で作成したグループの名前の入力を開始し、そのグループを選択します。
  4. AND ［User is（ユーザー：）］フィールドで、［Any user（任意のユーザー）］を選択します。
  5. AND ［Device State（デバイスの状態）］フィールドで、［Any（すべて）］を選択します。
  6. AND ［Device Platform is（デバイスプラットフォーム：）］に対して［Any platform（任意のプラットフォーム）］を選択し、AND ［User's IP is（ユーザーのIP：）］に対して［Any IP（任意のIP）］を選択します。AND ［The following custom expression is true（次のカスタム式をtrueとする）］フィールドは空白のままにします。式言語を使用して、認証ポリシーにカスタム式を追加できます。
  7. AND ［User must authenticate with（ユーザーが認証に使用する要素：）］フィールドに対して、［Any 1 factor type（任意の1要素タイプ）］を選択します。アプリレベルのサインオンポリシーでは、（パスワードを含む）登録済みオーセンティケーターのリストがユーザーに提示され、使用するオーセンティケーターを選択することができます。
  8. AND ［Access with Okta FastPass is granted（Okta FastPassを使用したアクセスを許可）］フィールドで、［If the user approves a prompt in Okta Verify or provides biometrics（ユーザーがOkta Verifyでプロンプトを承認するか、生体認証を提供する場合）］を選択します。
  9. AND ［Re-authentication frequency is（再認証の頻度：）］フィールドで、［Every sign-in attempt（すべてのサインイン試行）］を選択します。この場合、ユーザーがアプリにアクセスするたびにオーセンティケーターを検証する必要があります。［Re-authenticate after（次の期間ごとに再認証：）］を選択した場合は、オーセンティケーターが指定された時間枠内に使用されていない場合に再認証されます。再認証頻度のタイムスタンプは認証期間の開始時点を示し、別のアプリで認証が再利用されても変更されません。認証期間が期限切れになり、再認証が必要になると変更されます。
  10. ［Save（保存）］をクリックします。
  11. ［Applications（アプリケーション）］タブを開きます。
  12. Bookmark for low assurance appを検索し、［Add app（アプリを追加）］をクリックします。

高保証アプリのアプリサインオンポリシー

  1. Okta Admin Consoleで、［Security（セキュリティ）］>［Authentication Policies（認証ポリシー）］に移動します。

  2. ［Add a policy（ポリシーを追加）］をクリックします。

  3. ポリシーの［Name（名前）］と［Description（説明）］を入力します。

  4. ［Save（保存）］をクリックします。

  5. ［Rules（ルール）］タブで、［Add rule（ルールを追加）］をクリックします。
  6. AND ［User's user type is（ユーザーのユーザータイプ：）］フィールドで、［Any user type（任意のユーザータイプ）］をクリックします。
  7. AND ［User's group membership includes（ユーザーのグループメンバーシップ：）］フィールドで、［At least one of the following groups（次のグループのうち少なくとも1つ：）］を選択します。
  8. 前提条件で作成したグループの名前の入力を開始し、そのグループを選択します。
  9. AND ［User is（ユーザー：）］フィールドで、［Any user（任意のユーザー）］を選択します。
  10. AND ［Device State（デバイスの状態）］フィールドで、［Any（すべて）］を選択します。
  11. AND ［Device Platform is（デバイスプラットフォーム：）］に対して［Any platform（任意のプラットフォーム）］を選択し、AND ［User's IP is（ユーザーのIP：）］に対して［Any IP（任意のIP）］を選択します。AND ［The following custom expression is true（次のカスタム式をtrueとする）］フィールドは空白のままにします。式言語を使用して、認証ポリシーにカスタム式を追加できます。
  12. AND ［User must authenticate with（ユーザーが認証に使用する要素：）］フィールドで、［Any 2 factor types（任意の2要素タイプ）］を選択します。認証ポリシーでは、（パスワードを含む）登録済みオーセンティケーターのリストがユーザーに提示され、使用するオーセンティケーターを選択することができます。
  13. AND ［Access with Okta FastPass is granted（Okta FastPassを使用したアクセスを許可）］フィールドで、［If the user approves a prompt in Okta Verify or provides biometrics（ユーザーがOkta Verifyでプロンプトを承認するか、生体認証を提供する場合）］を選択します。
  14. AND ［Re-authentication frequency is（再認証の頻度：）］フィールドで、［Every sign-in attempt（すべてのサインイン試行）］を選択します。この場合、ユーザーがアプリにアクセスするたびにオーセンティケーターを検証する必要があります。［Re-authenticate after（次の期間ごとに再認証：）］を選択した場合は、オーセンティケーターが指定された時間枠内に使用されていない場合に再認証されます。再認証頻度のタイムスタンプは認証期間の開始時点を示し、別のアプリで認証が再利用されても変更されません。認証期間が期限切れになり、再認証が必要になると変更されます。
  15. ［Save（保存）］をクリックします。
  16. ［Applications（アプリケーション）］タブを開きます。
  17. Bookmark for high assurance appを検索し、［Add app（アプリを追加）］をクリックします。

ユーザーサインインエクスペリエンス

ユーザーが認証ポリシーによって管理されるアプリを起動しようとしたときのユーザーのエクスペリエンスは、ポリシーの保証レベルによって決定されます。保証レベルはビジネスニーズによって決定されます。以下に例を示します。

低保証アプリ

任意のデバイスからアプリケーションを起動します。1つの認証方法用のプロンプトが表示されます。

高保証アプリ

任意のデバイスからアプリケーションを起動します。2つの認証方法用のプロンプトが連続して表示されます。

キャッチオールアプリ

任意のデバイスからアプリケーションを起動します。パスワードのプロンプトのみが表示されます。

低保証と高保証を切り替える

  • 低保証アプリを起動してから、高保証アプリを起動します。追加認証のリクエストが表示されます。
  • 高保証アプリを起動してから、低保証アプリを起動します。追加認証なしで低保証アプリを開くことができます。

Okta FastPassをオンにする

必要に応じてOkta FastPassをオンにして、ユーザーエクスペリエンスを確認します。

  1. Okta Admin Consoleで、［Security（セキュリティ）］>［Authenticators（オーセンティケーター）］に移動します。
  2. ［Authenticators（オーセンティケーター）］セクションで、［Actions（アクション）］>［Edit（編集）］を選択して［Okta FastPass using Okta Verify（Okta Verifyを使用したOkta FastPass）］を有効にします。
  3. ［Okta Verify］画面の［Verification options（検証オプション）］セクションで、［Okta Fast (All platforms)（Okta Fast（すべてのプラットフォーム））］を選択します。
  4. ［Okta FastPass］セクションで、［Show the "Sign in with Okta FastPass" button（［Sign in with Okta FastPass（Okta FastPassでサインイン）］ボタンを表示）］を選択します。これにより、次の3つのことが行われます。
    • Okta Verifyのインストールとデバイスの登録について、初めてのユーザーに順を追って説明されます。
    • ユーザーの構成でサイレントサインオンが許可されていない場合は、別の方法を使用できます。たとえば、Jamf ProまたはamSafari SSOブラウザー拡張機能などのデバイス管理ソリューションを使用していないMacユーザーは、サイレントサインインができません。このボタンを有効にすると、このようなユーザーがサインインできるようになります。
    • Okta Verifyが自動的に読み込まれない場合のバックアップとして機能します。

Okta FastPassをオンにすると、認証ポリシーの［Available Authenticators（使用可能なオーセンティケーター）］の下に表示されます。

ユーザーは、生体認証が有効になっていることを前提として、Okta FastPassのみを使用してより高い保証のアプリにログインできるようになりました。
