要素シーケンシングに関するよくある質問

Okta Identity Engineは、少しずつお客様に提供が開始されています。アップグレードの適格性については、アカウント担当者にお問い合わせください。

要素シーケンシングとは何ですか？

Okta Classic Engineでは、この機能は標準のパスワードの代わりに、一連のMFA要素で認証してorgにサインインすることをエンドユーザーに許可します。これは、ログインコンテキストに基づいてパスワードレスエクスペリエンスを提供する方法です。管理者は、認証フローで1つ以上の要素を順序付けることができます。

Okta Classic Engine orgで要素シーケンシングが有効になっているかどうかを知るにはどうすればよいですか？

Okta Admin Consoleで、［Security（セキュリティ）］>［Authentication（認証）］に移動します。［Sign On（サインオン）］タブで、ルールを選択し、［Edit（編集）］をクリックします。［Authentication methods（認証方法）］セクションの［Factor Sequence（要素シーケンス）］オプションを参照してください。

自分のorgで要素シーケンシングが使用されているかどうかを確認するにはどうすればよいですか？

各orgのサインオンポリシーを確認します。

ポリシーを表示するには、次の手順を実行します。

  1. Okta Admin Consoleで、［Security（セキュリティ）］>［Authentication（認証）］に移動します。

  2. ［Sign On（サインオン）］タブで、ルールを選択し、［Edit（編集）］をクリックします。

    • 要素シーケンシング機能が有効になっているが、アクティブなチェーンがない場合は、［Password / Any IDP（パスワード/任意のIDP）］または［Password / Any IDP + Any factor（パスワード/任意のIDP + 任意の要素）］オプションが［Authentication methods（認証方法）］セクションで選択されています。
    • 要素シーケンシングが有効で、アクティブなチェーンがある場合、［Factor Sequence（要素シーケンス）］オプションと要素シーケンスチェーンが［Authentication methods（認証方法）］セクションで選択されています。

アップグレード後、要素シーケンシングを使用して、Oktaユーザー名を識別子先行フローの最初のサインイン画面に表示できますか？

はい、アップグレード後、識別子先行エクスペリエンスを直接確認できます。プレースホルダードメインを使用してIDPルーティングルールを作成する必要がなくなりました。

アップグレード後、要素シーケンシングを使用して、ユーザーがパスワード以外の任意の要素で認証できるようにすることはできますか？

はい、Okta Identity Engineにアップグレードした後、単一要素のパスワードなし認証を必要とするアプリケーションで次の手順を実行します。

  1. グローバルセッションポリシーで識別子先行フローに切り替えます。切り替える前に、パスワード優先フローから識別子先行フローに切り替えた場合のすべての影響を慎重に検討してください。

  2. ユーザーが所有要素で認証できるようにする認証ポリシーを作成します。

    1. Okta Identity Engine Admin Consoleで、［Security（セキュリティ）］>［Authentication Policies（認証ポリシー）］に移動します。
    2. ［Passwordless Authentication（パスワードなし認証）］という認証ポリシーを作成します。
    3. ［Single Factor Passwordless Authentication（単一要素のパスワードなし認証）］というルールを作成します。
    4. ［User Must Authenticate（ユーザーが認証に使用する要素）］ドロップダウンから、［Possession Factor（所有要素）］を選択します。このグループのユーザーは、パスワード以外の任意の所有要素で認証できるようになったため、同じ動作になります。

認証ポリシーを作成したら、それをアプリケーションに関連付けます。「認証ポリシー」を参照してください。

アップグレード後、要素シーケンシングを使用して、ユーザーがパスワードと別の要素で認証されるようにすることはできますか？

はい、Okta Identity Engineにアップグレードした後、認証にパスワードと別の要素を必要とするアプリケーションで次の手順を実行します。

  1. グローバルセッションポリシーで識別子先行フローに切り替えます。切り替える前に、パスワード優先フローから識別子先行フローに切り替えた場合のすべての影響を慎重に検討してください。
  2. Okta Admin Consoleで、［Security（セキュリティ）］>［Authentication Policies（認証ポリシー）］に移動します。
  3. ［Password +Another factor（パスワード+別の要素）］という認証ポリシーを作成します。
  4. このルールをグループに割り当てます。
  5. ［Password +Another factor（パスワード+別の要素）］を選択します。

アップグレード後、要素シーケンシングを使用して、ユーザーが認証に使用できる要素を制限できますか？

はい、これはOkta Identity Engineで所有要素に制約を適用することで限定的に行うことができます。

  1. グローバルセッションポリシーで識別子先行フローに切り替えます。切り替える前に、パスワード優先フローから識別子先行フローに切り替えた場合のすべての影響を慎重に検討してください。
  2. 所有要素の制約を使用して認証ポリシーを作成します。
  3. 方式の特徴 説明 例
    デバイスバウンド デバイスのキーまたはシークレットはデバイスに保存され、再登録せずに別のデバイスに転送することはできません。 メールと電話を除くすべての所有オーセンティケーター。
    ハードウェアで保護 シークレットまたは秘密鍵をハードウェアで保護するオーセンティケーター。デバイスのキーは、別のデバイス、Trusted Platform Module（TPM）、安全なエンクレーブ、またはRSA SecureIDなどの別のハードウェアトークンに保存されます。ハードウェアでの保護は、すべてのタイプのデバイスで提供されるわけではありません。 Okta Verifyの所有証明キー

アップグレード後、パスワード優先フローから識別子先行のフローに切り替える前に、何を考慮する必要がありますか？

アップグレード後は、次の考慮事項に留意してください。

  • Okta Identity Engineにアップグレードすると、既存のOkta Sign-on Policyは、パスワード優先フローのグローバルセッションポリシーになります。アップグレード後に新しいグローバルセッションポリシーを追加すると、デフォルトの［any factor used to meet authentication policy requirement（認証ポリシー要件を満たすために使用される任意の要素）］フローが使用されます。
  • パスワード優先フローでのエンドユーザーエクスペリエンスは異なります。
  • Okta Sign-in Widgetで、エンドユーザーは次の変更を確認できるようになりました。
    • セキュリティイメージが削除されます。
    • ［Don't Prompt（プロンプトを表示しない）］または［Remember me（記憶する）］チェックボックスは、［Keep me signed in（サインインしたままにする）］チェックボックスに置き換えられました。
    • サインインには、より多くの支援オプションを利用できます。
    • オーセンティケーターは、ドロップダウンメニュー内にネストされるのではなく、Okta Sign-in Widgetにリストされるようになりました。
    • 2種類のCAPTCHAとソーシャルログインを標準でサポートします。
  • 識別子先行フローでのエンドユーザーエクスペリエンスは異なります。
  • 認証ポリシーの要件を満たすために使用される任意の要素を使用してユーザーセッションが確立されると、ユーザー名のプロンプトが最初に表示されます。これは識別子先行フローと呼ばれます。
  • Okta FastPass、Device Trust 2.0、CAPTCHA、メールマジックリンク、プロファイル登録など、Okta Identity Engineで解放された新しい機能を使用するには、グローバルセッションポリシーで識別子先行フローに切り替える必要があります。切り替えた後、エンドユーザーエクスペリエンスは次のようになります。
    1. エンドユーザーは、サインインページに移動します。
    2. Okta Sign-in Widgetに［Username（ユーザー名）］フィールドが表示されます。
    3. エンドユーザーは、ドメインを含む完全なユーザー名を入力します。
    4. 必要に応じて、エンドユーザーは［Keep me signed in（サインインしたままにする）］を選択します。
    5. エンドユーザーは、［Next（次へ）］をクリックします。
    6. Okta Sign-in Widgetに、サインオンポリシーで許可されているセキュリティメソッドが表示されます。
    7. エンドユーザーは、セキュリティメソッドを選択し、確認手順を完了します。
    8. エンドユーザーは、アプリケーションにアクセスします。
  • 生体認証を使用した識別子先行フローでのエンドユーザーエクスペリエンスは異なります。
  • サインオンポリシーで生体認証が許可されている場合、Okta Sign-in WidgetにはOkta FastPassに最適化された識別子先行ページが表示されます。エンドユーザーエクスペリエンスは次のようになります。
    1. エンドユーザーは、サインインページに移動します。
    2. Okta Sign-in Widgetに、［Sign in with Okta FastPass（Okta FastPassでサインインする）］と［Username（ユーザー名）］フィールドの2つのオプションが表示されます。
      • エンドユーザーが［Username（ユーザー名）］を選択すると、識別子先行フローの手順が続行されます。
      • エンドユーザーが［Sign in with Okta FastPass（Okta FastPassでサインインする）］を選択すると、デバイス登録を通じて自動的に認証されます。
  • 組み込みSDK/組み込みウィジェットの互換性
  • Okta Identity Engineは後方互換性があります。したがって、すべてのアプリは引き続き機能します（リダイレクト/埋め込み/ダイレクトAPI）。
  • アップグレード後、認証ポリシーはデフォルトでパスワード優先に設定されるため、カスタムサインインページは引き続き機能します。
  • パスワードなしやデバイスコンテキスト2.0などの新機能を採用する場合は、認証ポリシーを変更して有効にする必要があります。新しいSDKを使用するには、新しい機能が必要なアプリのカスタムサインインページをアップグレードする必要がある場合もあります。
