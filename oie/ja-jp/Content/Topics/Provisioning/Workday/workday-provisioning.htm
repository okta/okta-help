Workday

概要

Oktaは、標準APIを介してWorkdayからユーザーとグループをインポートできます。次の3種類のインポートがサポートされています。

フル

フルインポートは、すべてのワーカーとすべての基本属性およびカスタム属性を取り込みます。完全インポートには時間がかかりますが、2つのシステム間で調整を実行し、他のインポートタイプでサポートされていない属性を取得するようにスケジュールする必要があります。通常、これは週に1回です。フルインポートに含まれるのは、基本属性、非未来および将来の有効期限のカスタム属性、インクリメンタルまたはRTSインポートで見逃される可能性のある変更点です。

増分

増分インポートは、Workdayが最後の増分インポート以降に更新されたと識別するワーカーのデータをもたらします。ワーカーを含めるには、変更がベースまたは無効な将来の日付のカスタム属性に含まれている必要があります。有効な日付のカスタム属性を変更しただけでは、増分インポートはトリガーされません。増分インポートに含まれるのは、基本属性、非未来および未来の有効期限付きカスタム属性です。インクリメンタルインポートは、通常のビジネスプロセスをサポートする間隔でスケジュールする必要があります。通常、これは少なくとも1日に1回であり、1時間に1回の頻度でスケジュールできます。

リアルタイム同期

リアルタイム同期はWorkdayからOktaへのアップデートをリアルタイムにトリガするために使用されます。従業員の突然の退職など、タイムリーな変更が重要な場合に使用されるべきです。Oktaにこのプロセスを開始するようトリガを送るために、Workdayで業務プロセスを構成する必要があります。RTSインポートには、ベース属性、将来（または将来でないが）有効な日付のカスタム属性が含まれています。詳細については、 Workday Real Time Sync | Oktaを参照してください。

これらのインポートタイプの最適な構成により、WorkdayからOktaに移動するデータの最適なデータ精度と適時性を確保できます。これらは、各インポートタイプの機能と制限を考慮して構成する必要があります。

Oktaは、以下に説明するように、WorkdayからのインポートとWorkday主導のITプロビジョニングの2つの典型的なシナリオをサポートしています。

Workdayからインポート

WorkdayからのインポートOktaは、他のアプリケーションと同様に、Workdayからユーザーとグループをインポートするだけです。インポートされたWorkdayユーザーはOktaユーザーの作成に使用され、インポートされたWorkdayグループはアプリの割り当てに使用できます。ただし、WorkdayユーザーがOktaにインポートされると、Workdayによって管理されなくなります。Workdayでユーザーを更新しても、関連付けられているOktaユーザーは変更されません。

Workday主導のITプロビジョニング

Workday主導のITプロビジョニングOktaはWorkdayと統合して、ITプロビジョニングを推進します。WorkdayユーザーがOktaにインポートされると、それらは引き続きWorkdayによって管理されます。Workdayで行われた更新と終了は、Oktaとダウンストリームアプリに反映されます。この取り決めにより、Workdayは従業員と請負業者のアプリへのアクセスを管理できます。Workday主導のITプロビジョニングは、Import from Workdayで提供される機能のスーパーセットであるため、このドキュメントの残りの部分では、Workday主導のITプロビジョニングの構成に焦点を当てますが、Import fromWorkdayシナリオにも関連します。

注：

  • プロファイルソーシングが有効になっている場合、Oktaは、 First Day Of Work および Hire Date 属性またはフィールドがWorkdayに入力されているユーザーのみをインポートします。
  • Active Directoryを使用していて、プロファイルプッシュが有効になっている場合、詳細については、Okta Active Direction（AD）エージェントのインストールと構成（具体的には Active Directoryのプロビジョニング設定を構成する）を参照してください。

Workday主導のITプロビジョニングにより、Oktaは次のワーカーライフサイクルイベントをサポートします。

  • 新規採用

    • Workdayで新しいワーカーが採用されました

    • Okta は新しいワーカーをインポートし、Okta ユーザー プロファイルを作成します。

    • Oktaはダウンストリームアプリでアカウントを作成します（ADを含む）

  • アップデート

    • Workdayユーザーの属性がWorkdayで変更されました

    • Oktaは属性の変更をインポートします

    • Oktaはダウンストリームアプリの属性を更新します（ADを含む）

  • 退職

    • ワーカーはWorkdayで終了します

    • Oktaはステータス変更をインポートします

    • Oktaは、ダウンストリームアプリ（ADを含む）のOktaユーザーとアカウントを非アクティブ化します

  • 再雇用

    • 解雇されたワーカーはWorkdayで再雇用されます

    • 非アクティブ化されたOktaユーザーは、Oktaで手動で再アクティブ化する必要があります

    • Oktaは、再雇用されたワーカーをインポートして、再アクティブ化されたOktaユーザーに再リンクします

前提条件

Oktaでプロビジョニングを構成する前に、以下の操作が必要です。

Workdayアプリのインスタンスを追加し、SSOを設定します

この手順ではWorkdayアプリのインスタンスが既にOktaに追加され、SSOを設定済みであることを前提とします。詳しくは［ServiceNow向けにSAML 2.0を設定するには」を参照してください。

アプリケーションの一般的な情報とアプリケーションを追加する方法については、「アプリケーション」を参照してください。

Workdayで統合システムユーザーを作成する

Oktaは、 統合システムユーザーと呼ばれる特別なタイプのWorkdayユーザーを使用してWorkdayAPIにアクセスします。作成するには、 検索 バーに create integration system user と入力し、結果のタスクを選択します。指示に従って、ユーザー名とパスワードを作成します。

統合システムユーザーに権限を付与する

次に、Workdayセキュリティグループを介してOkta Workday統合に必要なWebサービスにアクセスするために、この統合システムユーザーにアクセス許可を付与する必要があります。

  1. 新しい セキュリティグループを作成するには、 検索 バーに create security group と入力します。

  2. セキュリティグループの作成 検索結果をクリックします。

  3. [テナントセキュリティグループのタイプ] ドロップダウンメニューから 統合システムセキュリティグループ（制約なし） を選択します。

  4. グループの名前を作成します。

  5. 次の画面の 統合システムユーザーで、統合システムユーザーをリストに追加します。今後の編集のためにこのグループに戻るには、 検索 バーにセキュリティグループ名を入力するだけです。

  6. 作成したグループが、Okta統合に必要な一連のビジネスドメイン（以下にリスト）に適切にアクセスできることを確認してください。

    次のドメインについて、手順a〜e（下記）を繰り返します。

    • “External Account Provisioning” – Get and Put

    • “Worker Data: Public Worker Reports” – Get and Put

    • “Worker Data: Work Contact Information” – Get and Put

    • “Worker Data: Current Staffing Information” – Get

    • “Worker Data: All Positions” – Get

    • “Worker Data: Business Title on Worker Profile” – Get

    1. 検索 バーに ドメインセキュリティ構成 と入力します。

    2. 結果のレポートを選択します。

    3. 新しい検索フィールドで、 ドメイン フィールドにドメイン名を入力し、 OKをクリックします。

    4. 次のページで、ドメイン名の後の省略記号をクリックして、ドロップダウンメニューを表示します。 ドメイン>セキュリティポリシーのアクセス許可の編集]を選択します。

    5. 上記のドメインリストに基づいて、統合アクセス許可の下の適切なセクションに、前に作成したセキュリティグループを追加します。たとえば、 Get and Put vs Get ）。

    注：Workdayで確認して、セキュリティグループに必要なすべての権限が構成されていることを確認してください。

手順

Workdayプロビジョニングを設定する

API統合を設定するには、Workdayインスタンスの[Okta プロビジョニング タブに移動します。

［Enable API Integration（API統合を有効化）］をオンにします。

注：最初の3つのフィールドは必須で、残りはオプションです。

  1. APIユーザー名：APIユーザー名の形式は[統合システムユーザー名] @ [テナント]です。例： wd_integration @ oktademo。テナント名はWorkdayURLに記載されています。

  2. APIパスワード：上記で使用されている統合システムユーザーのパスワード。

  3. WebServicesエンドポイント：テナント名はWorkdayURLにあります。Webサービスエンドポイントを取得するには、次のように、組織内の任意のWebサービスのWSDLを検索するだけです。

    1. 検索 バーに public web services と入力します。

    2. レポートで、 パブリックWebサービスを選択します。

    3. パブリックWebサービス リストから、それらのいずれかを選択し、省略記号（…）をクリックしてドロップダウンメニューを表示します。 Webサービス> WSDLの表示]を選択すると、WSDL全体が別のウィンドウに表示されます。

    4. WSDLで soapbind：address を検索して、選択したWebサービスに対応するWebサービスエンドポイントを確認します。

    5. Okta構成の場合、必要なのはテナント名までのURLのみです。たとえば、WSDLの値が https://implcc.workday.com/ccx/service/okta_pt1/Human_Resources/v19の場合、次のように入力するだけで済みます： https：// impl-cc。 workday.com/ccx/service/okta_pt1。

  4. 統合システムID （オプション）： Workdayカスタム属性を参照してください。

  5. 部門フィールド （オプション）：部門フィールドは、Oktaのユーザーの部門属性に使用されるワーカー属性を決定します。デフォルトでは、値は Business Unitです。

  6. Pre-Start Interval （オプション）：Pre-Start Intervalは、Oktaでユーザーをインポートしてアクティブ化する必要がある採用日の何日前かを決定します。詳細については、開始前の間隔の詳細を参照してください。

  7. 個人の電話を同期する （オプション）：このオプションを使用すると、職場の電話番号が利用できない場合に、OktaがWorkdayからアクセスした個人の電話番号を使用できるようになります。

  8. Workdayを持つワーカーのみをインポートする：このオプションを使用すると、Workdayアカウントを持つWorkdayワーカーのみをインポートし、持っていないワーカーを自動的に除外できます。このボタンをクリックすると、次のインポートには有効なWorkdayワーカーのみが含まれます。

  9. 即時終了理由 （オプション）：リアルタイム同期機能を使用する場合は、このテキストボックスを使用して、Workday IntegrationIDで識別される各終了理由の Termination_Subcategory_ID と一致させます。このフィールドは正規表現である必要があります。詳細については、 Workday Real Time Sync | Okta を参照してください。

  10. 仕事の最終日に非アクティブ化 （オプション）：このオプションを使用すると、ユーザーは、終了日ではなく、仕事の最終日に基づいて非アクティブ化できます。ユーザーの非アクティブ化は、ユーザーの最終作業日が現在の日付と一致したときに発生します。

  11. タイムゾーン対応の終了 （オプション）：このオプションを使用すると、ユーザーはタイムゾーン/場所の現在の時刻に基づいて非アクティブ化できます。詳細については、 を参照してください。

Okta で Workday プロビジョニング機能を有効にする。

左側のパネルで To Okta ]を選択し、プロファイルソースを有効にして、インポートルールを設定します。

インポートのスケジュールを設定

ユーザーインポート プロビジョニング機能は、プロビジョニングが有効になると自動的に有効になります。必要に応じて、この機能の他の設定を編集するだけです。

Workday主導のITプロビジョニング シナリオの場合、Oktaは、スケジュールされたインポートと自動確認を設定して、Workdayのワーカーライフサイクルイベントが手動の介入を必要とせずに定期的にOktaに伝播されるようにすることをお勧めします。Workdayに多数のワーカーがいる場合、インポートの完了に時間がかかる可能性があるため、インポートを頻繁にスケジュールすることはお勧めできません。

注：Workday統合は、スケジュールされたインポートの一部として増分インポートをサポートします。インクリメンタルインポートの詳細については、 増分インポートを参照してください。

最初にユーザーを手動でインポートすることをお勧めします。そのインポートの結果に基づいて、それに応じてインポートをスケジュールできます。つまり、インポートに時間がかかる場合は、インポートスケジュールを調整する必要があります。

プロファイルソースとしてのWorkday

WorkdayがOktaユーザーを管理できるように、プロファイルソースとしてのWorkdayもWorkday主導のITプロビジョニングシナリオで有効にする必要があります。さらに、Workdayは、特にユーザーを作成するActive Directory（AD）インスタンスの上に、最も優先度の高いプロファイルソースとしてリストする必要があります。

雇用前インターバルの詳細

PreStart Interval は、Workdayユーザーの早期プロビジョニング用のオプションのフィールドです。これにより、公式の 労働者/従業員の日付 （これは従業員の実際の開始日です）の前にユーザーアカウントをOktaにオンボーディングできます。間隔は、OktaがWorkdayユーザーの早期インポートを評価するWorkdayユーザーが指定したWorker / Employeeの日付より前の日数を表します。この機能が有効になっている場合、OktaはWorkday PreHire Dateを評価し、設定された間隔内にある場合、Oktaはユーザーをインポートします。

たとえば、OktaのPreStart Intervalを7日に設定し、特定のWorkdayアカウントのPreHireDateがWorker / Employee Dateの7日前に設定されている場合、Oktaはアカウントをインポートします。この同じシナリオで、PreHire DateがOktaで構成された7日間隔よりも大きい場合、Oktaは、PreStart Intervalで定義されたウィンドウの開始まで、インポートの対象とは見なしません。

以下に注意してください。

  • ゼロ以外の事前開始間隔を設定すると、指定した日数だけ、将来の日付のWorkdayユーザー更新が事前にインポートされます。たとえば、2日後の発効日でスケジュールされたWorkdayプロビジョニンググループメンバーシップの変更は、Pre-Start Interval値が3に設定されている場合、次のインポートの直後にOktaに反映されます。

  • 事前開始間隔は、カスタムレポートを介してインポートされた終了日と属性値では無視されます。新入社員の開始前間隔を設定する必要があるが、他の更新を事前に行わせたくない場合は、Oktaの基本Workdayプロファイル属性を使用する代わりに、カスタムレポートからOktaに属性を作成してインポートします。Oktaでのグループ割り当てには、カスタムレポート属性に基づくグループメンバーシップルールを使用することもできます。

  • PreStart Intervalオプションを使用するには、 Profile Sourcing を有効にする必要があります。

  • ベストプラクティスは、PreHire Dateの前に必要となる可能性が最も高い時間（つまり、オンボーディングに必要な最大時間）を含むように間隔を構成することです。

  • 間隔は、ユーザーがいつインポートされるかを定義するのではなく、ユーザーがPreHireDateを持っている場合にいつインポートできるかを指定します。

Workdayプロビジョニング グループの管理

Workdayプロビジョニンググループを使用して、組織化された方法でワーカーをOktaにインポートできます。Active Directoryセキュリティグループと同様に、インポートされたWorkdayプロビジョニンググループは、 People> Group タブに表示されます。これらのグループは、他のOktaグループと同じように使用できます。アプリの割り当て、多要素認証（MFA）ポリシーの割り当てなどです。これらのグループを使用して、ActiveDirectoryやその他のアプリケーションへのプロビジョニングを推進することもできます。プロビジョニンググループは、Workday内で手動で作成する必要があります。作成されると、グループと関連するメンバーシップはOktaへのインポートの一部になります。

Workday管理者へのプロビジョニンググループ管理者権限の付与

Workday管理者がプロビジョニンググループを管理する前に、それらが正しい権限を持っていることを確認する必要があります。 検索 バーに プロビジョニンググループ ]と入力しても、[]プロビジョニンググループの作成、 プロビジョニンググループの削除、または プロビジョニンググループの編集[、のオプションが表示されない場合これは、管理者が必要な権限を持っていないことを示しています。

プロビジョニンググループアクセスを追加するには：

  1. 検索 バーに domain security と入力し、機能領域のドメインセキュリティポリシー を選択します。

  2. Systemと入力し、 OKをクリックします。

  3. 次の画面の左側のペインで、下にスクロールして セキュリティ管理 フォルダを展開します。次に、 プロビジョニンググループ管理をクリックします。

  4. 右ペインの Provisioning Group Administration ]の横にある省略記号をクリックしてドロップダウンメニューを表示し、 Domain Security Policyを選択します。2番目の項目に Enableと表示されている場合は、ポリシーが現在無効になっていることを意味します。有効にするには、［有効］をクリックします。選択後に確認ボックスが表示されます。2番目の項目に Disableと表示されている場合は、ポリシーが現在有効になっていることを意味します。次のステップに進みます。

  5. レポート/タスクのアクセス許可 リストで、管理者が ビュー および 変更 アクセス許可を持つセキュリティグループの1つのメンバーであることを確認します。そうでない場合は、 ドメインセキュリティポリシー[]の横にあるドロップダウンメニューをクリックし、 ドメインセキュリティポリシー>権限の編集 を選択して、右側の セキュリティグループ をリストに追加します。必ず view と modify の両方の権限でリストに追加してください。

Workdayワーカーのプロビジョニンググループへの割り当て

Workdayワーカーは、Workday内のプロビジョニンググループに手動で割り当てることができますが、プロビジョニンググループは、Workday内のビジネスプロセスで定義された条件付きルールに基づいて自動割り当てされるように構成されている場合に最も効果的です。Workday内のビジネスプロセスの変更が含まれるため、WorkdayHR管理者はこの手順を実行する必要があります。詳細については、Workdayサポートにお問い合わせください。

プロビジョニンググループを介したActiveDirectoryへのユーザーのプロビジョニング

プロビジョニンググループの変更

Workdayで既存のワーカーが別のプロビジョニンググループに追加されると、Oktaの関連グループのメンバーシップが変更されます。ただし、関連付けられたADユーザーのOUの場所は変更されません。これは、OktaがADユーザーの作成中に特定のOUにADユーザーを追加するだけであり、更新は適用されないためです。

知っておく必要があります

グループの追加：Workdayで新しく作成されたグループは、次のシナリオでのみOktaに同期されます。

  • 完全インポート：これにより、新しいWorkdayプロビジョニンググループが導入され、Oktaで作成されます。
  • インクリメンタルインポート：これにより、新しいWorkdayプロビジョニンググループが導入され、Oktaで作成されます。
  • RTS：Workday Provisioningグループを作成するだけでは、RTSイベントをトリガーしてOktaにグループを作成することはできません。ただし、WorkdayプロビジョニンググループがWorkdayで作成されると、トリガーされる他のRTSイベントにもグループ情報が含まれ、Oktaで新しいグループが作成されます。

グループの削除：Workdayから削除されたグループは、完全なインポートを介してのみOktaから削除されます。

  • 完全インポート：これにより、存在しなくなったWorkdayグループにリンクされているグループがOktaから削除されます。
  • インクリメンタルインポート および RTS は、削除されたWorkdayグループをOktaから削除しません。

グループ名の変更：Workday内からグループ名を変更すると、Oktaで次の動作が発生します。

  • トリガーされたRTSイベントは、Workdayグループ名の変更を取得し、この新しいグループ名をOkta内の新しい別個のグループとしてOktaに書き込みます。
  • RTSでは、グループのメンバーであるユーザーが更新されると、そのユーザーはOktaの元のグループ（古い名前）から削除され、新しく作成されたOktaグループ（新しい名前）に追加されます。そのユーザーが元のグループ（古い名前）からアプリケーションの割り当てを取得していた場合、それらは失われます。
  • RTSを使用すると、Workday Provisioningグループ（新しい名前）に追加された新しいユーザーによって、グループ（新しい名前）がOktaに書き込まれますが、この新しいユーザーはアプリケーションに割り当てられたり、から削除されたりすることはありません。 1つは、Oktaのグループ（新しい名前）にアプリケーションが関連付けられたことがないためです。
  • インクリメンタルインポートを実行した場合、結果は上記のRTSシナリオと同じになります。グループ（古い名前）は削除されませんが、最後のインポート以降に更新されたユーザーは、グループ（古い名前）からグループ（新しい名前）に移動され、アプリケーションの割り当てが解除されるか、削除されます。プロビジョニング。
  • 完全なインポートが実行されると、グループ（古い名前）が削除され、それに応じて、グループ内のすべてのユーザーの割り当てが解除されるか、関連付けられているアプリからプロビジョニングが解除されます。グループ（新しい名前）がインポートされ、関連付けられたユーザーはすべて（新しい名前で）グループに追加され、アプリは関連付けられません。

Workdayでグループの名前を変更する必要が生じた場合は、代わりに新しいグループを作成することをお勧めします。

上記のように、現在Workday Groupの名前を変更すると、Oktaの下流で望ましくない動作が発生する可能性があります。この問題を回避するには、Workdayで目的の名前の新しいグループを作成し、すべてのユーザーをそのグループに割り当てるのが最善の方法です。インポートおよび/またはRTSジョブがOktaに新しいグループを作成するのを待ちます。新しく作成したグループがOktaに取り込まれたら、名前を変更したいグループとまったく同じように設定します。すべてのユーザーメンバーシップ、グループルール、および/またはアプリケーションの割り当てが、目的の名前の新しいグループと古いグループの間で同じになったら、Workdayから元のグループを削除し、完全インポートを介してOktaを更新して古いグループを削除できます。 Oktaのグループ。すべてのユーザー、ルール、およびアプリケーションの割り当てが新しいグループに複製されているため、アプリケーションまたは割り当てへのアクセスを失うことはありません。

Workdayフィールドのオーバーライド

概要

フィールドオーバーライドは、既存のカスタムレポート機能を置き換えるWorkdayからカスタム属性情報を取得するための代替方法です。

上記の でインポートされたカスタム属性で説明されているように、カスタム属性は現在、別のカスタムレポートエンドポイントを介してインポートされています。これにより、ユーザーの完全なプロファイルを取得するために、コネクターが2つの別個のエンドポイントを処理し、両方からのデータをマージする必要があるため、インポートが複雑になります。Workdayでは、特に大量のデータについては、カスタムレポートも推奨されていません。カスタムレポートの制限に対応して、WorkdayはプライマリAPIにサポートを導入し、 フィールドオーバーライドを介してこれらのカスタム属性をフェッチします。

フィールドオーバーライドを使用すると、インポートプロセスが簡素化され、パフォーマンスが向上します。

カスタムレポートのインポート時間の最適化

多数のユーザーと多数のカスタム属性、特に計算フィールドを組み合わせると、Oktaへのインポート時間が長くなる可能性があります。数時間まで。また、Oktaがカスタムレポートをインポートして正しくフォーマットされていることを検証するため、プロビジョニング設定の保存に長い遅延が発生する可能性があります。

Oktaで完全な統合を設定せずにインポートが実行されるのにかかるおおよその時間を知るには、Webブラウザーでレポートリンクを開くか、Postmanなどのツールを使用して、WorkdayカスタムレポートのJSONURLにアクセスします。 。これを機能させるには、就業日の管理者クレデンシャルを入力するように求められます。

インポートの実行に2時間以上かかるというまれなケースでは、Oktaサービスは開いている接続をタイムアウトします。この場合、Oktaサポートに連絡して、接続タイムアウト期間を2時間以上に延長するように要求してください。

ページ化されたカスタムレポートの使用（推奨）

ページ付けされたカスタムレポートについて

まれに、ページ付けされたカスタムレポートを設定すると便利な場合があります。ページ付けとは、Oktaがすべてのユーザーに対して単一の呼び出しを行うのではなく、特定のユーザーのカスタムレポートをプルするためにユーザーごとの呼び出しを行うことを意味します。

2時間以上かかるインポートに対応するために、2時間の接続タイムアウト制限を延長できない場合、ページネーションはユーザーごとに個別の呼び出しを行います。ただし、全体的なインポート時間は大幅に増加します。

ページ付けされたカスタムレポートは、検証で1人のユーザーのカスタムレポートをチェックするだけでよいため、プロビジョニング設定を保存した後の遅延時間を短縮できます。ただし、これは、インポート時間が長くなるため、設定が頻繁に変更されない場合にのみ役立ちます。

Oktaは、ほとんどのユースケースでページ付けされていないレポートを使用することをお勧めします。

非アクティブ化

概要

インポート（Scheduled、RTS、およびIncremental）中に、Oktaはクエリを実行して、過去24時間以内に終了したワーカー、または今後24時間以内に終了するワーカーがあるかどうかを判断します。このカテゴリに分類される労働者は、以下の規則を適用して決定します。

  • ：ワーカーの終了理由がOkta内で構成された即時終了理由の1つと一致する場合、ワーカーはすぐに非アクティブ化されます。

    注：終了理由が一致する場合、終了日は参照されません。したがって、即時終了の理由があるワーカーは、次のインポートですぐに終了します。

  • ：終了理由が一致しない場合、ワーカーの終了/最終就業日が現在の時刻と比較されます。Oktaは、非アクティブ化時間が経過したかどうかを判断する際に、終了日内にWorkdayによって提供されたタイムゾーンに依存します。通常、Workdayは終了日を午前0時PSTとして返します。現在の時刻がその日付より後の場合、ワーカーは非アクティブ化されます。

注

  • 将来の退職日と一致する即時退職理由を持つ労働者は、1日早く退職します。

  • タイムゾーン対応の非アクティブ化が有効になっていない限り、ワーカーはUTCの午前0時にのみ終了します。

    • 回避策については、 を参照してください。

即時の非アクティブ化

即時の非アクティブ化の理由

デフォルトでは、OktaはWorkdayで終了したWorkerに対してアクションを実行するために1日の終わりまで待機します。このようなアクションには、Workdayアプリからの割り当ての解除や非アクティブ化が含まれる場合があります。ただし、Workerの終了理由が即時終了理由で指定された理由と一致し、終了日が現在の日付に設定されている場合、OktaはWorkdayからイベントを受信した直後にアクションを実行します。

タイムゾーン対応の非アクティブ化

Workday Integrationは、Workdayでのワーカーの場所のタイムゾーンに基づいてワーカーの終了が処理される、タイムゾーン対応の非アクティブ化をサポートするようになりました。

この機能はすべてのWorkdayアプリケーションで利用可能であり、 プロビジョニング タブの タイムゾーン対応の終了 をチェックすることで有効にできます。

タイムゾーン対応の非アクティブ化機能を正常に機能させるには、以下に示す統合システムユーザー/システムグループに追加の必要なアクセス許可を付与してから、以下に説明するようにそれらのアクセス許可をアクティブ化する必要があります。

ドメインセキュリティポリシーのアクセス許可。

  • 次のドメインについて、「Workday でのインテグレーターシステムユーザーの作成」の手順ae（手順6の一部）に従います

    • 「ワーカーデータ：場所の管理」

      ワーカーデータ：場所の管理 が利用できない場合は、 管理：場所 を使用します。

Oktaはワーカーの位置を検出し、その位置に関連するタイムゾーンに基づいて（退職日に基づき）予定された退職手続きを処理します。このワーカーは、ワーカーのタイムゾーンで深夜0時が過ぎた後に予定されているインポート時に非アクティブ化されます。

ワーカーのWorkdayに優先タイムゾーンが設定されている場合、そのタイムゾーンは、検出されたロケーションのタイムゾーンよりも優先されます。タイムゾーンを決定する優先順位は次のとおりです。

  1. ワーカーの優先タイムゾーン

  2. 労働者の場所のタイムゾーン

  3. 太平洋標準時（PST）

例えば、オーストラリアのシドニーに拠点を置くキャシーにとって、Workdayで彼女はGMT+10のタイムゾーンに位置します。

キャシーは7月4日に退職する予定です。タイムゾーン対応の雇用前／退職機能が有効化されていない場合、すべての非アクティブ化はUTCタイムゾーン（GMT）で固定されているため、UTC時で深夜0時が過ぎた後のインポートでキャシーの退職手続きが実行されます。タイムゾーン対応の雇用前／退職機能が有効化されている場合、シドニー時間（GMT+10）で深夜0時が過ぎた後のインポートでキャシーはOktaから非アクティブ化されます。キャシーは、以前に非アクティブ化されていたはずの1０時間前に、実際に非アクティブ化されます。
