Prerequisites for configuring passwordless authentication

Before you create app sign-on policy rules for the uses cases, perform the following tasks:

  • Enable Okta FastPass using Okta Verify as an authenticator
  • Create an Okta FastPass group
  • Create an Okta sign-on policy

Discovery questions

Before you deploy passwordless authentication, consider the following questions. The answers to these questions will impact configuration settings and determine the type of passwordless experience you should use.

  1. When is a user allowed this experience based on device context? Should it include all devices? Registered devices only? Managed devices only?
  2. What requirements must a sign-in event meet to allow this flow?
  3. Which types of devices are predominant in the environment? Do all of these have native biometric access? Do you want to enforce biometric flows each time?
  4. What are your authentication requirements for each app? You should understand Okta assurance levels and how they apply to authenticator types.
  5. What other factors are allowed in Okta?
  6. Does your organization use Active Directory?
  7. Do you allow users to bring their own devices?
  8. Do you want a different user experience depending on the device platform (for example, iOS or Windows)?
  9. Do you want users to be signed in silently, without user interaction?
  10. Do you want your app policies to apply to users coming in from specific network zones?

Enable Okta FastPass using Okta Verify as an authenticator

  1. In the Admin Console, go to Security > Authenticators.
  2. Click the Setup tab.
  3. For Okta Verify, click Actions > Edit.
  4. Select the Okta FastPass (all platforms) checkbox.
  5. In the Okta FastPass section, select Show the "Sign in with Okta FastPass" button checkbox.

The Sign in with Okta FastPass button acts as a backup if Okta Verify doesn't load automatically, and it offers an alternative if the user's device management solution prevents signing in silently. It also provides first-time users with a walkthrough for Okta Verify installation and device registrations.

The first sign-in event from an unknown device shows a standard flow with password and email options if authentication is allowed. When the user successfully signs in from that device, they are presented with all available authenticators for subsequent authentication attempts.

On subsequent authentication attempts, users see this button.

Note: If the user has Okta Verify installed on the device, with biometrics enabled for Okta Verify, Okta prompts for biometrics each time (regardless of the app sign-on policy). Biometrics satisfy a higher level of assurance, so the flow is simplified if a user signs in to any other app during their current session.

Create an Okta FastPass group

To simplify the examples in the rest of this document, create a group for passwordless authentication and add a person to it. This example uses bookmark apps. In your environment, use the app integration you need for each app.

  1. In the Admin Console, go to Directory > Groups.
  2. Click Add Group.
  3. Enter a Name (for example, OktaFastPass Group 1) and a Group Description, and click Add Group.
  4. Go to Directory > People and click Add person.
  5. Enter a First name, Last name, Username, and Primary email. Add your own email to the Secondary email field.
  6. In the Groups field, select the Okta FastPass group you created in Step 2.
  7. In the Password field, select Set by admin, and enter a password.
  8. Clear User must change password on first login.
  9. Click Save. This is the group you'll use to walk through the steps for the first use case: Simple security with one-factor authentication.
  10. Optional. Create two more groups (Okta FastPass Group 2 and Okta FastPass Group 3) for the other two use cases described in this document:
  • Configure two-factor authentication for registered and unmanaged devices
  • Configure FIDO2 (WebAuthn)

Create an Okta sign-on policy

Configuring passwordless authentication requires you to change your Okta sign-on policy by adding a higher priority rule. This change shifts responsibility for defining and enforcing strict authentication requirements to each of your app sign-on policies. Before you change your Okta sign-on policy, be sure your apps are protected with a strong app sign-on policy. It's a best practice to leave the default sign-on policy with its original settings.

  1. In the Admin Console, go to Security > Okta Sign-on Policy
  2. Select Add New Okta Sign-on Policy.
  3. Enter a Policy Name. A best practice is to add "Okta Sign-on Policy" to the name so that it's easy to see which policies are Okta sign-on policies when you read the logs.
  4. Assign this policy to the group you created in Create an Okta FastPass group.

  5. Click Create Policy and Add Rule.
  6. In the Add Rule dialog, enter a Rule Name.
  7. Change ANDPrimary factor is to Password / IDP / any factor allowed by app sign on rules.

    In the new sign-on flow, Okta first determines who the user is and then assesses MFA requirements based on the app they're trying to access. The combination of the user and app tells Okta which policy applies. The app sign-on policy determines which factors are required or allowed.

  8. Clear ANDRequire secondary factor.

Clearing this option removes the requirement for MFA for every app in this org. It also makes the Okta sign-on policy defer completely to the app sign-on policies to determine which authenticators are needed to access the app.
