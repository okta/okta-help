New user registration, activation, and sign-in experience

New users register and activate their profiles in the Sign-In Widget, allowing you to fully customize and brand the onboarding experience.

The Sign-In Widget varies by use case, according to the policies you've set for your org.

  • Self-service registration
  • Sign-in flows
    • Password / IDP / any factor allowed by app sign on rules
    • Password / IDP
  • First sign-in attempt on a new device
    • Email used for SSO

Self-service registration

If self-service registration is enabled and the end user enters a username that doesn't exist in the org, they are prompted to sign up and register for an account.

  1. End users enter their full app Username, including the domain, and then click Next.
  2. The Sign-In Widget displays this message: "There is no account with the Username. Sign up for an account."

  3. End users click Sign Up to launch the Create Account registration form.

If self-service registration is disabled and the end user enters a username that doesn't exist in the org, they are prompted to enter a password but aren't permitted to sign in. In this scenario, the option to sign up for an account is not displayed.

Sign-in flows

The sequence of a sign-in flow depends on the authentication assurance requirements that you set in your policies. Remember that Identity Engine requires that the authentication assurance specified in both the Okta and app sign-on policies are satisfied before it allows the end user to access an app.

Sign-on policies supply the context necessary for the user to advance to the next step and specifies actions to take, such as allowing access, prompting for a challenge, and setting the time before prompting for another challenge. The Okta sign-on policy defines access globally, across all apps. The app sign-on policies enforce end user authentication in the context of the requested application. The user’s location and profile are verified against both policies' group membership and authentication criteria.

Note the following about the end-user sign-in experience:

  • When a user is allowed to sign in without a password, Identity Engine optimizes the sign-in experience. If the device is enrolled with Okta Verify and a biometric authenticator is enabled, the biometric authenticator is always the first factor used for user authentication.
  • If assurance requires end users to sign in with a password, the password prompt will always be displayed first. This corresponds to any configuration where the app sign-on policy and password authenticator plus other factor types are defined in the sign-on policy.

Password / IDP / any factor allowed by app sign on rules

If any of your Okta sign-on policy rules has the primary factor set to Password / IDP / any factor allowed by app sign on rules, end users see the identifier-first Sign-In Widget as the first screen during their access flow.

  1. End users enter their full app Username, including the domain, and then click Next. Keep me signed in retains their identifier as well as authenticator verification information on their device for the amount of time designated by the sign-on policy. This replaces the Remember me option in Classic Engine orgs.
  2. On the security method page, end users select one of the primary authenticators allowed by the combined Okta sign-on and app sign-on policies.
  3. After they’ve selected an authenticator, end users move to the verification step. End users supply the required authenticator and then click Verify.

Password / IDP

If any of your Okta sign-on policy rules has the primary factor set to Password / IDP, end users see the password-first Sign-In Widget.

  1. End users enter their full app Username, including the domain, and then click Next. Keep me signed in retains their identifier as well as authenticator verification information on their device for the amount of time designated by the sign-on policy. This replaces the Remember me option in Classic Engine orgs.
  2. End users enter their password in the Password field, and then click Next.
  3. If either of the sign-on policies required a secondary factor, end users make a selection on the security method page. End users then complete the verification step.

Remember user on sign-in

Admins can configure the Remember user on sign-in feature in Security > General > Organization Settings. Remember user on sign in pre-populates the Username field on the Sign-In Widget each time the widget is shown for the lifetime of the browser window or until the user clears their cookies. Remember user on sign in doesn't keep the user signed in or store MFA preferences.

First sign-in attempt on a new device

All sign-in attempts from new devices follow the same process:

  • If your org uses email for SSO, the email and password sign-in options are always available to users, even when the user profile does not exist or the user can't sign in.
  • The user can't sign in if authentication errors occur.
  • If your org doesn't use email for SSO, the user is always prompted for a password.
  • When the same user attempts to sign in again from the browser, a full list of available authenticators is displayed.

Email used for SSO

If your org has email enabled for SSO, end users who try to access an app on a new device see the identifier-first Sign-In Widget.

  1. The end user enters their full app Username, including the domain, and then clicks Next.
  2. If the end user's profile exists, the Security Method page displays Email and Password options.
    1. If the end user selects Password, they enter their password and then click Next.
    2. The Sign-In Widget prompts the end user for Email verification.
    3. The end user selects Email, and then authenticates by clicking a link provided in the email.
  3. If the end user's profile doesn't exist, the Security Method page displays Password as the only option.
    1. The end user enters their password and then clicks Next.
    2. The Sign-In Widget displays the message "Unable to sign in."

When email is configured as a required policy factor, the user can either authenticate using a code provided in the email or click a link.

  • If no subsequent factors are required, the link signs the user in automatically, as long as the user clicks it in the same browser session.
  • If the user opens a link in a separate browser session (or on a different device entirely), the user is asked to return to the original browser session to be signed in.
