Create an endpoint security integration app sign-on policy

You can use Okta Expression Language (EL) to create or edit an Okta sign-on policy that allows Okta to evaluate the trust signals collected by your endpoint detection and response (EDR) solution.

Start this procedure

  1. In the Admin Console, go to Applications > Applications.
  2. Select an app that you want to protect with trust signals collected from your EDR solution.
  3. On the application’s main page, click the Sign On tab.
  4. Scroll to the Sign On Policy section.
  5. Create or edit an app sign-on policy.
  1. Configure IF conditions. These conditions specify when the rule is applied.
    IF conditionDescription
    IF User's user type isBy default, the rule applies to any user assigned to the app. Accept the default or specify a user type.
    AND User's group membership includesBy default, the rule applies to any group assigned to the app. Accept the default or specify user groups.
    AND User's IP isBy default, the rule applies to any IP. Accept the default or select network zones. You can sort by zones defined or not defined in Okta, or configure specific zones.
    AND Device state isSelect Registered.
    AND Device management is

    (Note that this IF condition is not available for all orgs. When available, it is not visible when Any is selected for AND Device state is.)

    		Devices don’t need to be managed by MDM software for you to integrate your EDR with Okta Verify. Select Managed or Not managed as applicable.
    AND Device platform isBy default, the rule applies to any platform. Accept the default or select specific platforms.
    AND Risk isBy default, this rule applies to authentication attempts of any risk level. Selecting a rule limits the rule to only the specified risk level. See Risk scoring.
    AND The following custom expression is trueEnter the required Expression Language (EL) to specify the EDR signals you want this policy to evaluate.
    • CrowdStrike examples

      The following format integrates CrowdStrike ZTA signals into your app sign-on policy. Use it to tailor your allow/deny rules to meet your security needs. For available CrowdStrike EDR signals, see Expression Language attributes for endpoint security integration vendors.

      The value shown here is only an example. To determine a value appropriate for your environment, see the CrowdStrike documentation or work with Crowdstrike directly.

      device.provider.zta.overall <= 60

      or

      device.provider.zta.overall >= 60

    • Windows Security Center examples

      For available Windows Security Center EDR signals, see Expression Language attributes for endpoint security integration vendors.

      The following format integrates Windows Security Center signals in to your app sign-on policy. Use it to tailor your allow or deny rules to meet your security needs:

      device.provider.wsc.fireWall == "GOOD"

      or

      device.provider.wsc.fireWall == "POOR"

    See About behavior and app sign-on policy rules and Okta Expression Language.

  1. Configure THEN conditions. These conditions specify how authentication is enforced.
    THEN conditionDescription
    THEN Access isSelect Allowed after successful authentication.
    AND User must authenticate withSelect either of following as applicable:
    • If the Okta Verify enrollment is set up for User Verification (Windows Hello), select Any 2 factors
    • If the Okta Verify enrollment is not set up for User Verification (Windows Hello), select Possession factor
    AND Possession factor restraints areSelect Device bound (excludes phone and email).
    AND Access with Okta FastPass is grantedThese options allow you to choose whether to require end users to prove that they're physically present when using Okta FastPass to authenticate.
    • If the user approves a prompt in Okta Verify or provides biometrics (meets NIST AAL2 requirements)

      • This option requires users to prove that they are physically present.

    • Without the user approving a prompt in Okta Verify or providing biometrics

      • If you select this option and the app sign-on policy requires a possession factor, Okta Verify may perform certificate-based authentication, which allows users to access the resource without proving that they're physically present. This option does not meet NIST AAL2 requirements.

    Password re-authentication frequency isSelect Every sign-in attempt.
    Re-authentication frequency for all other factors isSelect Every sign-in attempt.

  1. Click Save.
