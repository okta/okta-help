Add an app sign-on policy rule

App sign-on policies enforce authentication requirements on end users when they attempt to sign in to an app. Each app in your org already has a sign-on policy in place and you can add rules to that policy to customize the level of access.

For example, you may want all default Okta users to provide a password, but you want all Okta users in a designated group to provide both a password and email as authenticators. You can create one rule that challenges default users to provide just a password, and then create a second rule that challenges all members of the designated group to provide password and email authenticators.

The order in which the rules are evaluated matches the order of the rules on your policy page. If you don’t add any rules, the app authenticates against a default Catch-all Rule.

If your users use an Identity Provider to sign in to Okta, see Add an app sign-on policy rule with an Identity Provider for instructions.

If your users don't use an Identity Provider to sign in to Okta, follow the instructions on this page to add an app sign-on policy rule.

Add an app sign-on policy rule

  1. In the Admin Console, go to Applications > Applications.
  2. Select the app for which you want to create sign-on policy rules.
  3. On the application’s main page, click the Sign On tab.
  4. Scroll to the Sign On Policy section, and then click Add Rule.
  5. Enter a Rule Name.
  1. Configure IF conditions. These conditions specify when the rule is applied.

    AND User isBy default, the rule applies to any user assigned to the app. Accept the default or specify users to include and exclude.

Update an app sign-on policy

  1. In the Admin Console, go to Applications > Applications.
  2. Select the app where you want to manage sign-on policy rules.
  3. On the application’s main page, click the Sign On tab.
  4. In the Sign On Policy section, you can:
    • Change conditions in a rule by clicking the More Options menu and then selecting Edit.
    • Deactivate a rule by clicking the More Options menu and then selecting Deactivate.
    • Delete a rule by clicking the More Options menu, selecting Deactivate, and then selecting Delete.
    • Reorder the priority by dragging and dropping rules.

Once the app sign-on policy rules have been defined and saved, a preview of active rules configured for the policy will be displayed.

A rule is automatically deactivated if all of the included users or groups are deleted.

Security questions in an app sign-on policy

The security question authenticator prompts end users to enter a correct response to a question that they've selected from a list of possible questions.

The security question authenticator:

  • Supports authentication (MFA/SSO) and user password recovery scenarios. If disabled for MFA/SSO, it will not be included as part of sign-on policy evaluation.
  • Can be used as step-up or additional verification when password is used as the primary factor, specifically, if the primary factor in the user's Okta sign-on policy is Password / IDP. Okta recommends against using security questions in any authentication flow.

You can configure Okta to use this authenticator for just account recovery, or for both authentication and account recovery. If you choose the recovery-only option, Okta does not request authentication during the evaluation of your sign-on policy.

For example, if you want to enable Okta FastPass for your users, that is, allow users to access the resource without proving that they're physically present, you will not be able to use a security question as an additional factor. See Add an app sign-on policy rule. If admins want end users to use security questions, you must use the Password / IdP option in the Okta sign-on policy.
