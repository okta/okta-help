認証の委任

認証の委任を使用すると、ユーザーは組織のActive Directory （AD）、Windowsネットワーク シングルサインオン （SSO)、Lightweight Directory Access Protocol（LDAP）を使用するユーザーストアの資格情報を入力することでOktaにサインできます。

ADの認証の委任を有効にする

前提条件: ADインスタンスとOktaを統合します。Active Directory統合を参照してください。

［Setting（設定）］ページで［Active Directoryの新規インポートおよびプロビジョニング設定エクスペリエンス］が有効化されていない場合は以下の手順を実施します。

  1. 管理コンソールで、［Directory（ディレクトリ）］>［Directory Integrations（ディレクトリ統合）］の順に進みます。
  2. ADインスタンスを選択します。
  3. ［Settings（設定）］ タブをクリックして、［Delegated Authentication（認証の委任）］エリアで［Enable delegated authentication to Active Directory（Active Directoryへの認証の委任を有効にする）］を選択します。
  4. （オプション）認証の委任の設定をテストします。
  1. ［Test Delegated Authentication（認証の委任をテストする）］をクリックします。
  2. ADユーザー名とパスワードを入力し、［Authenticate（認証）］をクリックします。

    3. ［AD Username（ADユーザー名）］フィールドに入力する値は、Universal Principal Name （UPN）の後にサフィックスとしてActive Directory （AD）ドメイン名を付けます。例えば、ADドメイン名がoktaad.comの場合、AD Username UPNにはサフィックスとして@oktaad.comが含まれます。ADドメイン名のサフィックスが含まれない場合、認証の委任は失敗します。

  3. 認証が完了したら［Close（閉じる）］をクリックします。
  1. ［Save Settings（設定を保存）］をクリックします。

［Settings（設定）］ページで［New Import and Provisioning Settings Experience for Active Directory（Active Directoryの新規インポートおよびプロビジョニング設定エクスペリエンス）］が有効化されている場合は以下の手順を実施します。

  1. 管理コンソールで、［Directory（ディレクトリ）］>［Directory Integrations（ディレクトリ統合）］の順に進みます。
  2. ADインスタンスを選択します。
  3. ［Provisioning（プロビジョニング）］タブをクリックして［Settings（設定）］リストから［Integration（統合）］を選択します。
  4. ［Delegated Authentication（認証の委任）］までスクロールして［Enable delegated authentication to Active Directory（Active Directoryへの認証の委任を有効にする）］を選択します。
  5. （オプション）認証の委任の設定をテストします。
  1. ［Test Delegated Authentication（認証の委任をテストする）］をクリックします。
  2. ADユーザー名とパスワードを入力し、［Authenticate（認証）］をクリックします。
  3. 認証が完了したら［Close（閉じる）］をクリックします。
  1. ［Save（保存）］をクリックします。

デスクトップ シングル サインオンを有効にする

デスクトップSSOを使用すると、ユーザーがWindowsネットワークにサインインした時に、OktaおよびOktaを介してアクセスされるアプリによってユーザーが自動的に認証されます。OktaのIWA Webアプリは、MicrosoftのIWAとASP.NETを使用して、指定されたゲートウェイIPからユーザーを認証します。インストールおよび設定手順については、デスクトップSSO向けのOkta IWA Webアプリを参照してください。

LDAPへの認証の委任を有効にする

ユーザーがOktaにサインインしたときにLDAPで認証するには、認証の委任を有効にします。

前提条件：Okta LDAPエージェントをインストールし構成する

  1. Okta管理コンソールで、［Security（セキュリティ）］>［Delegated Authentication（認証の委任）］をクリックします。
  2. ［LDAP］ タブをクリックします。
  3. ［Delegated Authentication（認証の委任）］で、［Edit（編集）］をクリックします。
  4. ［Enable delegated authentication to LDAP（LDAPへの認証委任を有効にする）］を選択します。
  5. （オプション）認証の委任の設定をテストします。
  1. ［Test Delegated Authentication（認証の委任をテストする）］をクリックします。
  2. LDADユーザー名とパスワードを入力し、［Authenticate（認証）］をクリックします。
  3. 認証が完了したら［Close（閉じる）］をクリックします。
  1. ［Save（保存）］をクリックします。

エンドユーザーにLDAPパスワードの変更またはリセットを許可する

OktaでエンドユーザーにLDAPパスワードの変更を許可することができます。ユーザーのパスワードの有効期限が切れると、次回ユーザーがOktaにサインインしようとしたときにパスワードを変更するよう求めるプロンプトが表示されます。

エンドユーザーは名前によるドロップダウン メニューをクリックして、［Settings（設定）］>［Account（アカウント）］>［Change Password（パスワードを変更）］に進むことでホームページからパスワードを変更できます。

この機能ではバージョン5.3.0以降のOkta LDAPエージェントが必要です。この機能は、パスワードが失効した時にpwdReset属性を正しくTRUEに設定するLDAPディストリビューションと共に機能します（例：OpenLDAPおよびIBM) 5.3.0。バージョン5.3.0以降のバージョンをインストールする前に、必ずバージョン5.3.0よリ前のエージェントをアンインストールしてください。エージェントのインストールおよびアンインストール手順については、LDAP 統合を参照してください。

  1. Okta管理コンソールで、［Security（セキュリティ）］>［Delegated Authentication（認証の委任）］をクリックします。
  2. ［LDAP］ タブをクリックします。
  3. ［Delegated Authentication（認証の委任）］で、［Edit（編集）］をクリックします。
  4. ［Enable delegated authentication to LDAP（LDAPへの認証委任を有効にする）］を選択します。
  5. LDAPパスワード ポリシーの下で、［Users can change their LDAP passwords in Okta（ユーザーはOktaでLDAPパスワードを変更できる）］を選択します。
  6. ［Password Rules Message（パスワード ルール メッセージ）］フィールドに、エンドユーザーがパスワードを変更する際に従うべきパスワード ポリシー ルールを記述します。
  7. ［Users can reset forgotten LDAP passwords in Okta（ユーザーはOktaでLDAPパスワードを忘れた場合にリセットできる）］を選択します。

新しいユーザーを作成するかインポートしてアクティベートすると、予備メールアドレスの入力を求めるプロンプトが［Welcome（ウェルカム）］ページに表示されます。エンドユーザーがアドレスを入力すると、変更の確認を求める電子メールがそのユーザーに届きます。

エンドユーザーがパスワードを忘れた場合、またはサインイン試行が何度も失敗してLDAPアカウントがロックされた場合、 Oktaサインイン ウィジェットで［Need Help signing in?（サインインでヘルプが必要ですか？）］> ［Forgot password or unlock account（パスワードを忘れた場合やアカウントをアンロックする）］リンクをクリックすることで電子メールやSNSを使用してパスワードをリセットできます。

  • 電子メールでリセットする：エンドユーザーはユーザー名またはメールアドレスを入力して［Send Email（電子メールを送信）］ボタンをクリックします。24時間有効のアカウントのパスワード リセット用メールがユーザーに届きます。これによりユーザーのOktaおよびLDAPパスワードがリセットされます。アカウントがロックされたためにユーザーが ［Forgot password?（パスワードをお忘れですか？）］リンクをクリックした場合、これによってLDAPパスワードが変更されアカウントがロック解除されます。
  • SNSでリセットする：エンドユーザーはユーザー名またはメールアドレスを入力して［Send Text Message（テキスト メッセージを送信）］ボタンをクリックします。これにより、パスワード リセット コードを含むテキスト メッセージが送信されます。プロンプトを受信すると、ユーザーは電話からコードを入力してプロンプトに従いパスワードをリセットできます。
  1. ［Save（保存）］をクリックします。

代理認証のシステムログ情報を表示する

AD代理認証の障壁の特定に役立てるために、システムログにはそれぞれの認証の委任（代理認証）要求の期間に関する情報が含まれています。システムログには以下の時間がミリ秒単位で記録されます。

  • delAuthTimeTotal — Oktaで代理認証に費やされた合計時間。この時間には、エージェントの合計時間と、エージェントがリクエストの処理を開始するまでのOktaでのキューの待ち時間が含まれます。リクエストを処理するエージェント数が十分でない場合、キュー待ち時間が長くなります。
  • delAuthTimeSpentAtAgent — エージェントがリクエストの処理に費やした時間の合計です。これにはドメイン コントローラーで費やした時間が含まれます。
  • delAuthTimeSpentAtDomainController — ドメイン コントローラーで費やした時間です。

注：この機能を使用するにはバージョン3.1.0以降のADエージェントが必要です。

  1. 管理コンソールで、［Directory（ディレクトリ）］>［Directory Integrations（ディレクトリ統合）］の順に進みます。
  2. ADまたはLDAP インスタンスを選択します。
  3. ページのトップで［View Logs（ログを表示）］をクリックします。

ジャストインタイム プロビジョニング

ジャストインタイム（JIT） プロビジョニングについての詳細は

  • Active Directoryを使用する場合、Active Directoryのジャストインタイムプロビジョニングによるユーザーの追加とアップデートを参照してください。
  • デスクトップSSOを使用する場合、「カスタマイズ」を参照してください。

組織でJITが有効化され、ADまたはLDAP統合で代理認証が選択されている場合、ユーザープロファイルを作成しユーザーデータをインポートするためにJITが使用されます。
