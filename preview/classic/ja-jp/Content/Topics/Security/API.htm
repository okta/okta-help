APIトークン管理

APIページで、Okta APIトークンの管理と作成、オリジンURLの追加を行います。

Okta APIの詳細については、Oktaデベロッパーサイトをご覧ください。

トークン

ブラウザのHTTPクッキーがOktaアプリケーションへのリクエストの認証に使用されるのと同様に、APIトークンはOkta APIへのリクエストの認証に使用されます。APIトークンは特定のユーザー用に発行され、そのトークンでのリクエストはすべて、そのユーザーを代理するものになります。APIトークンは秘密情報であり、パスワードのように取り扱う必要があります。

APIトークンは、そのトークンを作成するユーザーの権限を持ちます。ユーザーの権限が変更されると、トークンの権限も変更されます。Oktaは、権限が変更されないサービスアカウントからAPIトークンを生成することを推奨しています。

APIトークンは30日間有効で、APIリクエストで使用されるたびに自動的に更新されます。トークンが30日以上非アクティブ状態になると、取り消されて再利用できなくなります。また、作成したユーザーがアクティブな場合のみトークンは有効です。ディアクティベートされたユーザーにより発行されたトークンは拒否されます。ユーザーアカウントがリアクティベートされると、APIトークンが追加の操作なしで受け入れられます。

Oktaエージェントには、Okta組織へのアクセスで使用するAPIトークンがインストール中に発行されます。このトークンは標準のAPIトークンに似ていますが、Oktaにより管理されます。

［API Token（APIトークン）］ページで、すべてのOkta APIトークンを管理します。エージェントトークンは通常、エージェントのアクティベート、ディアクティベート、リアクティベートの際に管理されます。

このページにエージェントトークンが表示されるので、ここから確認を行えます。また、そのエージェントトークンを起因として発生する可能性のあるセキュリティ上の問題も把握できます。ほとんどのエージェントはトークンを使用します。トークンの設定は通常、エージェントのアクティベート、リアクティベート時に自動的に処理されます。このトークンのリストには、組織でのOktaトークンの使用状況に関する情報が記載されます。

トークンの管理で行えるアクションは5つあります。

Okta APIトークンの作成

Okta APIでの認証に使用する独自のトークンを作成するには、［Security（セキュリティ）］>［API］の順に移動して、［Create Token（トークンの作成）］ボタンをクリックします。

重要：トークンを確認できるタイミングは作成プロセス中のみです。今後のためにこれのスクリーンショットを取得することをおすすめします。トークンの作成後、トークンは保護のためにハッシュとして保存されます。APIトークンはパスワードのように取り扱います。スクリーンショットは安全な場所に保管するようにしてください。

すべてのトークンの表示

［API Tokens（APIトークン）］のページを開くと、すべてのトークンが表示されます。すべてのエージェントおよびAPIトークンのトークンのステータス、タイプ、名前、使用、作成日、有効期限日、最後に使用された日が表示されます。表示を並べ替えるには、右側の［Sort by（並べ替え基準）］ドロップダウンメニューからsortを選択します。

トークンのステータスは、以下のカラーコードで示されます。

  • 緑色 – 過去3日間以内にトークンが使用されています。
  • 灰色 – 過去3日間でトークンが使用されておらず、そのトークンが有効期限切れになるまで少なくとも7日以上あります。
  • 赤色 – トークンは7日以内に有効期限切れになります。
  • 黄色 – トークンが不審です。

    不審なトークンは、Oktaで登録されていないエージェントに関連付けられています。通常のエージェントのデプロイでは、不審なトークンは作成されません。

    推奨：不審なトークンを調査します。トークンの名前をクリックして、関連付けられているエージェントのプロビジョニングを確認します。エージェントがOktaで登録されていない場合、または、エージェントをリアクティベートせずにディアクティベートした場合、このページからトークンの取り消しと削除を行えます。

トークンのタイプごとに表示

左側のリストでトークンのタイプを選択して、表示するトークンのタイプを制限します。ほとんどは、トークンのタイプごとに分類されています。また、［Suspicious Tokens（不審なトークン）］のカテゴリーには、Oktaに登録されていないエージェントに関連付けられたトークンが含まれます。

トークングループの他に、単一のトークンを表示することもできます。単一のトークンを見つけるには、トークンの値を入力してから［Find Token（トークンを探す）］を選択します。

特定のタイプのトークンの数が常に表示されます。このリストは動的で、トークンの数やタイプが変更されるたびに変化します。

トークンの取り消し

トークンを取り消すには、トークン情報の右側にあるごみ箱アイコンをクリックします。トークン情報の右側にあるごみ箱アイコンは、そのトークンを取り消せる場合にのみクリック可能になります。

  • APIトークンはいつでも取り消せます。
  • エージェントがアクティブでなければ、そのエージェントトークンを取り消せます。そうでない場合、トークンを取り消す前にエージェントをディアクティベートする必要があります。Okta ADエージェントなどの一部のエージェントでは、エージェントをディアクティベートするとそのエージェントのトークンが自動的に取り消されます。
システム・ログでの追跡

システム・ログには、APIトークンの作成と失効に関する情報が含まれます。この操作に関連して表示されるメッセージは、API Token created（APIトークンが作成されました）か、API Token revoked（APIトークンが取り消されました）のいずれかです。Okta APIを介してのみアクセス可能なシステム・ログv1における、これらのイベントのカテゴリーはToken Lifecycle（トークンのライフサイクル）です。

トークンを作成したユーザーと同じユーザーがトークンを取り消した場合、アクターとターゲットに同じ情報が含まれます。トークンを作成していない管理者がそのトークンを取り消した場合、アクターとターゲットには異なる情報が含まれます。

［Trusted Origins（信頼できるオリジン）］タブ

Origin（オリジン）は、ページのURIスキーム、ホスト名、ポート番号を組み合わせたセキュリティベースのコンセプトです。Oktaから組織のウェブサイトへのすべてのクロスオリジンウェブリクエストとリダイレクトは、明示的に許可されている必要があります。［Trusted Origins（信頼できるオリジン）］タブを使用して、自身が管理、信頼するウェブサイトへのアクセス権を付与して、Okta API経由でOkta組織にアクセスできるようにします。

組織のOktaまたはカスタムドメインURLと異なる、信頼できるオリジンでホストされているサインインページ用WebAuthnを使用できます。WebAuthn (MFA)をご覧ください。

信頼できるオリジンを追加するには：

  1. 管理コンソールで、[セキュリティー] > に移動します ［API］に移動します。
  2. ［Trusted Origins（信頼できるオリジン）］タブを選択します。
  3. ［Add Origin（オリジンを追加）］をクリックします。
  4. ［Add Origin（オリジンを追加）］のダイアログで、名前とオリジンのURLを入力します。
  5. 次のオリジンタイプを確認します：
    • CORS – クロス・オリジン・リソース・シェアリング（CORS）は、ウェブサイトでホストされているJavaScriptがOktaセッションのクッキーを使用してOkta APIに対してXMLHttpRequestを行えるようにする標準のブラウザ機能です。

      注

      CORSとは？クロス・オリジン・リソース・シェアリング（CORS）は、ウェブサイトでホストされているJavaScriptがOktaセッションのクッキーを使用してOkta APIに対してXMLHttpRequest（XHR）を行えるようにする標準のブラウザ機能です。

    • Redirect – サインインまたはサインアウトの後、ブラウザを組織の信頼できるウェブサイトにリダイレクトできます。
  6. ［Save（保存）］をクリックして新しい信頼できるオリジンを保存するか、［Cancel（キャンセル）］をクリックしてキャンセルします。

レート制限

APIエンドポイントのスロットリング制限は、（意図的かどうかやサービス妨害攻撃かに関わらず）送信リクエストにより生じるロードスパイクやサービスの中断から、本番環境およびプレビューテナントのOktaのサービスを保護するために作成されています。この手法は業界で広く採用されており、一貫したサービスレベルを顧客に提供するために多くのSaaSアプリケーションやプラットフォームで導入されています。リクエストがレート制限に達した場合、429 Too Many RequestsのHTTPステータスコードが返されます。

パブリックアプリケーションの場合、悪用防止のために積極的にレート制限が適用されます。また、ユーザーに関するメタデータをリリースする前に、プライマリ認証を正常に完了することが要求されます。詳細については、 Okta Authentication APIをご覧ください。各APIレスポンスの制限を報告するヘッダーの使用についてなど、デフォルトのOkta APIのレート制限に関する詳細は、レート制限をご覧ください。

その他のデフォルトのレート制限は、以下の表にエンドポイントごとに記載されています。Oktaは、サービス管理のプロセスにおいて通知なく制限を増減させる場合があります。

同時実行のレート制限

すべての顧客がサービスを利用できるように、Oktaは同時実行のレート制限を適用しています。この制限は、組織全体で適用される1分あたりのAPIレートの制限とは異なります。

同時実行のレート制限の場合、トラフィックは、エージェントトラフィック、Microsoft Office 365トラフィック、APIリクエストを含む他のすべてのトラフィックの3つの異なる領域で測定されます。1つの領域におけるカウントは、それ以外の2つの領域のカウントに含まれません。

  • エージェントトラフィックの場合、Oktaが各組織のトラフィックを測定して、過去4週間で最も高い使用量の上に制限を設定します。
  • Microsoft Office 365トラフィックの場合、同時トランザクション数の制限は1組織あたり75です。
  • APIリクエストを含む他のすべてのトラフィックの場合、同時トランザクション数の制限は1組織あたり75です。

リクエストが初めて同時実行の制限を超えるとHTTP 429エラーが返され、最初のエラーがログに書き込まれます（60秒ごとに1回）。同時実行の制限に関するレポートは1分ごとに実行されるため、ログの量が少なくなり管理しやすくなります。

デフォルトのレート制限

APIエンドポイントのスロットリング制限は、（意図的かどうかやサービス妨害攻撃かに関わらず）送信リクエストにより生じるロードスパイクやサービスの中断から、本番環境およびプレビューテナントのOktaのサービスを保護するために作成されています。この手法は業界で広く採用されており、一貫したサービスレベルを顧客に提供するために多くのSaaSアプリケーションやプラットフォームで導入されています。リクエストがレート制限に達した場合、429 Too Many RequestsのHTTPステータスコードが返されます。

パブリックアプリケーションの場合、悪用防止のために積極的にレート制限が適用されます。また、ユーザーに関するメタデータをリリースする前に、プライマリ認証を正常に完了することが要求されます。

エンドユーザーのレート制限

Oktaは、Oktaユーザーインターフェイスからのリクエスト数を40回/10秒（1ユーザー、1エンドポイントあたり）に制限しています。このレート制限は、ユーザーを相互に保護し、またユーザーをシステム内の他のAPIリクエストから保護するためのものです。

ユーザーがこの制限を超えると、レート制限が適用されなくなるまでそのユーザーがロックアウトされ、メッセージがユーザーインターフェイスとシステム・ログに記録されます。

例外のリクエスト

エンドポイントスロットルはすべてのOktaの顧客に一様に強制適用されますが、レート制限を一時的に引き上げるようにリクエストできる場合があります。たとえば、デプロイの初期段階で、非常に多くのユーザーやグループがOktaにインポートされる場合などが該当します。このような場合には、一時的な例外のリクエストを行えます。

このリクエストは、引き上げが必要なタイミングの少なくとも10営業日前までに受理される必要があります。例外をリクエストするには、Oktaサポートでケースを開いて次の詳細情報を提供します：

  1. 組織の名前
    • 完全なURLを提供します
    • 例：https://cloudcompany.okta.com、https://unicorn.oktapreview.com
  2. エンドポイントとレート
    • どのURIで制限を引き上げる必要がありますか？
    • どの程度の引き上げが必要ですか？
  3. 開始日と開始時刻
  4. 終了日と終了時刻
  5. ビジネス上の正当性
    • リクエストを行う必要性を詳細に説明してください。

Oktaは、悪用、スパム、サービス拒否攻撃などのセキュリティ上の問題を予防するために、その他の機能でレートを制限する権利を有しています。可能な場合には、Oktaは説明的なエラーコードを提供します。
