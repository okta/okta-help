Configure provisioning

The first step is to provision accounts for users within . This process is similar for most applications. Generally, admins select either specific users or groups of users who can then sign in to with their existing credentials. By default, provisions accounts for all super admins and access requests admins, but any other users or groups must be explicitly authorized.

  1. Access the Admin Console.
  2. Go to Applications Applications and click the Request Access app.
  3. Go to the Assignments tab, and click Assign.
  4. From the assignment window, identify users or groups.
  5. Click Assign.
  6. Click Done.

The system provisions accounts for the requested users.

Manually sync resources from

automatically groups logical representations of data as resources. Resources include applications and groups synced from . To automate access to resources, you can create configuration lists from a resource and reference them in Request Types.

The system syncs resources from daily, but you can sync resources manually (if needed).

This process is only available after completes the initial sync with .

It can take up to an hour for the initial sync to complete.

  1. From the Console, go to Settings Resources .
  2. Identify a resource type.
  3. Click the ellipses, and select Update Now.

To sync resources from integrations, such as Jira or Service Now, see Integrate Jira with [%=okta-feature-names.Access Requests%] and Integrate ServiceNow with [%=okta-feature-names.Access Requests%].

Push groups to

This process allows you to use your existing groups and better model your organizational structure and business process within . For example, an team might want to assign specific approvals or tasks to members of an group for review.

Users must already be provisioned to before they can be pushed as part of an group.

  1. Access the Admin Console.
  2. Go to Applications Applications and click the app.
  3. Go to the Push Groups tab, and click Push Groups.
  4. From the menu, select a search method.

Configure sign on policies

The next step is to configure a sign-on policy for . This is an optional process but allows organizations to control access to . Creating rules is straightforward and should be familiar to most admins. You can define various If/Then style rules from drop-down menus or by using the Expression Language. For details, see Okta Expression Language in Okta Identity Engine.

Create additional admins

By default, any user with Super Administrator permissions is also an admin. When a Super Administrator is first assigned to , their account is automatically assigned admin permissions within .

Changes to Super Administrator accounts aren't automatically synced for users already assigned to . You must manually reassign the application to the user after adding or removing the Super Administrator permissions within your organization.

  1. From the Admin Console, add the Super Administrator permission to a user.
    For more information, see     Assign administrator permissions.
  2. Unassign the user from .
  3. Reassign the user to .
