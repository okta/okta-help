submissions

Overview

Your connector can be submitted to the () for review and publication. By publishing your connector to the , you gain visibility in a public marketplace. Other customers can view and deploy any connectors created through the .

Submitting a connector to the requires a review process to confirm that it meets 's quality assurance standards for publication. Any connector available to the general public must meet the following objectives:

  • Developed according to Best practices in

  • Doesn't negatively affect platform performance

  • Includes help topics for each action card, including Custom API Action cards

  • Executes as expected according to the provided help documentation

  • Meets all basic security standards

The team reviews submitted connectors on a best-effort basis. There are no service level agreements in place around connector submissions. After you submit your connector, the team will contact you with an estimated time frame for the review and publication process.

After it's approved for publication, deploys the connector to preview cells for a one week trial period before being deployed to production cells.

Validation

As part of the evaluation, the team expects the submission to be made by an official account. This account needs to have legal access to the associated platform and any APIs used in the connector. It can be directly associated with the company, or through a subsidiary business or a contractor of the company.

If the account isn't clearly associated with the company (typically through the email domain), the team requests proof through an email from a domain that matches the connector, or from a particular owner at the company.

Requirements

For the testing process, you need to provide the following items when submitting your connector.

Support contact

Provide an actively monitored email address for customer support. publishes this email address as part of your catalog entry.

Escalation contact

You also need to supply an escalation contact address. The team uses this address for any issues encountered during the testing process, or if there's an issue with a published connector. This escalation email isn't made available publicly.

Documentation

You need to have technical documentation that describes the proper use and configuration of your connector. See submission documentation.

Okta doesn't host the documentation for your connector, so it must be available from your own domain or a publicly accessible third-party site. You can keep this content private during the review process, and then coordinate with the team to make your documentation public when publishes your connector.

Test environment

The team needs a valid account or tenant where they can test your connector. This test tenant should have enough sample data to properly evaluate the connector. For example, if the test connector uses pagination, then you must have enough samples so that the connector can return multiple pages of data.

The test data must not contain any personally identifiable information (PII). Don't use real or pseudonymous profile data.

Provided test credentials should have all the necessary permissions to test the endpoints implemented by your connector.

Test guidance

You also need to provide:

  • A list of test cases, detailing the actions performed by your flows. See Prepare connector documentation for specific requirements.

  • A copy of your test flows, exported as a folder file.

  • A link to any API documentation associated with the connector. This can be public or private, as long as the team can access these API docs using the provided test credentials.
