Deployment tasks

Deploying follows a well defined set of tasks, performed in a specific order.

This page is intended to provide an overview of the tasks required to deploy in one of the supported virtual environments. This page doesn't address sizing, capacity planning, and similar concerns. See Capacity planning, sizing, and scaling for more information.

Common deployment tasks include:

Task Description
Select and download OVA Based on the target deployment environment, select and download the correct OVA. Currently supports the following production environments. VMWare (ESXi and Workstation), Microsoft Azure, Amazon Web Service EC2, and Oracle Cloud Infrastructure(OCI) virtual environments.

Note: Oracle VirtualBox is only supported for test and development use.
Deploy image into virtual environment. The first step in deploying is to import the virtual appliance file into the target virtual environment. This task varies based on the selected environment. For development and testing VMWare and Oracle Virtual Box are the simplest deployment environments.
See Deploy [%=okta-feature-names.Access Gateway%] for development environments for details of bootstrapping a simple development environment.
Reset the instance Once has been imported into the virtual environment it should be reset. Resetting the instance involves accessing the instance using the and running a simple reset command.  See [%=okta-feature-names.Access Gateway CMD%] for a general introduction to the .
Determine or manage IP address Depending on environment requirements can use a specific or assigned IP address. This IP address is used to access the instance to defined IDP and other settings and is often entered into /etc/hosts or its equivalent. See Add admin host entries for details of the initial entries required for initializing an instance.
Initialize the Once an is up and running it must be assigned a cookie domain. This domain represents the general domain that the gateway is protecting. See Initialize the [%=okta-feature-names.Access Gateway UI%].
Associate with an org All instances use an org to manage applications, end users, and the access of one from the other. For more information see Configure your Okta tenant as an Identity Provider.

Once complete, a functioning instance of is ready for use.

For complete details and all steps for configuring a local development environment see Deploy [%=okta-feature-names.Access Gateway%] for development environments.
