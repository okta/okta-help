Amazon Web Servicesデプロイ後のタスク

このタスクでは、デプロイ後に必要な一般的なタスクを行います。

タスク 説明
Access Gatewayをリセットして確認する Access Gatewayは、最初のブースト後に初期化する必要があります。これは、基本的なシステム機能を管理するコマンドラインインターフェイスであるAccess Gateway 管理者コンソールを用いて行われます。
ホスト名を指定 Access Gatewayは、デフォルトで変更可能な既知のゲートウェイホスト名に設定されています。
必要なDNSを構成
必要な/etc/hosts管理者とその他のDNSエントリを構成します。
Access Gateway Admin UIコンソールを初期化する クッキードメインとインスタンスホスト名を初期化します。
OktaテナントをIDプロバイダーとして構成する OktaテナントをIDプロバイダーとして構成する
OktaテナントからのSAMLアクセスを構成する Oktaテナントを構成し、SAMLを使ったAccess Gatewayへのアクセスを可能にします。
アプリケーションを統合する

Access Gatewayサンプルまたはその他のアプリケーションを1つ以上追加します。

インスタンスをリセットする

コマンドラインメインメニュー：

  1. Access Gateway 管理者コンソールにサインインします。

    ssh oag-mgmt@adminを使用するか、仮想環境（例：Oracle VirtualBox）の場合は、その環境が提供するコマンドウィンドウを使用します。

    ユーザー名：oag-mgmt パスワード： <default-password>

    Access Gateway 管理者コンソールに初めてサインインする場合、デフォルトパスワードを変更する必要があります。「Access Gateway 管理者コンソールに初めてサインインする」をご参照ください。

  2. [5 - Sysetem（システム）]を選択します。
  3. [7 - Reset（リセット）]を選択します。
  4. 構成をクリアするには、Yを選択します。
  5. システムを初期化するには、Yを選択します。

リセット中、インスタンスは1-2分停止します。

システムが正常に初期化されたら、任意のキーを押してメニューに戻ります。

構成の確認

  1. xを押してメインメニューに戻ります。
  2. [1 - Network（ネットワーク）]を選択します。
  3. [2 - Test network configuration（ネットワーク構成のテスト）]を選択します。
    Okta Access Gatewayは、www.okta.comへのコンタクトを試み、エラーを報告します。
  4. [7 - Connectivity Test（接続テスト）]を選択します。
    ホストとしてwww.okta.comおよびポートとして443を入力します。
  5. 接続が成功したことを確認します。

ホスト名を指定

初回ログイン後、基盤となる仮想アプライアンスのホスト名を変更します。

アプライアンスのホスト名をAccess Gatewayのホスト名とは異なる名前に設定します。ホスト名管理は、管理者 UIのために予約されるため、アプライアンスホスト名として使うことはできません。

  1. Access Gateway 管理者コンソールにサインインします。

    ssh oag-mgmt@adminを使用するか、仮想環境（例：Oracle VirtualBox）の場合は、その環境が提供するコマンドウィンドウを使用します。

    ユーザー名：oag-mgmt パスワード： <default-password>

    Access Gateway 管理者コンソールに初めてサインインする場合、デフォルトパスワードを変更する必要があります。「Access Gateway 管理者コンソールに初めてサインインする」をご参照ください。

  2. [5 - システムを入力します。
  3. [1 - Change Hostname（ホスト名を変更する）]を入力します。
  4. 新しいホスト名を入力する：プロンプトで、ダイアログを表示して交換するホスト名を入力します。
  5. 続行する場合はYを入力し、キャンセルする場合はNを入力します。
  6. システムメニューに戻る場合はXを入力します。

必要なDNSを構成

初回デプロイメント後、Access Gatewayには、いくつかの初期構成に固有のDNSエントリ（ローカル/etc/hostsのエントリまたはWindowsと同等なものを含む）が必要です。次の/etc/hostsとDNSエントリを構成します。

値 説明

例
管理者 Access Gatewayの初期IPアドレス。
local /etc/hostsまたはWindowsと同等のものに入力します。
最初にAccess Gatewayを構成するときのみに使用します。
注：AWSの場合、これはエラスティックIPで、それ以外の場合はAccess GatewayインスタンスのインスタンスIPアドレスです。

/etc/hosts:
192.168.A.B admin
gw-admin.[yourdomain.tld]

Access GatewayのIPアドレスは通常、AレコードとしてDNSに入力されます。

テストおよび初期デプロイメント中は、この値を/etc/hostsに追加することができますが、本番環境ではDNSに記録する必要があります。

gw-admin.atko.xyz which might point to 192.168.A.B
gw [.yourdomain.tld] Access Gatewayサービスリスナー。
この値は通常、gw-admin[.yourdomain.tld]を指すDNS CNAMEレコードとして入力されます。

oag-admin.atko.xyzを指すgw.atko.xyz CNAMEレコード

以下の点にご注意ください：

  • ホストエントリは、ステータスチェックを行う場合にのみ必要です。
  • エントリは、特定のAccess Gatewayノード向けで、アプリケーションドメインではありません。
  • エントリは常にAccess GatewayノードのホストIPを指している必要があります。

構成されると、Access Gateway 管理者 UI コンソールは、ローカルブラウザーから https://gw-admin.[yourdomain.tld]エントリとhttp://adminを使用して到達できるようになります。

Access Gateway Admin UIコンソールを初期化する

クッキーおよび関連のドメインアドレスを指定することによってAccess Gatewayを初期化します。

  1. ブラウザーでhttps://admin/を開き、デフォルトの資格情報を使ってサインインします。「デフォルト資格情報」を参照してください。

    デフォルトパスワードの変更を求められます。「Access Gateway 管理者 UI コンソールに初めてサインインする」をご参照ください。

  2. Access GatewayインスタンスのCookieドメインとホスト名を入力し、［次へ］をクリックします。一般には、Cookieドメインの冒頭に「gw.」を加えたものをホスト名とします。例：

    • SSOクッキードメイン:mysite.mycompany.com
    • Access Gatewayホスト名：gw.mysite.mycompany.com

    SSO CookieドメインとAccess Gatewayホスト名は小文字で入力してください。

  3. ［完了］をクリックします。

  4. Access Gatewayブラウザーに戻り、新しい管理ドメインを使って再度サインインします。

  5. Access Gateway管理者トポロジページが開きます。

OktaをIDPとして構成する

OktaをAccess GatewayのIDプロバイダー（IdP）として構成する場合、3つの個別のタスクが含まれます。

これらのタスクを完了するには、アプリ管理者権限とorg管理者権限が必要です。

  1. Access GatewayのOktaサービスアカウントを作成する

    Oktaでは、 Access Gateway APIキーを作成する際に、専用のサービスアカウントを作成・使用することを推奨します。Oktaは、キーを作成したユーザーの下でAPIキーによって実行されたすべてのアクションをログに記録するため、こうすることで正確なログを維持するのに役立ちます。

    1. Admin Consoleで、［ディレクトリ］［ユーザー］に進みます。
    2. ［Add Person（ユーザーの追加）］をクリックします。
    3. サービスアカウントには、ファーストネームとファミリーネームを入力します。
    4. ユーザーネーム値とプライマリ メール値には、 プレースホルダーメールを入力します。たとえば、service.admin@domain.com。

      サービスアカウントとユーザーアカウントの間の干渉を回避するには、ユーザー名とプライマリ メールのプレースホルダー値を使用します。セカンダリメールとしてメールアドレスを入力します。次に、パスワード・リセットのリクエスト必要な場合、サービス アカウントをアクティブ化および維持することができます。

    5. セカンダリメールには、有効な管理者のメールを入力します。
    6. ユーザーアクティベーションメールを送信する のチェックボックスを選択し、 ［保存］をクリックし アクティブ化されたユーザータブの下に新しく作成されたサービスアカウントが、パスワードリセットステータスで確認できるはずです。
    7. Admin Consoleで、［セキュリティ］ ［Administrators（管理者）］に移動します。
    8. ［Add Administrator（管理者を追加）］をクリックします。
    9. 管理者ロールを値に付与する場合は、先に作成されたサービスアカウントの名前を入力します。
    10. [Super Administrator（スーパー管理者）]チェックボックスを選択し、[Add Administrator（管理者を追加）]をクリックします。この時点で2つのスーパー管理者アカウントを保有している必要があります。
    11. Okta管理者アカウントからサインアウトします。
    12. サービスアカウントのメールアカウントで、Oktaから受信したアクティベーションメールを開いて、[Activation link（アクティベーションリンク）]をクリックします。
    13. パスワードとセキュリティ質問を設定し、アカウントのセキュリティ画像を選択します。
    14. 終了したら、新しいサービスアカウントの資格情報でサインインします。
  2. Okta APIトークンを作成
    1. Admin Consoleで、［セキュリティ］［API］に移動します。
    2. ［Create Token（トークンを作成する）］をクリックします。
    3. トークの目的を特定するトークン名を入力します。たとえば、名前にAccess GatewayまたはOAGを含めます。
    4. [Create Token（トークンを作成する）]をクリックします。
    5. [Token Value（トークン値）]をコピーし、将来参照できるように安全な場所（パスワードマネージャーなど）に保存します。ウィンドウを閉じた後は、トークンを再度表示することはできません。
    6. [Ok, got it（OK、了解です）]をクリックします。
  3. Access GatewayにIdPを構成する
    1. ブラウザーで、Access Gateway 管理者 UI コンソールに移動し、管理者としてさサインインします。
    2. [設定]タブを選択します。
    3. [Identity Providers（IDプロバイダー）]ペインをクリックします。
    4. [+]をクリックし、[OKTA]を選択します。

    5. 以下を入力します：

      • 名前：IdPにとって意味のある名前を入力します（たとえば、Okta IdP）。

      • Okta Org：org（例えばorgname.oktapreview.com、orgname.okta.com、または類似）を入力します。

      • Okta APIトークン：Okta APIトークンを作成したら、Okta orgからコピーしたトークをペーストします。

    6. [Not Validated（検証されていません）]をクリックします。このラベルは、Okta APIトークンが正常に検証されたときに検証済に変わります。
    7. [Okay（OK）]をクリックします。設定タブには、有効なにする必要のあるOkta IdPステータスが表示されます。
    8. [Topology（トポロジ）]タブをクリックします。入力した名前のラベルが付いたアイコンでユーザーIdPが表示されます。

    9. IdPのアイコンをクリックします。正しく構成されていれば、Oktaテナントにリダイレクトされます。
