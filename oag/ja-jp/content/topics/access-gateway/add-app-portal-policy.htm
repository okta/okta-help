必要なアクセスポリシーを追加する

このタスクでは、リクエストをバックエンド保護されたWebリソースにルートするために必要なポリシーを作成します。
オリジナルの例では、以下が使用されました：

URL リダイレクト
/2nd 2ndbackend.myportal.com
/3rd 3rdbackend.myportal.com

必要なリダイレクトごとにポリシーを追加します。

ポリシーは、次のように分類されます。

  • Access Gateway管理者UI コンソールを使用して作成された全体ポリシー。/2ndなどの特定のURIを定義し、カスタム構成のコンテナとして機能します。

  • 次のセクションを含むカスタム構成「コード」：

    • policy_typeなどのポリシー上書き設定。
    • セッション変数
    • ポリシーの属性を表すSAMLからのキー
    • 必須のboilerplateコード
    • インバウンドリクエストの書き換え
    • アウトバウンドリクエストの書き換え
    • 302リダイレクトの書き換え
    • ヘッダーフィールド

第2、第3、およびその他のバックエンドごとにアクセスポリシーを追加します：

ポリシーを追加

  1. 必要な場合、[Policies（ポリシー）]タブを選択します。
  2. ポリシーリストのトップにある追加アイコン ()をクリックし、［Protected（保護対象）］を選択します。

    ポリシータイプの詳細については、「ポリシータイプ」を参照してください。
  3. ポリシータイプを［Custom（カスタム）］に変更します。
  4. [Advanced（詳細設定）]を展開します。
  5. [Custom Configuration（カスタム構成）]テキストエリアに以下のコードを入力します：
    注：コードを複数のセクションに入力します。
    はじめに、ポリシータイプを入力します。 set $policy_type "PROTECTED"

    以下のポリシータイプをサポートします：

    ポリシータイプ対応するpolicy_type
    保護対象保護対象
    保護対象外NO_AUTH
    保護対象ルールPROTECTED_REGEX
    アダプティブルールサポート対象外

    すべてのポリシータイプのリストについては、「ポリシータイプ」をご覧ください。

セッション変数を指定する

  1. set $policy_*ステートメントの後には、次のように記述します。
    #Session variables set $UserName ''; set $oag_username ''; set $RemoteIP ''; set $RelayDomain ''; set $SESSIONID '';

SAMLキーのマッピング

  1. 各属性に対して、対応するインデックス付き変数を指定します。これらSAMLアサーションからのキーであり、アプリケーション属性とインデックス付き変数の間が1対1の関係でなければなりません。
    以下のように3つの属性があるとすると、3つのsetステートメントに加えて、追加のsetステートメントを作成します。
    フィールド名は[Name（名前）]列で指定した名前と一致していることを確認してください。また、$_argc変数は合計属性の数と一致している必要があります（この例では3）。
    #SAML key mapping set $_1 'oagusername'; set $_2 'firstname'; set $_3 'lastname'; set $_argc 3;

boilerplateを追加する

  1. 以下に示す2つの必須ステートメントで、setステートメントを実行します。
    # process request policies access_by_lua_file conf/authSession.lua; # resolver -required if using domain and not IP resolver 127.0.0.1;

インバウンドの書き換え

単純なテキスト置換を表す次のものに注意してください：
<APP_PATH> - 追加のバックエンドプロキシサーバーに渡されるべきOAGアプリのURI。
例：/2nd

  1. インバウンドの相対および絶対リクエストを書き換えます。

    # rewrite incoming requests to remove the /2nd # for relative links # subs_filter href=“/ href=“/<APP_PATH>/ gir; # template subs_filter href=“/ href=“/2nd/ gir; # example #for absolute links: #subs_filter ‘/’ ‘/<APP_PATH>/’; # template subs_filter ‘/’ ‘/2nd/’; # example

アウトバウンドの書き換え

  1. アウトバウンドの相対および絶対リクエストを書き換えます。

    # Rewrite outbound requests to add back in /2nd # for absolute links subs_filter http://<APP_PATH>./2ndbackend.myportal.com https://$http_host/<APP_PATH>/ gir; # for relative links: subs_filter href="/ href="/,<APP_PATH>/ gi;

302リダイレクト

  1. 302リダイレクト。

    # rewrite for 302 redirects proxy_redirect http://2ndbackend.myportal.com/ https://www.myportal.com/2nd/;

新しいバックエンドへのプロキシ

  1. 最後のproxy_passステートメントを入力します。proxy_pass http://2ndbackend.myportal.com/;

ヘッダーフィールド

  1. 以下のステートメントを追加します。
    元の属性値のサブセットである可能性があるすべての必須ヘッダーをリクエストに追加する必要があることに注意してください。 # common managed directives include /etc/nginx/conf/icsgw_location_common.conf; # Include headers for application proxy_set_header oag_username $_1; proxy_set_header firstname $_2; proxy_set_header lastname $_3; # set to hostname that the protected upstream app needs proxy_set_header host localhost;
  2. [Not validated（未検証）]をクリックしてコードブロックを検証します。
    成功すると、[Not Validated（未検証）]ボタンが[Valid（検証済み）]になります。
  3. エラーがある場合は修正し、[Okay（OK）]をクリックしてポリシーを確定します。
  4. [完了]をクリックしてアプリケーションを完成させます。

次の例は、www.myportal.com/2ndに対するすべてのリクエストを2ndbackend.myportal.comにリダイレクトするための完成したコードブロックを示しています。
set $policy_type "PROTECTED"; # The values from auth Session set $UserName ''; set $oag_username ''; set $RemoteIP ''; set $RelayDomain ''; set $SESSIONID ''; # Key(s) from the SAML Assertion # authSession will look in the session for these key # and populate the indexed variables set $_1 'oagusername'; set $_2 'firstname'; set $_3 'lastname'; set $_argc 3; # process request policies access_by_lua_file conf/authSession.lua; # resolver -required if using domain and not IP resolver 127.0.0.1; # rewrite incoming requests to remove the /2nd # for relative links: subs_filter href=“/ href=“/2nd/ gir; # for absolute links: subs_filter ‘/’ ‘/2nd/’; # Rewrite outbound requests to add back in /2nd # for absolute links subs_filter http://2nd./2ndbackend.myportal.com https://$http_host/2nd/ gir; # for relative links subs_filter href="/ href="/2nd/ gi; proxy_pass http://2ndbackend.myportal.com/; # common managed directives include /etc/nginx/conf/icsgw_location_common.conf; # Include headers for application proxy_set_header oagusername $_1; proxy_set_header firstname $_2; proxy_set_header lastnamename $_3; # set to hostname that the protected upstream app needs proxy_set_header host $host;

次の手順

このプロセスを繰り返し、リクエストされたURI／リダイレクトごとに新しいポリシーを追加します。この例では、/3rdをhttps://3rdbackend.myportal.com/にリダイレクトする追加のポリシーを付加します。
