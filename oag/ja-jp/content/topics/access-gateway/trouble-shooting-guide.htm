トラブルシューティングガイド

Access Gatewayを正しくトラブルシューティングするには、以下の前提条件を満たす必要があります：

  • Okta orgへの管理者アクセス権。

  • Access Gateway 管理者コンソールへのアクセス権。

  • ネットワークアプライアンス、アプリケーションサーバーなどからログを取得して監視する方法に関する知識。

  • ログステートメントに表示されるHTTPステータスコードを識別する方法に関する知識。

ログを監視するには

メモ：ログはダウンロードするか、ログサーバー（Graylogなど）に転送できます。詳細については、「ロギングの管理」をご参照ください。

  1. Access Gateway 管理者コンソールにログインします。デフォルトの資格情報は次のとおりです。
    • ユーザー名： oag-mgmt
    • パスワード：OktaMgmt@123
  2. Access Gateway 管理者コンソールメニューから[4-Monitoring（モニタリング）]を選択します。
  3. モニタリングメニューから1- ログを監視を選択します。

HTTPステータスコード

Access Gatewayとその他のアプリケーションは任意のイベント中に以下のステータスコードをブラウザーに返します。また、これらは問題のトラブルシューティングのためにアクセスログに記録されます。

Status Code（ステータスコード） 説明
200 成功
400 Access GatewayがIPアドレスまたはホスト名で呼び出されたアプリケーションに対応していない
401 セッションが存在しない
403 ポリシールールによってリソースへのアクセスが拒否された。
404 不明なページ、コンテンツ、またはリソース
405 [Session Integrity Failed（セッション整合性エラー）]
500 サーバー側のエラー
502 バックエンドアプリケーションが利用不可能
503 アプリケーションは、メンテナンス中、非アクティブ、オフラインモードのいずれかです
504 バックエンドアプリケーションへのリクエストがタイムアウトした

ステータスコードを見つけるには？

Access Gatewayは以下のようなステータスコードを表示します。

  • ブラウザーに表示されるHTTPステータスコードがバックエンドアプリケーションから返されるため、誤解を招く可能性があります。Access Gatewayが返すすべてのHTTPステータスコードは、ユーザーフレンドリーなエラーメッセージとしてページ上に表示されます。前の画像は、その一例です。

  • 次の画像は、Access Gatewayによって生成されたメッセージではなく一般的なエラーメッセージの例です。このタイプの一般的なエラーメッセージは、アプリケーションごとに異なります。Access Gateway以外のエラーページには、Access Gatewayによって保護されているアプリケーションが返すHTTPステータスコードをが表示されます（例として、次の画像を参照してください）。

ステータスコードの中には、バックエンドアプリケーションのエラーによって発生するものもあるため、アプリケーション側で調査する必要があります。

HTTPステータスコードを取得する手順：

アプリケーションやエラーによっては、エンドユーザーにAccess Gatewayのエラーページが表示されない場合があるため、ステータスコードをブラウザーから取得する必要があります。以下の手順に従い、ブラウザーの開発者用ツールを使用してステータスコードを取得します。

  1. 新規ブラウザーウィンドウを開きます。

  2. ページを右クリックし、[Inspect（検査）]をクリックして開発者用ツールを開きます。

  3. [Network（ネットワーク）]タブをクリックします。

  4. テストしたいURLを入力し、Enterを押します。

  5. 必要に応じてログインプロセスを完了します。

  6. 問題のページのステータスコードを取得します。下図を参照してください。

上記の開発者用ツールを開く手順はGoogle Chrome向けです。IEでは、F12を押すと開発者用ツールを開くことができます。

トラッキングID

内部サーバーエラーの場合、Access Gatewayがエラーメッセージに表示されるトラッキングIDを生成します。このトラッキングIDを使用すると、トラブルシューティング中のログファイルからイベントとそれに対応するログメッセージを識別できます。

エラーメッセージにトラッキングIDが含まれる場合、[Tracking ID（トラッキングID）]をクリックし、ログに提供されたトラッキングIDとそれに関連するエラーメッセージをコピーできます。このメッセージには、問題をトラブルシューティングする際に役立つ重要な情報が含まれています。

ログステートメントGateway host:[<host URL>]referer:[<IDP SSO URL>]error:[Login Error] tracking ID:[6eff1f9ca3] details:[Requester/RequestDenied: Could not validate the following SAML AuthnRequest from partner Test App: ]

ステータスコード400

未知のホストステータス

要求されたホスト：<Requested Hostname>（要求されたホスト名）は、このAccess Gatewayではサービスされていません。

考えられる原因

DNSレコードはAccess Gatewayに解決され、対応するホスト名を持つAccess Gatewayで利用可能なサービスまたはアプリケーションはありません。

ログステートメント

Mar 7 15:26:26 localhost.localdomain icsDefault443Access <host URL> <IP ADDRESS> - - [07/Mar/2018:15:26:26 -0600] "GET / HTTP/1.1" 400 1992 "-" "Mozilla/5.0 (Windows NT 6.1; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/64.0.3282.186 Safari/537.36" "-" 0.035 0.035

修正／検証手順

  1. 正しいURLを使用しているか確認します。

  2. Access Gatewayアプリケーションのパブリックドメインフィールドが正しいことを確認します。

  3. DNSまたはローカルホストファイルがホスト名およびIPアドレスを正しく指定していることを確認します。

  4. アプリケーションが関連するホスト名で正しく設定されていることを確認します。

上記の手順で問題は解決しましたか？解決しない場合は、サポートチケットを提出してください。

ステータスコード403

リソースへのアクセスが拒否された

アプリケーション「要求されたアプリケーション」のリソース<Requested Resource>（要求されたリソース）へのアクセスが拒否されました。

考えられる原因

ポリシーエンジンが保護されたリソースへのアクセスを拒否した場合、Access Gatewayはこのステータスコードを返します。リソースへの一部のアクセスが意図的に拒否される条件がある場合、このステータスコードを受信する場合があります。

ステータスコード405

アクセス拒否

Access Gatewayが、<Requested Application>（要求されたアプリケーション）へのユーザーアクセスで異常を検出しました。

考えられる原因

セッション整合性に問題の可能性が検出された場合、セッションのハイジャックを防ぐため、Access Gatewayはこのステータスコードを返します。これは、ユーザーがアクティブなセッションが設定されているネットワークを切り替えた時も発生します。

  2. 手順1を確認したら、ユーザーはブラウザーを再起動してログインし直し、新しいセッションを作成する必要があります。

上記の手順で問題は解決しましたか？解決しない場合は、サポートチケットを提出してください。

どのような問題が発生しましたか？

ステータスコード413

エンティティにリクエストしたコードが大きすぎる

アップロードするファイルが1MBを超える場合、Access Gatewayはエラー413を表示します。

考えられる原因

デフォルトのAccess Gatewayは、容量が1MB以下のファイルのアップロードを許可するよう設定されています。

修正／検証手順

  1. 開発者用ツールを使用して、Access Gatewayによって返されるHTTPステータスコードが413であることを確認します。

  2. ファイルのアップロード上限を増やすには、[Applications（アプリケーション）]タブをクリックします。

  3. 該当するアプリケーションの[Edit App（アプリの編集）]アイコンをクリックします。

  4. アプリケーション内で[Advanced（詳細設定）]ドロップダウンメニューを開きます。

  5. [Maximum File Upload Size Adjuster（最大ファイルアップロード容量の調整）]をスライドして適切な容量に合わせます。

上記の手順で問題は解決しましたか？解決しない場合は、サポートチケットを提出してください。

ステータスコード502

アプリケーションが応答しない

バックエンドWebアプリケーション<Requested Application> （要求されたアプリケーション）がAccess Gatewayからのユーザーリクエストを受信していないため使用できません。

考えられる原因

保護されたバックエンドアプリケーションへの接続に失敗すると、Access Gatewayはこのエラーを返します。

ログステートメント

Apr 5 04:01:38 oag01 icsadmin <Gateway domain URL> <IP address> - - [05/Apr/2018:04:01:38 -0500] "GET / HTTP/1.1" 502 2130 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/65.0.3325.181 Safari/537.36" "-" 0.006 0.000, 0.000 : 0.005

修正／検証手順

  1. アプリが動作しており、Access Gatewayで指定された保護されたURLが到達可能であることを確認します。

  2. DNSレコードが解決可能かを確認します。

  3. バックエンドアプリケーションが、Access Gatewayアプライアンスをホストするサーバーから到達可能かを確認します。「トラブルシューティングツール」をご参照ください。

  4. 負荷分散ソリューションを使用している場合、この問題が生じているのはAccess Gatewayアプライアンスの1つかまたはすべてかを個別に検証します。

上記の手順で問題は解決しましたか？解決しない場合は、サポートチケットを提出してください。

どのような問題が発生しましたか？

ステータスコード503

アプリケーションが非アクティブになっている

アプリケーション<Requested Application>（要求されたアプリケーション）が無効になっており、使用できません。

これらの503ステータスコードは、管理者がアプリケーションへのアクセスを一時的に削除した場合に発生することがあります。このような場合、アプリケーションはIDPでも無効になります。Access Gateway 管理者 UI コンソールの設定を編集する前に、アプリケーションの所有者または適切な管理者にアプリケーションステータスを確認してください。

考えられる原因

アプリケーションが非アクティブになっている場合、Access Gatewayはこの警告ページを表示します。

ログステートメント

Mar 7 16:56:39 localhost ACCESS_GATEWAY ACCESS AUTHZ POLICY INFO USER_AUTHZ [SESSION_ID="N/A" SUBJECT="" RESOURCE="/" METHOD="GET" POLICY="INACTIVE" POLICY_TYPE="NO_AUTH" DURATION="0" APP="IDP Sample Header App" APP_TYPE="SAMPLEIDPHEADER2015_APP" APP_DOMAIN="<App domain URL>" RESULT="ALLOW" REASON=" - N/A" REMOTE_IP="<Remote IP address>" USER_AGENT="Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/64.0.3282.186 Safari/537.36"] allow access to resource.

修正／検証手順

  1. Access Gateway 管理者 UI コンソールを開きます。

  2. [Applications（アプリケーション）]タブに進みます。

  3. アプリケーションのステータスが[Inactive（非アクティブ）]に設定されているか確認します。

  4. [Application（アプリケーション）]を編集します。

  5. アプリケーションステータスを[Application is Active（アプリケーションがアクティブ）]に変更します。

上記の手順で問題は解決しましたか？解決しない場合は、サポートチケットを提出してください。

どのような問題が発生しましたか？

アプリケーションがメンテナンス中

アプリケーション<Requested Application>（要求されたアプリケーション）は一時的に使用できません。

考えられる原因

メンテナンスアクティビティが理由でアプリケーションステータスがメンテナンスモードに変更されている場合、Access Gatewayはこの警告ページを表示します。

ログステートメント

Mar 7 16:58:23 localhost ACCESS AUTHZ POLICY INFO USER_AUTHZ [SESSION_ID="N/A" SUBJECT="" RESOURCE="/" METHOD="GET" POLICY="ACTIVE_MAINT" POLICY_TYPE="NO_AUTH" DURATION="0" APP="IDP Sample Header App" APP_TYPE="SAMPLEIDPHEADER2015_APP" APP_DOMAIN="<App domain URL>" RESULT="ALLOW" REASON=" - N/A" REMOTE_IP="<Remote IP address>" USER_AGENT="Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/64.0.3282.186 Safari/537.36"] リソースへのアクセスを許可します

修正／検証手順

  1. Access Gateway 管理者 UI コンソールを開きます。

  2. [Applications（アプリケーション）]タブに進みます。

  3. アプリケーションのステータスが[Maintenance（メンテナンス）]に設定されているか確認します。

  4. アプリケーションのメンテナンスが完了しバックエンドアプリケーションが利用可能であるか確認します。

  5. アプリケーションを編集します。

  6. アプリケーションのメンテナンスが終了したら、アプリケーションステータスを[Application is Active（アプリケーションが非アクティブ）]に変更します。

上記の手順で問題は解決しましたか？解決しない場合は、サポートチケットを提出してください。

どのような問題が発生しましたか？

アプリケーションがオフラインになっている

アプリケーション<Requested Application>（要求されたアプリケーション）が正しく機能せず、オフラインになっている。この機能停止はログに記録されています。

考えられる原因
