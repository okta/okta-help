容量計画、サイジング、およびスケーリング

容量計画は、ユーザーニーズを満たすために必要なシステムリソースの決定、ピークへの対応、成長の実現に役立ちます。また、余分な容量がアイドル状態になり続けたり、ストレージとメンテナンスのコストが急増したりすること防ぐ上でも役立ちます。

容量計画、サイジング、スケーリングには2つのプライマリツールがあります。サイジングは、容量のサポートに必要なメモリ、ハードディスク容量、CPU処理能力、スループットの計算です。スケーリングは、計算結果に合わせてAccess Gatewayクラスターのサイズを増減するプロセスです。

次のトピックでは、環境に適した戦略を決定できるように容量計画の要素を評価する方法について説明します。

容量計画のコンセプト

デプロイメントに何が必要かは、環境の現在の統計情報を評価することで決定できます。

計算の際は、これらの要素を個別にではなく、相互の関係を考慮して評価することをお勧めします。

  • ユーザー：システムユーザーの総数。
  • アクセス：特定の期間にユーザーがシステムにアクセスする合計回数。週単位や月単位の長期間を分析することをお勧めします。そうすることで、環境のピークや平均を含む、より正確なパターンを見いだすことができます。
  • 認証または承認のピークレート：想定される最大のアクセスレベル。ピークは、月、四半期、年度の終わりに、またはシステムのアップグレード、機能停止、季節のイベントに応じて発生します。
  • 認証または承認の平均レート：特定の期間で想定される標準的なアクセスレベル。
  • Access Gatewayが保護するアプリケーションの総数：環境内でAccess Gatewayが保護するアプリケーションの数。アプリケーションが少ないIT環境では、Access Gatewayのデプロイに必要な容量が少なくなる可能性があります。しかし、認証を頻繁に行うユーザーが多い環境では、必要な容量は多くなる可能性があります。

容量は、ネットワークスループットやバックエンドアプリケーションのパフォーマンスなど、他の要因によって制限される場合があります。「ネットワークインターフェイス」を参照してください。

アクセスレートを推定する

アクセスレートの推定は、Access Gatewayが対応しなければならないアクセス数と認証数を計算するプロセスです。平均アクセスレートの計算では、ユーザーをアクセスの認証が頻繁なグループ、頻繁でないグループ、まれなグループに分割し、全ユーザーと各グループのレートを決定します。

計算例では、次の算術記号が使われています。

* 乗算
/ 除算
+ 加算
- 減算
= 等号

平均アクセスレートを計算する

平均アクセスレートは、Access Gatewayの特定のインスタンスがサポートする必要があるアクセス数の下限を表します。

システムにアクセスするユーザーのセットを調べることで、平均アクセスレートを推定できます。

まず、次の値を決定して計算例で使用します。

  • 総ユーザー数：この値は、Access Gatewayにアクセスする可能性のあるユーザーの総数を表します。
  • 1日あたりの推定ユーザー数：特定の日にアプリケーションを使用するユーザーの割合（%）。
  • 1日あたりの推定アクセス数：特定のユーザーが特定の日にアプリケーションにアクセスする回数。
  • 各セッションのページアクセス数：単一セッション中に所定の認証済みユーザーセットで想定されるページアクセス数。

計算例：平均アクセス数と全体的アクセス数

アクセスタイプ別のユーザーのグループ化

システムにアクセスする頻度に応じてユーザーをグループ化します。

  • 頻繁なユーザー：通常は1日に複数回、定期的にシステムにアクセスします。
  • 頻繁でないユーザー：システムに時折アクセスしますが、頻繁なユーザーに比べて頻度ははるかに低くなります。
  • まれなユーザー：多くても1週間に1～3回しかシステムにアクセスしません。

ハードウェアのサイジング要件を推定する

ハードウェアのサイジングには、Access Gatewayデプロイメントの最適パフォーマンスを確保するために必要なメモリとハードディスクの容量、CPUとコアの数、スループットのレベルが反映されます。

メモリのサイジング

Access Gatewayに必要なアプライアンスメモリの量は次のとおりです。

  • オペレーティングシステム、Access Gatewayエンジン、マイクロサービス：本番環境では1.5GB以上。
  • キャッシュセッション：128MB以上。

オペレーティングシステム、コアAccess Gateway、マイクロサービスのメモリは固定です。そのために、メモリ要件の決定では、キャッシュセッションのサイジングに重点が置かれます。次の要素を使ってキャッシュセッションのサイジングを計算します：

  • 総セッション数：任意の時間におけるインメモリセッションの最大数。総セッション数の計算には、次のいずれかの要素を使用します。
    • ユーザー数
    • 1日当たりのユーザーサインインイベントの割合（%）
    • アクセスされたアプリケーションの数
    • 総セッション数 = ユーザー数 * 1日当たりのユーザーサインインの割合（%） * アクセスされたアプリケーションの数
  • 平均セッションサイズ：任意のセッションの平均推定サイズ。セッションサイズは次の要素の関数です。
    • アプリケーションセッションとアプリケーション属性、デフォルトサイズは約1024バイト
    • Kerberosチケット（該当する場合）、約1024バイト（ただし、アクセスされるIISアプリケーションの数に応じて多くの場合はこれより大きくなります）

次の式を使ってキャッシュセッションのサイジングを計算します：セッションキャッシュ = 総セッション数 x（平均セッションサイズ x 2）

セッションの考慮事項：

  • セッションは、LRU（Least Recently Used）アルゴリズムを使用してクリアされます
    キャッシュが満杯になってから新しいセッションが作成されると、最も古いアイドルセッションが削除されます。
  • キャッシュが満杯になった状況と満杯に近い状況は、セッションモニタリングロガーが発するアラートによって示されます。統計情報は、Access Gateway管理コンソールで調べることができます。
    キャッシュ満杯のインシデントを減らすには、アプライアンスメモリを増やすことを検討してください。
  • 常にピークセッションの使用状況を考慮し、それに応じて計画を行ってください。
    たとえば、従業員の登録時期、朝と昼食後のサインインイベント、類似状況など、一般的に負荷が増加する時間帯を考慮します。

ハードディスクのサイジング

Access Gatewayでは、次の要素向けにハード ディスク容量が必要になります。一定期間におけるこれらの要素の数とサイズを決定してください：

  • ソフトウェア：Access Gatewayソフトウェアとオペレーティングシステム。ソフトウェアファイルのサイズは通常は小規模です。
  • バックアップ：毎晩実行され、30日間維持されます。バックアップファイルのサイズは通常は小規模です。
  • システムログ：ログファイルはローカルディスクにスプールされ、認証、承認、監査、アクセス、セッション、および全ログのファイルがあります。通常、HTTP/HTTPSリクエストごとに1エントリとなります。ユーザー数、ユーザーがアプリケーションにアクセスする回数、そのアプリケーションで生じるページビューの数を考慮してください。
  • ログアーカイブ：ロール、圧縮され、30日間維持されます。

ハードディスクの考慮事項：

  • 最大リクエストまたはピークリクエストのディスク残量レベル低下の警告を回避するために、ディスク容量減少時のロガーアラートをモニタリングします。チェックは1時間ごとに行われ、使用量70%で警告、90%でアラートが発せられます。
  • すべてのHTTPリクエストは、監査ログとアクセスログを生成します。
  • ディスクIOの高速化により、スループットが向上します。
  • セッションサイズは、承認を伴う監査ロギングに影響し、監査ログにはセッションコンテンツが含まれます。
  • ハードディスクのサイジング計算では、控えめにならないようにしてください。バーストや大きなページリクエストによってディスク容量減少の警告が発生するのを避けるために、推定ディスク要件の2倍を割り当てます。

CPUのサイジング

Access Gatewayエンジンは、CPU全体にわたって自動的にスケーリングされ、その結果、1 CPUごとに1ワーカーが生じます。コアを追加するたびにスレッドが追加され、追加の処理が可能になります。

  • CUPやコアの数が多いほど、容量は増加します。

  • 通常、ネットワークスループットは、CPU処理ではなく処理のボトルネックとなります。

スループットのサイジング

スループットは、ネットワークを介したデータ配信のレートを意味します。Access Gatewayでは、ネットワークを介して配信されるデータには、認証、承認、リターンコンテンツのデータが含まれます：

  • 認証 = 処理されたSAMLアサーション（Oktaから各アプリケーションへ）
  • 承認 = HTTPリクエストごとのポリシーチェック（すべてのHTTPリクエスト）

認証と承認の各リクエストは、それぞれ約1024バイトを使用し、2048バイトのリターンデータを生じます。ネットワークスループットは、次の式を使って計算できます：

  • 1秒あたりのサインイン回数 * （1秒あたりの認証 + 承認 + リターンデータ）

平均ネットワーク帯域幅の計算には、次の式を使用します：

平均応答サイズ * 平均リクエスト到着レート

平均的な応答のサイズが約20KBであるとすると、500リクエストの場合、結果は20KB×500となり、1秒あたり10MBに相当します。

リクエストの実行時間やリターンデータのサイズなど、正確なデータはAuthNとすべてのログで調べることができます。

インスタンスのサイジング

各インスタンスサイズでサポートできるユーザー数と認証数は、多くの複雑な要因に応じて異なります。サーバーで利用できる総ディスク容量とメモリー容量の関係、プロセッサーとネットワーク接続の速度など、すべての要因が容量に影響します。Access Gatewayは、環境に合わせて異なる構成でデプロイできます。複数のコアと大容量のメモリーモジュールを備えたハードウェアには、Access Gatewayのインスタンスを1つ、または2つセットアップできます。または、クラスター内の複数のサーバーに分散した、より多く、より小規模なインスタンスにAccess Gatewayをデプロイすることもできます。Access Gatewayインスタンスがサポートできるユーザー数の決定については、このページの計算を使用するか、独自の環境についてOktaサポートまでお問い合わせください。

スケーリング要件を推定する

スケーリングは、Access Gatewayクラスターサイズを増減するプロセスです。

クラスターは、垂直または水平にスケーリングできます：

  • 垂直スケーリング：特定インスタンスのメモリ、ディスク、CPUを追加または削除します。
  • 水平スケーリング：クラスターのAccess Gatewayインスタンスを追加または削除します。

すべてのAccess Gateway高可用性クラスターメンバーを同じCPU、メモリ、ディスク構成でデプロイすることをお勧めします。

クラスターのパフォーマンスを調べる際は、次のヒントを考慮してください。

  • 最高のパフォーマンスは、CPUを追加したり、ソリッドステートディスクを使用したりすることで得られます。
  • 全体的なクラスタースループットを改善するには、水平スケーリングする、または新たなAccess Gatewayインスタンスを追加することをお勧めします。たとえば、1,500のリクエストを処理する2つのノードクラスターであれば、同じCPU、メモリ、ディスク構成の2つの新たなノードを追加することで容量を倍増できます。
  • 一般に、水平方向のスケーリングは、Access Gatewayがスティッキーセッション（セッションアフィニティ）を使用するため、線形となります。Access Gatewayは、ノード間でセッションを共有しません。

容量は、ネットワークスループットやバックエンドアプリケーションのパフォーマンスなど、Access Gatewayに関連しない他の要因によって制限される場合があります。
Access Gatewayネットワーキングとネットワークスループットの拡張の詳細については、「ネットワークインターフェイス」を参照してください。
