容量計画およびサイジング

パフォーマンスの達成には、Okta Access Gatewayの実装で必要な容量を決定することがきわめて重要です。

容量計画とサイジングの概念については、「Access Gateway容量計画およびサイジング」を参照してください。

  • アクセスレートを推定する
  • サイジング
  • スケーリング

アクセスレートを推定する

平均アクセスレートは、Access Gatewayの特定のインスタンスがサポートする必要があるアクセス数の下限を表します。システムにアクセスするユーザーのセットを調べることで、平均アクセスレートを推定できます。

平均アクセスレートを推定するには、次のことを決定します：

  • ユーザー総数：このインスタンスは、合計で何人のユーザーにサービスを提供するか？この値は、ゲートウェイにアクセスする可能性のあるユーザーの総数を表します。
  • 1日当たりの推定ユーザー数：特定の日にアプリケーションを使用するユーザーの割合（%）。
  • 1日当たりの推定アクセス数：特定のユーザーが特定の日にアプリケーションにアクセスする回数。
  • 各セッションのページアクセス数：認証済みユーザーの所定のセットについて、1回のセッション中に予想されるページアクセス数はいくらか？

これらの概念を念頭に置いて、平均認証レートを次のように推定することができます：

  • 平均ユーザー数 = ユーザー総数/1日当たりの推定ユーザー数
  • 平均アクセス数 = 平均ユーザー数/1日当たりの推定アクセス数

次のことを調査して、全体のアクセス数を外挿します：

  • 全体のアクセス数 = 平均アクセス数 x ページアクセス数

ユーザーをシステムにアクセスする頻度でグループ化することを検討します。

  • 頻繁なユーザー：頻繁なユーザーは、通常1日に複数回、定期的にシステムにアクセスします。
  • 頻繁でないユーザー：ユーザーはシステムに時折アクセスしますが、頻度は頻繁なユーザーに比べてはるかに低くなります。
  • まれなユーザー：まれなユーザーは、1週間に1回～3回システムにアクセスします。

次の例では、推定値を計算する方法を示しています。

  • ユーザー総数が10,000人であると仮定します。
  • 頻繁な平均アクセス数 = 10,000 x 頻繁なユーザーの数

ユーザーの50%が頻繁なユーザーがであると仮定した場合、基準値は5,000となります。

頻繁なユーザーは、通常1日当たり5回以上システムにアクセスします。頻繁なユーザーの数を計算するには、次の式を利用します：

  • 頻繁なユーザー数 x 1日当たりのアクセス数 = 5 x 5,000 = 25,000

頻繁でないユーザーのシステムへのアクセスは1日当たり2～3回で、ユーザーベースのその他の25%を表します。

  • 頻繁でないユーザー x 1日当たりのアクセス数 = 10,000 x 25 = 2,500だとして、2,500の頻繁でないユーザーがシステムに3回アクセスすることにより、1日当たりのアクセス数が7,500になります。

まれなユーザーとは、ユーザーベースのうちの残りの25％を示します。これらのユーザーは、最大で1日に1回システムにアクセスしますが、通常は数日に1回しかアクセスしません。

まれなアクセス = まれなユーザー総数 x アクセス総数 x まれなアクセス（一日おき）= 2,500 x 1 x .5 = 1,250（アクセス総数）。

毎日のピーク時の使用量を以下のように推定できます：

  • 頻繁なアクセス数：25,000
  • 頻繁でないアクセス数：7,500
  • まれなアクセス数：1,250
  • これによって、1日のアクセス総数は33,750となります。

サイジング

サイジングを計算する際には、以下の項目を考慮してください：

  • コア - CPU/コアの総数
  • RAM - 一般的なメモリー要件
  • ストレージ - ディスクスペースはどの程度必要か？主にロギングの目的で使用します。
  • NIC - スループット要件は？

アプリケーション

Access Gateway 管理 UI コンソールで定義され、アプリケーションページにリストされているアプリケーションへの参照

認証

認証またはAuthNは、IDを確立するプロセスです。認証は、ユーザーがAccess Gateway内の特定のアプリケーションに初めてアクセスするときに行われるものです。認証は、アプリケーションのリソースにアクセスするときにも行われる可能性があります。

権限付与

AuthZの権限付与は、特定のページまたはリソースへのアクセス権を決定するプロセスです。権限付与は、ユーザーがページなどのアプリケーションリソースにアクセスしようとするたびに行われます。

セッション

Access Gatewayセッションまたは単純セッションは、Access Gatewayによって保持され、使用される情報を指します。通常、これには、従来のHTTPセッションのすべての情報と、属性やKerberosチケット（使用されている場合）など、Access Gateway固有のセッションデータが含まれます。

メモリーサイジング

Access Gatewayアプライアンスのメモリーの使用量は、次のように分けられます。

  • OS、Access Gatewayエンジン、およびマイクロサービス。実働環境では1.5GBが最小と見なされます。
  • キャッシュされたセッション：128MB以上。

OS、コアAccess Gateway、およびマイクロサービスメモリーは固定されているため、メモリー要件の決定は、主にキャッシュセッションサイジングに焦点が当てられます。

メモリーサイズを決定するには、次の点を考慮します。

  • 総セッション数 - 任意の時間でのメモリーセッションの最大数。
  • 平均セッションサイズ - 任意のセッションの平均予想サイズ。

総セッション数は、以下を使用して計算されます。

  • ユーザー数
  • 1日当たりのユーザーサインインイベントの割合（%）
  • アクセスされたアプリケーション数

  • 総セッション数 = ユーザー数 x 1日当たりのユーザーサインインの割合（%） x アクセスされたアプリケーション数

セッションサイズは、次の関数です。

  • デフォルトサイズが1024bまでのアプリケーションセッションとアプリケーション属性

    • および

  • 1024bまでのKerberosチケット（該当する場合）、ただし、アクセスされるIISアプリケーションの数に基づき多くなります

セッションキャッシュは次ように計算します。

  • セッションキャッシュ = 総セッション数 x（平均セッションサイズ x 2）

ハードディスクサイジング

概要

Access Gatewayは、ソフトウェア、システムログ/ログアーカイブ、バックアップ用のハードディスクを必要とします。

ディスクの使用は、次のもので構成されます。

  • オペレーティングシステムおよびAccess Gatewayを含むソフトウェア。
  • 毎晩実施され、30日間保持されるバックアップ。
  • ローカルディスクにスプールされるシステムログ出力。監査、アクセス、すべてのログファイルを含む。
  • 30日間保持され、ロールされ、圧縮されるログアーカイブ。

ソフトウェアおよびバックアップサイズの要件は、通常小さく、主な考慮事項がログサイズとなります。

ログエントリ

ログエントリには、主にセッション情報、認証（AuthN）、権限付与（AuthZ）コンテンツが含まれます。通常、HTTP(s)リクエストごとにエントリは1つとなります。

ディスクのサイズを正しく決定するには、特定の期間におけるこれらのエントリの数とサイズを決定する必要があります。
ログエントリのサイズを決定するには、システムユーザーの数、ユーザーアクセスアプリケーションの数（AuthNエントリ）、アプリケーションの以降のページビューの数（AuthZイベント）を考慮する必要があります。

全体として、ログエントリの構成は、以下に基づいています。

  • セッション - 各ログエントリには、Access Gatewayセッション情報が含まれます。
  • AuthN - 認証監査・ロギング情報。
  • AuthZ - 権限付与ログ情報（アクセスされたリソースとポリシールールを含む）

ディスク使用の観点からは、これらの各々のサイズは重要な得るべき情報です。

例

例を見ていきましょう。
特定のユーザーベースが10,000ユーザーであり、そのうち75%が1日当たりシステムにアクセスし、各ユーザーが平均して10のアプリケーションにアクセスするとします。この場合、平均日次アクセスは次のように決定されます。

  • アクティブユーザー = ユーザー数合計 * アクセスの割合（%）
    10,000 * 75% = 7,500 アクティブユーザー

各ユーザーが10のアプリケーションにアクセス：

  • 1日当たりのアクセス = アクティブユーザー * アクセスされるアプリケーション数
    7,500 * 10 = 75,000アクセス/日

各アクセスにセッション、認証、権限付与が必要であると仮定すると、次のようにログエントリサイズを見積もることができます。

  • ログエントリサイズ = Access Gatewayセッションサイズ + AuthNサイズ + AuthZサイズ + いくつかの小さなフォーマットオーバーヘッド

もっと現実的な例として、Access Gatewayセッション、AuthZ、AuthNサイズがほぼ同じで、それぞれ約1024Bであると仮定しましょう。その場合、各アクセスには約3Kバイトが必要になります。

アクセスが24時間にわたってほぼ均等に配分されると仮定されると、1時間当たりのアクセス数は約75,000/24（3,125）となります。

各アクセスのサイズが3Kバイトの場合、ログの1時間当たりの成長は3K*3125（9.6MB）/時間、すなわち1日当たり約230mbになります。アクセスパターンが一貫していると仮定すると、ログおよびログ関連のコンテンツには約7,000MBのディスクが毎月必要になります。

合理的なルールとしては、予想消費量の2倍に加え、ソフトウェア更新、設定、およびバックアップのための追加のオーバーヘッドスペースを足したものを割り当てることですこの例では、約14GB+10～20％または約17～20GB/月のディスク要件となります。

ハードディスクの考慮事項：

  • 最大リクエストまたはピークリクエストのディスク残量減少の警告を回避するために、ディスク容量減少時のロガーアラートをモニタリングします。チェックは1時間ごとに行われ、70%で警告、90%でアラートを発します。
  • すべてのHTTPリクエストは、監査ログとアクセスログを生成します。
  • ディスクIOの高速化により、スループットが向上します。
  • セッションサイズは、権限付与を使用した監査ログに影響を与え、監査ログにはセッションコンテンツが含まれます。
  • ハードディスクのサイズを控えめにしないでください。バーストや大きなページリクエストによってディスク容量減少の警告が発生するのを避けるために、推定ディスク要件の2倍を割り当ててください。

CPUサイジング

Access Gatewayエンジンは、CPU全体にわたって自動的にスケーリングされ、その結果、CPUごとにワーカーが発生します。コアを追加するたびにスレッドが追加され、追加の処理が可能になります。

CPUの考慮事項

  • より多くのCPU/コアは容量を向上します。

  • 通常、ネットワークスループットはボトルネックです。

スループットサイジング

スループットは、AuthN、AuthZ、リターンコンテンツの直接関数です。

  • AuthN = 処理されたSAMLアサーション（Oktaから各アプリケーションへ）
  • AuthZ = HTTPリクエストごとのポリシーチェック（すべてのHTTPリクエスト）

以下の値を仮定します（実際の値に関しては「ログ」を参照してください）：

1024B 1024B 2048B

ネットワークスループットは、次の関数になります。

  • サインイン / 秒 * AuthNサイズ
  • AuthZリクエスト/秒
  • 返されたデータの平均サイズ

総ネットワークスループットは、次の関数になります。

  • サインイン/秒 *（AuthN + AuthZ/秒 + 返されたデータのサイズ）

スケーリング

スケーリングは、Access Gatewayクラスターサイズを増減するプロセスです。

クラスターは、次のようになります。

  • 垂直方向にスケーリング - 特定のインスタンスからメモリー、ディスク、CPUを追加または削除します。
  • 水平方向にスケーリング - クラスターからAccess Gatewayインスタンスを追加または削除します。

Oktaは、すべてのAccess Gatewayの高可用性クラスターメンバーを同じCPU、メモリー、およびディスク構成で同様に定義することを推奨します。

クラスターのパフォーマンスを調べる際には、次の点を考慮してください。

  • 特定のインスタンスに対して、CPUを追加したり、ソリッドステートディスクを使用したりすることで、著しいパフォーマンス向上が期待できます。
  • 全体的なクラスタースループットを改善するために、Oktaは水平方向のスケーリングまたは新たなAccess Gatewayインスタンスの追加を推奨します。
    例えば、1,500のリクエストを処理する2つのノードクラスターでは、同じCPU、メモリー、およびディスク構成を持つ2つの新たなノードを追加することで、容量を倍増できます。
  • 一般に、水平方向のスケーリングは、Access Gatewayがスティッキーセッション（セッションアフィニティ）を使用するため、線形となります。Access Gatewayは、ノード間でセッションを共有しません。

容量は、ネットワークスループットやバックエンドアプリケーションのパフォーマンスなど、Access Gatewayに関連しない他の要因によって制限される場合があります。
Access Gatewayネットワーキングとネットワークスループットの拡張の詳細については、「ネットワークインターフェイスについて」を参照してください。
